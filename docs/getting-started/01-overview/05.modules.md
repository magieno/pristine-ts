# Modules

Modules are how Pristine apps are composed. Each module bundles a set of related
services, configuration definitions, and lifecycle hooks behind a single `ModuleInterface`
declaration. Your AppModule imports the modules it needs; the framework handles
registering everything with the DI container in the right order.

This overview gives you the shape. The [Modules chapter](../05-modules/00.index.md) is
the deep dive.

> **You'll learn**
> - What a `ModuleInterface` looks like
> - How to define a minimal module
> - Which official modules ship with Pristine and what each is for

## The minimal module

```ts
export const YourFeatureModule: ModuleInterface = {
  keyname: "your.feature",
  importModules: [
    CoreModule,
    NetworkingModule,
  ],
};
```

That's it. Three things matter:

- **`keyname`** â€” a unique string identifier. Convention: prefix with something namespaced
  to your project (`acme.products`, `internal.billing`, etc.) to avoid colliding with
  framework modules.
- **`importModules`** â€” other modules this one depends on. Order doesn't matter; the
  framework resolves the import graph.
- (Optional) **`configurationDefinitions`**, **`providerRegistrations`**, **`onInit`**,
  **`afterInit`**, **`onShutdown`** â€” explained in the deep-dive chapter.

## Pitfalls

- **Keyname collisions are silent.** Two modules with the same keyname both register
  successfully and the second wins. Always namespace your keyname.
- **`importModules` order doesn't matter, but cycles do.** A module that imports a module
  that imports the first one creates a cycle the framework can't resolve.
- **A module isn't automatically loaded just by installing the package.** You have to
  reference it from your AppModule's `importModules`. The CLI's `pristine info` shows
  which modules ended up loaded.
