# Contexts

Most backends accumulate a handful of values that everything wants to know: the
correlation id of the current request, the authenticated user, the request itself. The
naive approach is to pass these as parameters through every method signature, or to
park them on a thread-local in languages that have one. Node has neither convention,
and Pristine handles this with **contexts** — small typed objects propagated
implicitly through `AsyncLocalStorage`.

The mechanism is foundational, alongside [dependency injection](03.dependency-injection.md)
and [modules](05.modules.md), because it's what lets the rest of the framework work
without ceremony: logs auto-tag with the right event id, tracers know which trace
they're producing for, controllers see the right request.

> **You'll learn**
> - What `AsyncLocalStorage` is and why Pristine uses it
> - The two contexts Pristine maintains and what each owns
> - How they nest and what code can read from each
> - The escape hatch for background work that breaks the natural async chain

## `AsyncLocalStorage` in 30 seconds

A Node built-in (`node:async_hooks`) that propagates a value across `await`,
`Promise.all`, `setTimeout`, and `setImmediate`. You install a value at the start of a
logical operation (`als.run(value, fn)`), and any code reachable from `fn` — no matter
how deep the async chain — can read the same value back. Same idea as
`request-context` in Fastify, `cls-hooked` in NestJS, or OpenTelemetry's context API.

Pristine installs values at the right boundaries; your code reads them through small
typed managers.

## The two contexts

| Context | Owned by | Holds | Installed at | Lives until |
|---|---|---|---|---|
| `EventContext` | `@pristine-ts/common` | `eventId`, `traceId?`, `container?` | start of every event in the pipeline | the event finishes |
| `RequestContext` | `@pristine-ts/networking` | `request`, `methodNode?`, `identity?` | start of router request handling | the response is returned |

Two contexts, two `AsyncLocalStorage` instances, two managers. They **nest**: when an
HTTP request is being handled, the router installs a `RequestContext` *inside* the
framework's `EventContext`. Both are queryable from controller code:

```ts
import {EventContextManager} from "@pristine-ts/common";
import {RequestContextManager} from "@pristine-ts/networking";

const eventId = EventContextManager.eventId();    // always available inside any event
const request = RequestContextManager.request();   // only inside HTTP-routed code
```

If your app doesn't import `@pristine-ts/networking` (e.g. an event-only worker),
`RequestContext` simply isn't part of your world. The `EventContext` is always
available because it's installed by the framework's core event pipeline regardless of
which modules you import.

> **Don't confuse `EventContext` with `ExecutionContextInterface`.** They look similar
> by name but answer different questions. `ExecutionContextInterface` (in
> `@pristine-ts/core`) is a *static* descriptor of *where* the framework is running
> — AWS Lambda, CLI, Express, etc. — set once per `kernel.handle()` call.
> `EventContext` is per-event runtime state, set fresh for every event in the
> pipeline. Different lifetimes, different purposes.

## What's in `EventContext`

```ts
class EventContext {
  eventId!: string;                  // correlation id, unique per event
  traceId?: string;                  // filled in when tracing starts
  container?: DependencyContainer;   // the per-event DI child container
}
```

Three accessors, both as instance methods (DI-resolvable for testing) and as static
convenience methods:

```ts
EventContextManager.eventId();      // string | undefined
EventContextManager.traceId();      // string | undefined
EventContextManager.container();    // DependencyContainer | undefined
```

When no `EventContext` is active (boot-time code, unit tests calling a service
directly), every accessor returns `undefined`. Code that reads from contexts must
tolerate that — never assume an id is present.

## What's in `RequestContext`

```ts
class RequestContext {
  request!: Request;                 // the inbound HTTP request
  methodNode?: MethodRouterNode;     // the matched route
  identity?: IdentityInterface;      // populated after auth runs
}
```

`identity` is `undefined` until the authentication pipeline completes. Code that runs
*before* auth (e.g. a request interceptor with high priority) reads `identity()` as
`undefined`.

`RequestContextManager` also exposes `requestId()` for convenience — what HTTP code
typically calls the "request id" is the framework's `eventId`. There's no separate
storage; the accessor delegates to `EventContextManager.eventId()` at lookup time:

```ts
RequestContextManager.requestId();   // same value as EventContextManager.eventId()
```

This way HTTP-shaped code finds the correlation id where it expects to (next to
`request()` / `identity()`) without the framework maintaining two fields that could
drift.

## What this means for application code

You almost never call `als.run(...)` yourself — the framework handles installation.
What you get for free, by virtue of these contexts existing:

- **`logHandler.info("...")` auto-tags every entry with the current event's id.** No
  more threading `request.id` through every service. (Details in
  [Logging](../07-logging/00.index.md).)
- **`@traced` and `runWithSpan(name, fn)` resolve the right `TracingManager`
  automatically.** (Details in [Telemetry](../09-telemetry/00.index.md).)
- **Identity is reachable from anywhere inside a request** via
  `RequestContextManager.identity()` — voters, audit loggers, services deep in the
  stack don't need it threaded as a parameter.

## The escape hatch: `bind()`

ALS propagates across most async patterns, but a few specific ones break the chain:

- Raw `EventEmitter`-style callbacks registered outside the event lifecycle.
- `setImmediate` callbacks that run after the parent function returned and the
  framework already tore down its `als.run(...)`.
- Some native modules that use libuv work queues without honoring async-hooks.

When you spawn work that escapes the natural chain, use `bind()` to capture the
current context and restore it on the callback:

```ts
import {EventContextManager} from "@pristine-ts/common";

const followUp = EventContextManager.bind(() => this.audit(item));
setImmediate(followUp);    // inside followUp, EventContextManager.eventId() still works
```

`bind()` exists on both `EventContextManager` and `RequestContextManager`.

## Pitfalls

- **Detached work outlives the event but the container doesn't.** If a request handler
  kicks off a background promise and returns the response without awaiting, the
  promise can keep running — and `EventContextManager.eventId()` will keep returning
  the original event's id for a while. But the per-event child container has been
  torn down, so resolving services from `EventContextManager.container()` from
  detached work is unsafe. Resolve at the start and capture references, or restructure
  to avoid the detachment.
- **Don't cache contexts in module-level state.** They're per-event by design. Stashing
  a `RequestContext` reference in a singleton leaks request state between unrelated
  requests.
- **Testing without a kernel.** Unit tests that call service methods directly (without
  going through `kernel.handle`) run outside any `EventContext`. `@traced` is a no-op,
  `EventContextManager.eventId()` returns `undefined`, and log calls have no eventId
  attached. This is correct behavior — but if a test relies on a context-derived
  value, install a fake context yourself:
  ```ts
  const ecm = new EventContextManager();
  const ctx = new EventContext();
  ctx.eventId = "test-evt";
  await ecm.run(ctx, () => myService.work());
  ```
