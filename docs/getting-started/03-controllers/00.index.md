# Controllers

Controllers respond to HTTP requests in a clean, separated manner — your business logic
sits in classes decorated with `@controller` and `@route`, and the framework wires up
routing, parameter binding, authentication, authorization, validation, and response
serialization around them.

This chapter walks through every moving part. Before that, though: a map of what runs
when, so the rest of the chapter has a frame.

## The request lifecycle (what runs in what order)

When a request enters the kernel, the router executes a fixed sequence of stages. Each
stage can short-circuit the rest by throwing an error (which feeds into the error
interceptor pipeline). Here is the actual order, drawn from
[`packages/networking/src/router.ts`](../../../packages/networking/src/router.ts):

```
                Request arrives at kernel.handleRequest()
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   1.   │  Route matching                                 │  → 404 if no route matches
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   2.   │  Controller resolution (DI container)           │  → constructor side-effects
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   3.   │  Authentication (@authenticator)                │  → 403 on throw
        │   ├─ authenticator.authenticate(request)        │
        │   └─ identity providers chain (provide())       │
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   4.   │  Authorization (@guard, all guards)             │  → 403 if any returns false
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   5.   │  Request interceptors                           │
        │   (sorted by priority, high → low)              │
        │   ├─ RequestBodyConverterInterceptor (1000)     │  JSON parse
        │   ├─ BodyMappingRequestInterceptor   ( 900)     │  @bodyMapping → instance
        │   │                                  ↑          │  (class-validator runs here)
        │   └─ your custom interceptors        ( default) │
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   6.   │  Parameter resolution                           │
        │   (one per @routeParameter / @body / @identity) │
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   7.   │  CONTROLLER METHOD EXECUTES                     │
        └─────────────────────────────────────────────────┘
                              │
                              v
        ┌─────────────────────────────────────────────────┐
   8.   │  Response interceptors                          │
        │   (sorted by priority, high → low)              │
        └─────────────────────────────────────────────────┘
                              │
                              v
                Response returned to the runtime server

           ─────────────────────────────────────────

   Any throw from stages 3-7 routes to:

        ┌─────────────────────────────────────────────────┐
   E.   │  Error interceptors (interceptError chain)      │
        │   ├─ chained in priority order                  │
        │   └─ each receives the previous one's Response  │
        └─────────────────────────────────────────────────┘
```

### Why this ordering matters

| Stage | What can it see? | What can't it see? |
|---|---|---|
| **Authentication (3)** | Raw request, route context | The body isn't parsed/validated yet |
| **Authorization (4)** | Identity, raw request | Still no parsed body, no instance-of-options-class |
| **Body converter (5, priority 1000)** | Raw request body (string) | Class instance — that happens in the next interceptor |
| **Body mapping (5, priority 900)** | Parsed JSON, target class | Custom interceptors — they run after |
| **Custom interceptors (5, priority 0)** | Parsed + class-mapped + validated body, identity | The handler's response (different phase) |
| **Parameter resolution (6)** | Everything | (Nothing — final step before the handler) |
| **Handler (7)** | Bound parameters, identity, request | — |
| **Response interceptors (8)** | The Response object the handler returned | — |
| **Error interceptors (E)** | The error + an in-progress Response | — |

A few practical consequences:

- **Don't try to read `request.body` as an instance inside an authenticator or guard.**
  Both stages run before request interceptors. The body is still raw at that point. Auth
  checks should rely on headers, route params, or the identity claims — not parsed
  request payloads.
- **A custom request interceptor sees the *converted* body**, because the framework's
  built-in interceptors run first (priority 1000 and 900). If you want to see the raw
  body, set your interceptor's priority above 1000 (or check `request.body`'s type and
  parse it yourself).
- **`@bodyMapping(Class)` triggers class-validator inside the body-mapping interceptor.**
  Validation failures here throw, which routes to error interceptors. By the time the
  controller method runs, validation has already passed.
- **Guards run before interceptors.** If a guard rejects, you save the cost of body
  parsing and validation. This is intentional — auth is cheaper than data conversion.
- **The handler is one step.** Once stage 7 starts, you're in user code. Errors thrown
  there go to error interceptors, not back through the request pipeline.

### When does each piece of "the chapter" plug in?

| Section | Plugs in at stage |
|---|---|
| [Decorators](04.decorators.md) — `@route`, `@routeParameter`, `@body`, `@identity` | 6 (parameter resolution) |
| [Validation](05.validation.md) — class-validator | 5 (inside body mapping, priority 900) |
| [Interceptors](06.interceptors.md) — `RequestInterceptorInterface` | 5 (request), 8 (response), E (error) |
| [Guards](07.guards.md) — `@guard()` | 4 |
| [Enrichers](08.enrichers.md) — `@bodyMapping` + `DataMappingInterceptor` | 5 (priority 900) |
| Authentication — `@authenticator()` | 3 (covered in [chapter 06](../06-authentication/00.index.md)) |
| Authorization — `PermissionManager` | called from inside the controller method (stage 7) |

> Note the **distinction between "guards" and "permission voters"**: guards run at stage
> 4 (route-level, before the handler). Permission voters (via `PermissionManager`) are
> something *you* call from inside your handler or service when you need a per-resource
> authorization check. They use the same security module but plug in at different
> stages. See [Authorization](../08-authorization/00.index.md) for the per-resource case.
