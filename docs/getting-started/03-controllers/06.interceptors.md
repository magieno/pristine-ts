# Interceptors

Interceptors are functions the framework calls at well-defined points in the
request/response lifecycle. Use them when you want to transform every request (or every
response, or every error) without modifying each controller method.

Three flavors exist, distinguished by *when* they fire:

| Interceptor | Fires | Used for |
|---|---|---|
| **Request interceptor** | After mapping, before the route handler | Body conversion, header normalization, request logging |
| **Response interceptor** | After the route handler returns, before serialization | Adding response headers, response logging, format coercion |
| **Error interceptor** | When the route handler throws | Error → response mapping, error logging |

A single class can implement any combination of the three by defining matching methods.

> **You'll learn**
> - What each interceptor type can change
> - How to register an interceptor with the framework
> - When interceptors are the right tool vs. guards or controller logic

## The minimal example: request body logger

```ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable, inject} from "tsyringe";
import {Request} from "@pristine-ts/common";
import {RequestInterceptorInterface} from "@pristine-ts/networking";
import {MethodRouterNode} from "@pristine-ts/networking";
import {LogHandlerInterface} from "@pristine-ts/logging";

@tag(ServiceDefinitionTagEnum.RequestInterceptor)
@injectable()
export class RequestBodyLoggingInterceptor implements RequestInterceptorInterface {
  priority = 100;  // higher = runs first

  constructor(@inject("LogHandlerInterface") private readonly logHandler: LogHandlerInterface) {}

  async interceptRequest(request: Request, methodNode: MethodRouterNode): Promise<Request> {
    this.logHandler.info(`${request.httpMethod} ${request.url}`, {
      extra: {body: request.body},
    });
    return request;
  }
}
```

`@tag(ServiceDefinitionTagEnum.RequestInterceptor)` registers the class. Pristine calls
`interceptRequest()` for every request, passing the request and the matched route
metadata. You return a (possibly modified) request — or the same one if you're only
observing.

## Request interceptors

```ts
interface RequestInterceptorInterface {
  priority?: number;
  interceptRequest?(request: Request, methodNode: MethodRouterNode): Promise<Request>;
  interceptResponse?(response: Response, request: Request, methodNode?: MethodRouterNode): Promise<Response>;
  interceptError?(error: Error, response: Response, request: Request, methodNode?: MethodRouterNode): Promise<Response>;
}
```

All three methods are optional — implement only the ones you need. `priority` defaults to
`0`; higher numbers run first. Use it to order interceptors deterministically (e.g. an
auth header normalizer should run before a body-mapping interceptor).

The framework ships a few built-in request interceptors that your code coexists with:
- `BodyMappingRequestInterceptor` — applies `@bodyMapping(Class)` mappings
- `RequestBodyConverterInterceptor` — coerces JSON bodies into typed objects
- `HttpRequestLoggingInterceptor` (in `@pristine-ts/http`) — logs incoming requests

## Response interceptors

Same interface, but implement `interceptResponse`:

```ts
@tag(ServiceDefinitionTagEnum.RequestInterceptor)
@injectable()
export class AddRequestIdInterceptor implements RequestInterceptorInterface {
  async interceptResponse(response: Response, request: Request): Promise<Response> {
    response.setHeader("x-request-id", request.id);
    return response;
  }
}
```

Use response interceptors for:
- Adding response headers (CORS, security headers, correlation IDs)
- Reshaping response bodies (wrapping every response in `{data: ..., meta: ...}`)
- Conditional content negotiation

## Error interceptors

`interceptError` fires when the route handler throws. You receive the error, the
in-progress response object, the request, and the route metadata. Return a `Response` —
either the one you were handed (potentially mutated) or a fresh one:

```ts
@tag(ServiceDefinitionTagEnum.RequestInterceptor)
@injectable()
export class ErrorToJsonInterceptor implements RequestInterceptorInterface {
  async interceptError(error: Error, response: Response, request: Request): Promise<Response> {
    response.status = error instanceof NotFoundError ? 404 : 500;
    response.body = {
      error: error.message,
      requestId: request.id,
    };
    response.setHeader("content-type", "application/json");
    return response;
  }
}
```

Chained error interceptors run in priority order. Each receives the response from the
previous interceptor in `response`, so you can compose error-mapping logic.

## When NOT to use an interceptor

- **Auth checks**: use [`@authenticator`](../06-authentication/02.authenticator-decorator.md)
  or [`@guard`](07.guards.md). Interceptors run for every request; auth applies to
  specific routes.
- **Per-route logic**: if only one controller method needs the transform, put it in the
  method. Interceptors are global by design.
- **Heavy work**: interceptors are on every request's hot path. Don't put expensive
  operations in them — even DB-backed feature checks add 5-10ms × every request.

## Pitfalls

- **Forgetting to return the request/response.** All three methods return — the framework
  uses what you return, not the original. Returning `undefined` from `interceptRequest`
  drops the request entirely.
- **Mutating shared state.** Interceptors get the same `request` object the route handler
  will see. Adding a field is fine; deleting fields the controller needs breaks things
  silently.
- **Priority collisions.** Two interceptors with the same priority have undefined relative
  order. If order matters, set distinct priorities.
