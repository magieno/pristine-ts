# Guards

A guard is a per-route authorization check. It runs *after* authentication has identified
the caller and *before* the route handler executes. If the guard returns `false`, the
request is rejected with a 403; if `true`, the handler runs.

Guards complement [voters](../08-authorization/01.voters.md) — voters are about *whether
an identity can act on a resource*; guards are about *whether a request can reach a
route*. The two layers compose: a route might pass a guard (request is allowed in) but
then fail a voter check inside the handler (this specific resource can't be deleted by
this user).

> **You'll learn**
> - What `GuardInterface` looks like
> - How `@guard()` attaches a guard to a route or controller
> - When to choose a guard over a voter (or both)

## The interface

```ts
import {Request, IdentityInterface} from "@pristine-ts/common";
import {GuardInterface} from "@pristine-ts/security";

export interface GuardInterface {
  keyname: string;
  guardContext?: GuardContextInterface;
  setContext(context: any): this;
  isAuthorized(request: Request, identity?: IdentityInterface): Promise<boolean>;
}
```

The contract:
- **`keyname`** — a stable name for this guard, used in error messages and tracing.
- **`setContext(context)`** — receives whatever you passed as the second argument to
  `@guard(Guard, options)`. Use it for per-route configuration ("which role is required?").
- **`isAuthorized(request, identity)`** — return `true` to allow, `false` to deny.

## The minimal example

```ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {GuardInterface, GuardContextInterface} from "@pristine-ts/security";
import {Request, IdentityInterface} from "@pristine-ts/common";

@injectable()
export class RoleGuard implements GuardInterface {
  keyname = "role";
  guardContext?: GuardContextInterface;
  private requiredRole?: string;

  setContext(context: any): this {
    this.guardContext = context;
    this.requiredRole = context?.role;
    return this;
  }

  async isAuthorized(request: Request, identity?: IdentityInterface): Promise<boolean> {
    if (this.requiredRole === undefined) return true;       // no role required → allow
    if (identity === undefined) return false;               // role required but no identity → deny
    return identity.claims.role === this.requiredRole;
  }
}
```

Apply with the `@guard()` decorator:

```ts
import {guard} from "@pristine-ts/security";

@injectable()
@controller("/admin")
export class AdminController {
  @authenticator(JwtAuthenticator)
  @guard(RoleGuard, {role: "admin"})
  @route(HttpMethod.Delete, "/users/{id}")
  deleteUser(@routeParameter("id") id: string) {
    // only reached if the caller has identity.claims.role === "admin"
  }
}
```

## Class-level guards

Like `@authenticator`, `@guard` works at the class level to protect every method:

```ts
@authenticator(JwtAuthenticator)
@guard(RoleGuard, {role: "admin"})
@injectable()
@controller("/admin")
export class AdminController { /* every route requires admin role */ }
```

Method-level guards on a class-guarded controller stack — both run, both must pass.

## Multiple guards on one route

You can list multiple guards. All must pass for the request to reach the handler:

```ts
@guard(RoleGuard, {role: "editor"})
@guard(IpAllowlistGuard, {ranges: ["10.0.0.0/8"]})
@route(HttpMethod.Post, "/internal/publish")
publish() { /* runs only if both guards return true */ }
```

Guard order is the order they're declared, top to bottom. The first one to return
`false` short-circuits — subsequent guards don't run.

## Guards vs. voters: pick the right layer

| Need | Tool |
|---|---|
| "Only admins can hit this endpoint" | Guard (`@guard(RoleGuard, {role: "admin"})`) |
| "Only the user who owns this resource can update it" | Voter (called inside the handler, with the resource loaded) |
| "Block requests from outside our office IP" | Guard (`IpAllowlistGuard`) — no identity needed |
| "Admins can delete anyone's posts; users can only delete their own" | Both: a guard for "authenticated" + a voter for ownership |

Rule of thumb: guards check *the request and the identity*; voters check *the identity
and the resource the request is about*. If you need to load the resource to make the
decision, you need a voter (you don't have the resource yet at the guard stage).

## Pitfalls

- **`isAuthorized` can return `false` even without an authentication failure.** A guard
  declining a request returns 403, not 401. Use authenticators for "you're not
  authenticated"; use guards for "you're authenticated but not allowed."
- **Guards run on every matching request.** Keep them fast. A guard that hits a database
  on every request adds DB latency × every guarded route. Cache or use voters (which can
  reuse already-loaded data) instead.
- **`guardContext` is set fresh per request.** Don't store state on `this` across
  requests — the framework constructs a new instance per request.
