# Enrichers

"Enricher" isn't a single decorator or interface in Pristine — it's the collective name
for the layer of the request lifecycle that **transforms raw inbound data into typed
domain objects** before your controller method runs. Three pieces make up that layer:

1. **`@bodyMapping(Class)`** — declares which class an incoming JSON body should be
   coerced into.
2. **`RequestBodyConverterInterceptor`** — built-in interceptor that does the JSON parse
   + `plainToInstance` step.
3. **`DataMappingInterceptorInterface`** — pluggable before/after hooks on the data-mapper
   itself, for cases where you need to mutate rows during the coercion.

If you want a request body to arrive in your handler as a real instance of your options
class — with class-validator decorators having been applied — this is the layer that
does it.

> **You'll learn**
> - How `@bodyMapping(Class)` integrates with controllers
> - What `RequestBodyConverterInterceptor` does automatically
> - When to write your own `DataMappingInterceptor`

## The minimal example: `@bodyMapping`

From the catalog demo:

```ts
// src/catalog/controllers/upsert-product.options.ts
import "reflect-metadata";
import {IsNumber, IsString} from "@pristine-ts/class-validator";

export class UpsertProductOptions {
  @IsString() name!: string;
  @IsNumber() priceCents!: number;
}
```

```ts
// src/catalog/controllers/products.controller.ts
import {body, bodyMapping, route, routeParameter} from "@pristine-ts/networking";

@bodyMapping(UpsertProductOptions)
@route(HttpMethod.Put, "/{id}")
upsert(@routeParameter("id") id: string, @body() options: UpsertProductOptions) {
  // options is a real UpsertProductOptions instance.
  // class-validator has already run against it; if it failed, this handler never ran.
}
```

Two decorators in play:
- `@bodyMapping(UpsertProductOptions)` on the **method** — declares the target class
- `@body()` on the **parameter** — binds the converted body to that parameter

The framework wires it up. By the time `upsert` runs, `options` is an instance, not a
plain object. Validation has happened. If validation failed, the framework returned a 400
with the constraint errors and your handler never executed.

## What `RequestBodyConverterInterceptor` does

The conversion step is implemented as a request interceptor that ships with
`@pristine-ts/networking`. It:

1. Reads the request body (assumed JSON for `application/json` content types).
2. Finds the matched route's `@bodyMapping(Class)` declaration.
3. Calls `plainToInstance(Class, parsedBody)` to produce a real instance.
4. Runs class-validator against the instance.
5. If validation passes, attaches the instance to the request for the `@body()`
   parameter binding to pick up.
6. If validation fails, throws — the framework's error interceptor chain converts that
   into a 400 with the constraint errors.

You don't need to register this interceptor — it's part of the NetworkingModule. You
just need to declare `@bodyMapping(Class)` on routes that should benefit.

## `DataMappingInterceptorInterface`

For more advanced enrichment — say, you want to transform every incoming row during the
mapping step (normalize keys, strip extra fields, look up references) — implement
`DataMappingInterceptorInterface`:

```ts
import {tag} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {DataMappingInterceptorInterface} from "@pristine-ts/data-mapping-common";

@tag("DataMappingInterceptorInterface")
@injectable()
export class CamelCaseToSnakeCaseInterceptor implements DataMappingInterceptorInterface {
  async beforeMapping(row: any): Promise<any> {
    // Runs before plainToInstance. Useful for key normalization.
    return Object.fromEntries(
      Object.entries(row).map(([k, v]) => [k.replace(/([A-Z])/g, "_$1").toLowerCase(), v]),
    );
  }

  async afterMapping(row: any): Promise<any> {
    // Runs after plainToInstance. Useful for cross-field validation or computed fields.
    return row;
  }

  getUniqueKey(): string {
    return CamelCaseToSnakeCaseInterceptor.name;
  }
}
```

These interceptors run per-row during data mapping operations. Most apps don't need them
— `@bodyMapping(Class)` + class-validator covers the common case.

## What enrichers are NOT

- They're not how you add request-scoped context (use the kernel's child-container
  pattern, or store on the request itself in an interceptor).
- They're not auth (use authenticators + guards).
- They're not response transformation (use response interceptors).

The "enricher" framing is specifically about **input data → typed domain object**.

## Pitfalls

- **`@bodyMapping(Class)` without `@body()` doesn't help.** The mapping happens, but the
  result isn't bound to any handler parameter. Both decorators are needed.
- **Class-validator only validates instances with decorated fields.** If your options
  class has fields without `@IsString` / `@IsNumber` / etc., they're not validated and
  any value is accepted. The new `optionsType: ClassConstructor` contract in commands
  applies the same way to body mapping — pass the class, not an instance.
- **Conversion fails silently if `content-type` isn't `application/json`.** The
  RequestBodyConverterInterceptor only kicks in for JSON content types. Requests with
  `application/x-www-form-urlencoded` need an additional interceptor or controller-level
  parsing.
