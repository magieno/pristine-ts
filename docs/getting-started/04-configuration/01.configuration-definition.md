# Configuration definitions

A configuration definition is a declarative record that says "this module needs this
parameter at runtime." Each module that ships with Pristine тАФ `LoggingModule`, `HttpModule`,
your own тАФ declares its configuration definitions in its `ModuleInterface.configurationDefinitions`
array. The `ConfigurationManager` then resolves every declared parameter into a concrete
value at boot time.

> **You'll learn**
> - What a `ConfigurationDefinition` looks like
> - How to declare required vs. optional parameters
> - How default resolvers (environment variables, etc.) work

## The minimal example

From the catalog demo (`demo/projects/catalog/src/catalog/catalog.module.ts`):

```ts
import {ModuleInterface} from "@pristine-ts/common";
import {CatalogModuleKeyname} from "./catalog.module.keyname";

export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  configurationDefinitions: [
    {
      parameterName: `${CatalogModuleKeyname}.upstream-url`,
      isRequired: false,
      defaultValue: "https://example.com/catalog/feed",
    },
  ],
};
```

After boot, any class can inject this value:

```ts
import {injectConfig} from "@pristine-ts/common";

constructor(
  @injectConfig("catalog.upstream-url") private readonly upstreamUrl: string,
) {}
```

`@injectConfig(key)` resolves the value the `ConfigurationManager` registered under that
parameter name. Internally it wraps tsyringe's `@inject("%" + key + "%")` тАФ the `%...%`
sigils are an implementation detail you no longer have to type.

## The two shapes of `ConfigurationDefinition`

```ts
type ConfigurationDefinition =
  | {
      parameterName: string;
      isRequired: true;
      defaultResolvers?: (...)[];
    }
  | {
      parameterName: string;
      isRequired: false;
      defaultValue: string | boolean | number | DynamicConfigurationResolverInterface<any>;
      defaultResolvers?: (...)[];
    };
```

Required parameters (`isRequired: true`) have no `defaultValue`. If no value is provided
at boot and no `defaultResolvers` succeed, the kernel fails to start with a clear error.

Optional parameters (`isRequired: false`) must have a `defaultValue`. They fall through to
that value when nothing else resolves.

## Naming convention

Parameter names follow `<moduleKeyname>.<parameterName>`, e.g.
`pristine.http.kernel-server.port`. The convention prevents collisions across modules and
makes it obvious where a parameter comes from when you see it in logs.

## Default resolvers

A `defaultResolvers` array gives the manager a list of fallbacks to try in order before
falling back to `defaultValue` (or failing, for required parameters). Common pattern:
read from the environment first, then use a hard-coded default.

```ts
import {EnvironmentVariableResolver, NumberResolver} from "@pristine-ts/configuration";

{
  parameterName: "my-app.http.port",
  isRequired: false,
  defaultValue: 3000,
  defaultResolvers: [
    new NumberResolver(new EnvironmentVariableResolver("MY_APP_HTTP_PORT")),
  ],
}
```

Resolution order at boot:
1. The value passed via `kernel.start(module, configurationValues)` тАФ if present, wins.
2. Each `defaultResolvers[]` entry, in order. First one that returns a value (synchronously
   or via a resolved Promise) wins.
3. `defaultValue`, for optional parameters.
4. Failure, for required parameters with no resolver that succeeds.

Built-in resolvers covered in [Custom resolvers](03.configuration-resolver.md):
- `EnvironmentVariableResolver` тАФ reads a `process.env` key
- `NumberResolver`, `BooleanResolver`, `EnumResolver` тАФ coerce a value to the right type
- `DynamicConfigurationResolverInterface` тАФ your own logic

## How it fits with `pristine.config.ts`

The `kernelConfiguration` block in `pristine.config.ts` is just sugar for passing values
to `kernel.start()`:

```ts
// pristine.config.ts
export default defineConfig({
  // ...
  kernelConfiguration: {
    "pristine.http.kernel-server.port": 4000,
  },
});
```

This is layer 1 in the resolution order above тАФ it beats every `defaultResolver`. Useful
for project-level overrides that shouldn't pollute the module's definitions.

## Pitfalls

- **You can declare a definition that's never injected.** No warning. The parameter just
  sits in the container.
- **Required parameters with `defaultResolvers` look optional but aren't.** If every
  resolver throws or returns undefined, boot fails. Treat required + resolvers as "best
  effort тАФ if these don't work, surface the error." Don't use them as silent fallbacks.
- **Parameter names are case-sensitive strings.** A typo in `@injectConfig("my-app.api-key")`
  vs the declaration is a runtime "cannot resolve dependency" error, not a compile error.
