# The configuration manager

`ConfigurationManager` is the service that reads every module's
`configurationDefinitions`, resolves each parameter into a concrete value, and registers
the result with the DI container so the rest of your app can inject it.

You rarely interact with it directly — declaring definitions and injecting `%key%`-
prefixed tokens covers most use cases. This page is for the times you do.

> **You'll learn**
> - What happens between `kernel.start()` and your service's `constructor`
> - How to programmatically register a configuration value
> - When to inject the manager itself vs. the resolved values

## What it does at boot

During `kernel.start()`, after every module has been registered with the container, the
ConfigurationManager runs in two passes:

1. **Register pass**: walks every module's `configurationDefinitions` and records the
   declarations. No values resolved yet.
2. **Load pass**: for each definition, applies the resolution order (explicit value →
   `defaultResolvers[]` → `defaultValue` → fail). Successful resolutions are registered
   in the container under the token `%<parameterName>%`.

The two-pass design lets you spot missing required values *before* paying for resolver
work — the `ConfigurationCheck` phase in `pristine verify` runs only the register pass.

## Injecting resolved values

Pristine apps almost always consume configuration this way:

```ts
import {inject, injectable} from "tsyringe";

@injectable()
export class ProductService {
  constructor(
    @inject("%catalog.upstream-url%") private readonly upstreamUrl: string,
  ) {}
}
```

The `%...%` token is what ConfigurationManager registered the resolved value under. No
async, no Promise — by the time your constructor runs, the value is a primitive (`string`,
`number`, or `boolean`).

## Injecting the manager directly

For dynamic lookups — say, a service that loads different configuration based on a feature
flag — you can inject the manager itself:

```ts
import {ConfigurationManager} from "@pristine-ts/configuration";

@injectable()
export class FeatureFlagService {
  constructor(private readonly configurationManager: ConfigurationManager) {}
}
```

This is rare in practice. Most "I want to look up something dynamic" use cases are better
served by a service that wraps the configuration with domain-specific logic.

## Adding values at runtime

`ConfigurationManager.registerConfigurationValue(key, value, container)` adds a value
after boot:

```ts
configurationManager.registerConfigurationValue("my-app.runtime-flag", true, container);
```

The container immediately starts resolving `%my-app.runtime-flag%` to `true`. Use this
when you have a value that's computed at runtime (e.g. read from a remote config service
during `afterInit`) and want it accessible to the rest of your app via injection.

## Inspection: where do my values come from?

For diagnostics, the closest tool today is `pristine p:config:print`, which shows your
`pristine.config.ts` and tells you which file it came from. It doesn't show every
resolved kernel parameter — to inspect those, raise log verbosity and watch the
`ConfigurationManager` debug output at boot.

## Pitfalls

- **The manager isn't ready in `onInit`.** `onInit` runs *before* configuration is loaded,
  so any value injection inside an `onInit`-triggered code path will fail with "cannot
  resolve token". Use `afterInit` for code that needs configuration.
- **Resolved values are cached.** Setting `process.env.MY_VAR` after boot doesn't change
  the injected value. If you need live config reload, build a service that wraps the
  manager and re-reads from the source explicitly.
- **`registerConfigurationValue` doesn't validate against a definition.** You can register
  any string key for any value. This is intentional (it lets you bridge external config
  sources) but means there's no compile-time guarantee that consumers using `@inject`
  will find what they expect.
