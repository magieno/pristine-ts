# Custom resolvers

A resolver tells the ConfigurationManager *how* to read a value. `@pristine-ts/configuration`
ships a handful for the common cases (environment variables, type coercion). This page
covers the built-ins and how to write your own for sources like AWS Parameter Store,
HashiCorp Vault, or a remote config service.

> **You'll learn**
> - What built-in resolvers exist and how to chain them
> - How to write a `ResolverInterface` for a custom source
> - When to use `DynamicConfigurationResolverInterface` instead

## The built-ins

```ts
import {
  BooleanResolver,
  EnumResolver,
  EnvironmentVariableResolver,
  NumberResolver,
} from "@pristine-ts/configuration";
```

| Resolver | Reads from | Returns |
|---|---|---|
| `EnvironmentVariableResolver(name)` | `process.env[name]` | `string` or `undefined` |
| `NumberResolver(innerResolver)` | Whatever the inner resolver returns | Number, coerced from string |
| `BooleanResolver(innerResolver)` | Whatever the inner resolver returns | Boolean, coerced from string |
| `EnumResolver(innerResolver, enumType)` | Whatever the inner resolver returns | Enum value, validated against the provided enum |

Resolvers compose. A typical declaration:

```ts
{
  parameterName: "my-app.http.port",
  isRequired: false,
  defaultValue: 3000,
  defaultResolvers: [
    new NumberResolver(
      new EnvironmentVariableResolver("MY_APP_HTTP_PORT"),
    ),
  ],
}
```

Reads `process.env.MY_APP_HTTP_PORT`, coerces it to a number, registers it. If the env
var isn't set, the resolver returns undefined, the manager falls through to `defaultValue`
(3000).

## Writing your own resolver

Implement `ResolverInterface<T>`:

```ts
import {ResolverInterface} from "@pristine-ts/common";

export class AwsParameterStoreResolver implements ResolverInterface<string> {
  constructor(private readonly parameterName: string) {}

  async resolve(): Promise<string | undefined> {
    const ssm = new SSMClient({});
    try {
      const out = await ssm.send(new GetParameterCommand({
        Name: this.parameterName,
        WithDecryption: true,
      }));
      return out.Parameter?.Value;
    } catch (err) {
      // Returning undefined lets the next resolver (or defaultValue) take over.
      return undefined;
    }
  }
}
```

Use it like any built-in:

```ts
{
  parameterName: "my-app.db.password",
  isRequired: true,
  defaultResolvers: [
    new AwsParameterStoreResolver("/my-app/prod/db-password"),
    new EnvironmentVariableResolver("MY_APP_DB_PASSWORD"),  // local-dev fallback
  ],
}
```

Two resolvers in priority order. Parameter Store first (prod); env var second (local
dev). If both fail and the parameter is required, the kernel won't boot.

## `DynamicConfigurationResolverInterface`

For resolvers that need access to the kernel itself — e.g. reading from a service that's
already registered in the container — implement `DynamicConfigurationResolverInterface`
instead:

```ts
import {DynamicConfigurationResolverInterface} from "@pristine-ts/common";
import {DependencyContainer} from "tsyringe";

export class RemoteConfigResolver implements DynamicConfigurationResolverInterface<string> {
  constructor(private readonly key: string) {}

  async resolve(container: DependencyContainer): Promise<string | undefined> {
    const remote = container.resolve(RemoteConfigService);
    return remote.fetch(this.key);
  }
}
```

The kernel passes the current container in. You resolve whatever service you need from
it. Same fallthrough semantics — return `undefined` to skip to the next resolver.

> **Note on lifecycle**: dynamic resolvers run during the `ConfigurationLoad` phase of
> boot. The services you resolve must be registered by then — i.e. registered in `onInit`,
> not `afterInit`.

## Pitfalls

- **Returning `null` is not the same as `undefined`.** The manager treats `null` as "value
  is null" and registers it. Use `undefined` to signal "I have no value, try the next
  resolver."
- **Throwing from a resolver is silently swallowed.** The manager catches the throw,
  treats it as no-value, and moves on. If you want to surface a failure, return undefined
  AND log the error yourself.
- **Async resolvers serialize.** Each resolver's `resolve()` is awaited in order before
  the next one runs. A slow remote-config resolver as the first in your chain blocks boot
  on every invocation. Put fast/local resolvers first.
