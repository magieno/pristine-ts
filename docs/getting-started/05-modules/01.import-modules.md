# Importing modules

The `importModules` array on your AppModule is how you compose features. Every Pristine
app's AppModule lists every framework module it uses. The framework registers the
imports' services, configuration definitions, and lifecycle hooks in topological order
before your own code runs.

> **You'll learn**
> - How to add a framework module to your AppModule
> - How import order affects boot
> - When a module needs configuration to import cleanly

## The minimal example

From the catalog demo:

```ts
// demo/projects/catalog/src/app.module.ts
import {AppModuleInterface} from "@pristine-ts/common";
import {CoreModule} from "@pristine-ts/core";
import {HttpModule} from "@pristine-ts/http";
import {NetworkingModule} from "@pristine-ts/networking";
import {AppModuleKeyname} from "./app.module.keyname";
import {CatalogModule} from "./catalog/catalog.module";

export const AppModule: AppModuleInterface = {
  keyname: AppModuleKeyname,
  importModules: [
    CoreModule,
    HttpModule,
    NetworkingModule,
    CatalogModule,
  ],
  importServices: [],
};
```

Three framework modules + one of our own. The framework picks up everything reachable
from this list ŌĆö including the transitive imports each module declares.

## Order doesn't matter (but cycles do)

The framework walks the import graph topologically. If `HttpModule` depends on
`LoggingModule` (declared inside `HttpModule.importModules`), it gets resolved either
way you list them. Pristine handles this transparently.

A **cycle** ŌĆö module A imports B, B imports A ŌĆö is unresolvable. The framework throws
at boot with the offending pair. If you find yourself wanting a cycle, the modules
probably need to be split or merged.

## Configuration prerequisites

Some modules declare required configuration parameters. If you import them without
providing values, `pristine verify` and `pristine start` will fail with a clear error
naming the missing parameter.

For example, importing `@pristine-ts/auth0` requires:

```ts
// pristine.config.ts
kernelConfiguration: {
  "pristine.auth0.domain": "my-tenant.auth0.com",
  "pristine.auth0.audience": "https://api.my-app.com",
}
```

Or equivalent env vars. The module's README documents which values are required vs.
optional. Running `pristine verify` is the fastest way to discover what's missing ŌĆö its
ConfigurationCheck phase lists every required-but-unset parameter without trying to
resolve defaults.

## Transitive imports

You don't need to import a module's dependencies yourself. `HttpModule` already imports
`LoggingModule` internally, so this is enough:

```ts
importModules: [
  CoreModule,
  HttpModule,         // ŌåÉ LoggingModule, CommonModule, ConfigurationModule pulled in
]
```

But it's idiomatic to list the modules your code directly uses anyway. Helps a reader
understand what your AppModule is reaching for without grepping through transitive
imports.

## `importServices`

Separate from `importModules` is `importServices` ŌĆö a list of *class constructors* the
AppModule wants registered with the container. Two reasons to use it:

1. **Force-load classes that the import graph doesn't reach via barrel files.** If your
   `ProductsController` lives in a directory that nothing imports directly, the `@tag`
   side-effect won't run and tsyringe won't know the class exists. Listing it in
   `importServices` guarantees the file is imported.
2. **Document the contract**: an explicit list of every controller, command, listener, etc.
   the AppModule promises to wire up.

```ts
importServices: [
  ProductService,
  ProductsController,
  SyncCatalogCommand,
],
```

## Pitfalls

- **Importing twice doesn't double-register.** Pristine deduplicates by reference. Your
  AppModule can safely import a module that one of your own modules also imports.
- **A missing module is a runtime error, not a compile error.** If you call
  `container.resolve(SomeController)` but the controller's module isn't in
  `importModules`, you get "cannot resolve" at request time. The fastest way to debug:
  `pristine info` lists every loaded module so you can confirm what's registered.
- **The order of `importModules` only affects the order of `onInit` / `afterInit`
  hooks**, and only for siblings at the same dependency level. Don't rely on a specific
  sibling being initialized before another ŌĆö declare an explicit dependency instead.
