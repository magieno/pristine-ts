# Creating your own module

A module is just an object that implements `ModuleInterface`. The framework discovers
everything it needs from that one declaration: keyname, dependencies, configuration,
lifecycle hooks. This page covers every field, with the catalog demo's `CatalogModule`
as the running example.

> **You'll learn**
> - Every field on `ModuleInterface`
> - The lifecycle order: `onInit` → configuration → `afterInit` → request handling → `onShutdown`
> - When to use `providerRegistrations` vs. tag decorators

## The minimal module

```ts
// src/catalog/catalog.module.keyname.ts
export const CatalogModuleKeyname = "catalog";

// src/catalog/catalog.module.ts
import {ModuleInterface} from "@pristine-ts/common";
import {CatalogModuleKeyname} from "./catalog.module.keyname";

export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
};
```

That's a valid module. It does nothing yet — no services registered, no configuration, no
lifecycle work. But it's a real entry in the kernel's instantiated-modules graph and your
AppModule can import it.

## The keyname-in-its-own-file convention

You'll notice the keyname lives in a separate `.keyname.ts` file. This is convention:

```
src/catalog/
├── catalog.module.ts          (← imports keyname)
├── catalog.module.keyname.ts  (← just exports the string)
├── services/
│   └── product.service.ts     (← imports keyname for @moduleScoped)
└── ...
```

Why a separate file? Services use the keyname in `@moduleScoped(CatalogModuleKeyname)`.
Putting the keyname in `catalog.module.ts` would create a circular import (service
imports module, module imports service via `importServices`). The keyname file breaks
the cycle.

## Adding configuration

```ts
export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  configurationDefinitions: [
    {
      parameterName: `${CatalogModuleKeyname}.upstream-url`,
      isRequired: false,
      defaultValue: "https://example.com/catalog/feed",
    },
  ],
};
```

Configuration declarations live on the module so the kernel can pick them up at boot.
Services inject the resolved values via `@injectConfig("<parameterName>")` (from
`@pristine-ts/common`). Full coverage in the [Configuration chapter](../04-configuration/00.index.md).

## Importing dependencies

```ts
export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  importModules: [
    LoggingModule,        // CatalogModule's services use LogHandlerInterface
  ],
  configurationDefinitions: [/* ... */],
};
```

If your module's services depend on services from another module, declare the import.
Pristine deduplicates — if your AppModule also imports `LoggingModule` directly, it
loads once.

## Provider registrations

`providerRegistrations` is the explicit registration path — useful when you can't use
decorators (because the class isn't yours, or you want a factory). It accepts every shape
tsyringe accepts:

```ts
import {ProviderRegistration} from "@pristine-ts/common";

export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  providerRegistrations: [
    {token: "CatalogClock", useValue: () => Date.now()},
    {token: "CatalogSeed", useFactory: (c) => c.resolve(SeedService).load()},
  ],
};
```

Most app code uses tag decorators (`@tag(ServiceDefinitionTagEnum.Command)`,
`@injectable()`) instead. Provider registrations are the escape hatch for the cases
those don't cover.

## Lifecycle hooks

Three optional async hooks fire at well-defined points:

```ts
export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  // ...

  async onInit(container) {
    // Runs DURING module registration, BEFORE configuration is loaded.
    // Don't inject configuration values here — they aren't resolved yet.
    // Useful for: registering custom services programmatically.
  },

  async afterInit(container) {
    // Runs AFTER all modules' onInit and configuration is loaded.
    // Configuration values are available; the LogHandler is available.
    // Useful for: warming caches, opening connections, registering listeners
    //             whose registration depends on configuration.
    const apiClient = container.resolve(CatalogApiClient);
    await apiClient.warmUp();
  },

  async onShutdown(container) {
    // Runs when Kernel.stop() is called (SIGTERM/SIGINT under `pristine start`).
    // Runs in outer-to-inner order: AppModule first, leaf dependencies last.
    // Per-hook timeout defaults to 10s; configurable via Kernel.stop({perHookTimeoutMs}).
    const pool = container.resolve(DatabaseConnectionPool);
    await pool.drain();
    await pool.close();
  },
};
```

The lifecycle order at boot:
1. Recursively walk `importModules`.
2. For each module: register tagged services, register `providerRegistrations`, run
   `onInit`.
3. Load configuration definitions across all modules.
4. Run `afterInit` for each module.
5. Resolve `LogHandlerInterface` (smoke check that boot is fully done).
6. Dispatch commands / handle requests.

At shutdown:
1. SIGTERM/SIGINT triggers `Kernel.stop()`.
2. Walk modules outer-to-inner, awaiting each `onShutdown` (with per-hook timeout).
3. Process exits.

## A complete-ish module

```ts
import {ModuleInterface} from "@pristine-ts/common";
import {LoggingModule} from "@pristine-ts/logging";
import {ConfigurationModule} from "@pristine-ts/configuration";
import {CatalogModuleKeyname} from "./catalog.module.keyname";

export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,

  importModules: [
    ConfigurationModule,
    LoggingModule,
  ],

  configurationDefinitions: [
    {
      parameterName: `${CatalogModuleKeyname}.upstream-url`,
      isRequired: false,
      defaultValue: "https://example.com/catalog/feed",
    },
    {
      parameterName: `${CatalogModuleKeyname}.cache-ttl-seconds`,
      isRequired: false,
      defaultValue: 300,
    },
  ],

  async afterInit(container) {
    const logHandler = container.resolve("LogHandlerInterface");
    logHandler.info(`Catalog module initialized`);
  },

  async onShutdown(container) {
    const logHandler = container.resolve("LogHandlerInterface");
    logHandler.info(`Catalog module shutting down`);
  },
};
```

Services inside the module (`@injectable()` classes tagged with
`@moduleScoped(CatalogModuleKeyname)`) self-register when imported. The module itself
declares the configuration + lifecycle and lets the rest happen via decorators.

## Pitfalls

- **`onInit` can't inject configuration values.** Configuration is loaded *between* the
  onInit pass and the afterInit pass. Trying to inject `%catalog.upstream-url%` inside
  onInit gets you a runtime error.
- **`onShutdown` runs even if `afterInit` threw.** The framework wants to release as much
  as possible during shutdown, so don't assume the module is fully initialized inside
  onShutdown. Guard with `try/catch` if your shutdown code depends on something that
  might not exist.
- **A module without any tagged services is just an empty container.** Importing a
  module that doesn't tag/register/define anything is a no-op. Make sure your services
  are tagged (`@injectable()` + `@moduleScoped()` + optionally `@tag()`).
