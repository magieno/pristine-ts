# Identity providers

An identity provider takes a base identity — produced by an authenticator (JWT decoder,
session lookup, OAuth callback) — and **enriches** it. Maybe it adds the user's roles
from your database. Maybe it pulls fresh permissions from a remote service. The base
identity says "this is user 123"; the provider hydrates that into the full domain identity
your app reasons about.

**You don't need to register one to get authentication working.** If your app has no
identity providers tagged, the framework simply passes the authenticator's identity
through unchanged — exactly what a no-op provider would do. Define your own provider
when you want to attach app-specific fields on top of what the authenticator produced.

> **You'll learn**
> - What `IdentityProviderInterface` looks like
> - How the provider fits between authentication and authorization
> - What happens when no provider is registered
> - When to define one of your own

## The interface

```ts
import {IdentityInterface} from "@pristine-ts/common";

export interface IdentityProviderInterface {
  provide(identity: IdentityInterface): Promise<IdentityInterface>;
}
```

One method. Receives the base identity (id + claims map). Returns an enriched identity.

The returned identity must still implement `IdentityInterface` — `{id: string, claims:
{[k: string]: any}}` — but can extend the type with extra fields you assign to claims.

## Writing your own provider

A typical pattern: the authenticator decodes a JWT and produces an identity with the
user ID + token claims. Your provider hits the database to attach roles, team membership,
preferences:

```ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {IdentityInterface} from "@pristine-ts/common";
import {IdentityProviderInterface} from "@pristine-ts/security";

@tag(ServiceDefinitionTagEnum.IdentityProvider)
@injectable()
export class DatabaseIdentityProvider implements IdentityProviderInterface {
  constructor(private readonly users: UserRepository) {}

  async provide(identity: IdentityInterface): Promise<IdentityInterface> {
    const user = await this.users.findById(identity.id);
    if (user === undefined) {
      return identity;  // unknown user — pass through, let voters decide
    }
    return {
      id: identity.id,
      claims: {
        ...identity.claims,        // keep whatever the authenticator put there
        role: user.role,           // your enrichment
        teamId: user.teamId,
        emailVerified: user.emailVerified,
      },
    };
  }
}
```

`@tag(ServiceDefinitionTagEnum.IdentityProvider)` registers the class. The framework's
authentication pipeline calls `provide()` on every successfully-authenticated request.

## What if no provider is registered?

The framework injects identity providers using `@injectAll(..., {isOptional: true})`.
When zero classes are tagged `ServiceDefinitionTagEnum.IdentityProvider`, the resulting
array is simply empty — the authenticator's identity flows through to the authorization
layer unchanged. There's no default provider object, no placeholder, no extra hop.

Skip writing a provider entirely when:
- The claims your authenticator already attaches (e.g. JWT payload) are enough for
  every authorization check.
- You're in early development and want to see what the authenticator alone produces.

Define a provider when you need to attach data the authenticator can't see — typically
because it requires a database lookup or a call to another service.

## Multiple providers

You can register more than one. The authentication manager calls them in registration
order, threading the identity through each:

```
authenticator → provider A.provide() → provider B.provide() → ... → final identity
```

Each provider receives the *previous* provider's output. Use this to compose enrichments
(e.g. one provider hydrates from your user DB, another adds role claims from a separate
permissions service). Keep the chain short — every provider runs on every authenticated
request.

## What ends up in `claims`

The `claims` field is a free-form bag — whatever your authenticator puts there. With a
JWT authenticator, it's typically the decoded JWT payload (`sub`, `iat`, `exp`, custom
claims). With a session-based authenticator, it might be the session's stored attributes.

Your identity provider can:
- Read from `claims` to look up the user (`identity.id` is conventionally the user ID).
- Add to `claims` to attach app-specific data your downstream code needs.

Voters and controllers read from `claims` to make decisions. Keep it small — every claim
travels with every request and every authorization check.

## Pitfalls

- **Don't return `null` or throw on unknown users.** Return the unchanged identity and
  let the authorization layer (voters) decide. Throwing inside a provider is treated as
  authentication failure by some authenticators.
- **Don't do expensive work every request.** `provide()` runs on every authenticated
  request. If you need to hit the DB to enrich, cache aggressively. A bad provider can
  10x your request latency.
- **The provider doesn't replace authentication.** It runs *after* authentication has
  already succeeded. You can't use it to reject a request — only to add data.
