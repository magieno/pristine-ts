# OAuth2

OAuth2 is the de-facto standard for "let users sign in with their existing account from
some other service" — Google, GitHub, your enterprise IdP, etc. Pristine doesn't ship a
generic OAuth2 authenticator (the protocol is too varied), but it does ship pre-wired
adapters for the two most common providers via dedicated packages:

- **`@pristine-ts/auth0`** — for [Auth0](https://auth0.com/) tenants
- **`@pristine-ts/aws-cognito`** — for [AWS Cognito](https://aws.amazon.com/cognito/)
  user pools

Both work the same way: import the module, configure the provider, decorate your routes
with the provided authenticator. The provider handles the OIDC dance (token validation,
JWKS fetching, claim mapping) for you.

> **You'll learn**
> - Which OAuth2 providers Pristine ships
> - How to configure Auth0 as an example
> - When to use a JWT authenticator directly vs. an OAuth2-specific authenticator

## Auth0 example

Install:

```sh
npm install @pristine-ts/auth0
```

Import the module:

```ts
import {Auth0Module} from "@pristine-ts/auth0";

export const AppModule: AppModuleInterface = {
  keyname: "my-app",
  importModules: [
    CoreModule,
    HttpModule,
    NetworkingModule,
    Auth0Module,
    // ...
  ],
  // ...
};
```

Configure (in `pristine.config.ts` or via env var):

```ts
kernelConfiguration: {
  "pristine.auth0.domain": "my-tenant.auth0.com",
  "pristine.auth0.audience": "https://api.my-app.com",
},
```

Or env:

```sh
PRISTINE_AUTH0_DOMAIN=my-tenant.auth0.com \
PRISTINE_AUTH0_AUDIENCE=https://api.my-app.com \
  pristine start
```

Protect routes:

```ts
import {Auth0Authenticator} from "@pristine-ts/auth0";

@authenticator(Auth0Authenticator)
@injectable()
@controller("/api")
export class ApiController {
  @route(HttpMethod.Get, "/me")
  me(@identity() identity: IdentityInterface) {
    return {
      userId: identity.id,
      email: identity.claims.email,
      roles: identity.claims["https://my-app.com/roles"],
    };
  }
}
```

`Auth0Authenticator` handles:
- Reading the `Authorization: Bearer <token>` header.
- Validating the token's signature against Auth0's JWKS endpoint (with caching).
- Validating audience + issuer + expiration.
- Mapping the JWT's `sub` claim to `identity.id` and the entire payload to `identity.claims`.

If any step fails, it throws → framework returns 401.

## AWS Cognito

Same pattern with `@pristine-ts/aws-cognito`:

```ts
import {AwsCognitoModule, AwsCognitoAuthenticator} from "@pristine-ts/aws-cognito";

// AppModule imports AwsCognitoModule
// kernelConfiguration: { "pristine.aws.cognito.user-pool-id": "..." , ... }

@authenticator(AwsCognitoAuthenticator)
@controller("/api")
export class ApiController { /* ... */ }
```

The configuration parameters differ (user pool ID, AWS region) but the protected-route
pattern is identical.

## Generic JWT (when no module fits)

If you're using an OAuth2 provider Pristine doesn't ship a module for (Okta, Keycloak,
your own IdP), use the generic JWT authenticator. It validates a JWT against a
configured JWKS endpoint or public key, without provider-specific knowledge. Covered in
the next section.

## How sessions vs. tokens differ

OAuth2 in API contexts is typically **token-based** (every request carries the JWT in the
Authorization header). The token IS the credential — no server-side session to invalidate
on logout.

If your app needs server-side sessions (so you can force-revoke), wrap the OAuth2 token
exchange in a session-issuance step: after the OAuth2 dance, mint your own session
cookie pointing at a server-stored record. Then write a session authenticator that reads
the cookie. The framework doesn't ship session middleware out of the box; build it on
top of the authenticator pattern.

## Pitfalls

- **JWKS fetching is async + cached.** First request after process boot pays a JWKS
  lookup; subsequent requests reuse the cached keys. The cache TTL is provider-specific.
  If you rotate keys, expect a brief window of 401s.
- **`audience` mismatches are silent.** A token issued for `audience=A` and validated
  against `audience=B` fails with "invalid token" — the underlying error is "audience
  mismatch" but the framework just says "401." Double-check both sides match exactly
  (including the `https://`).
- **Don't trust `identity.claims` blindly.** Anything in claims came from the token
  issuer. If you're using the same token for multiple apps and one of them controls
  custom claim contents, treat that data as user-controlled input.
