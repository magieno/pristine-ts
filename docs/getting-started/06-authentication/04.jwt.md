# JWT

`@pristine-ts/jwt` ships a generic JWT authenticator that validates `Authorization:
Bearer <token>` headers against a configured public key or JWKS endpoint. Use it when:

- Your auth provider isn't covered by `@pristine-ts/auth0` or `@pristine-ts/aws-cognito`
  (Okta, Keycloak, your own IdP).
- You're minting tokens yourself (service-to-service, internal apps).
- You want a JWT layer underneath your own custom claim-mapping logic.

> **You'll learn**
> - How to wire the JWT module
> - How to configure key sources (static, JWKS)
> - When to mint tokens yourself vs. delegate to an IdP

## Installing and configuring

```sh
npm install @pristine-ts/jwt
```

```ts
import {JwtModule} from "@pristine-ts/jwt";

export const AppModule: AppModuleInterface = {
  keyname: "my-app",
  importModules: [CoreModule, HttpModule, NetworkingModule, JwtModule],
  // ...
};
```

Two key-source modes:

**Static public key** — for cases where you know the verifying key statically (e.g.
service-to-service with a shared key):

```ts
kernelConfiguration: {
  "pristine.jwt.public-key": "-----BEGIN PUBLIC KEY-----\n...\n-----END PUBLIC KEY-----",
  "pristine.jwt.algorithm": "RS256",
}
```

**JWKS endpoint** — for cases where the IdP rotates keys and publishes them via JWKS
(every OIDC provider, including custom ones):

```ts
kernelConfiguration: {
  "pristine.jwt.jwks-uri": "https://my-idp.com/.well-known/jwks.json",
}
```

## Protecting routes

```ts
import {JwtAuthenticator} from "@pristine-ts/jwt";

@authenticator(JwtAuthenticator)
@injectable()
@controller("/api")
export class ApiController {
  @route(HttpMethod.Get, "/me")
  me(@identity() identity: IdentityInterface) {
    return {sub: identity.id, claims: identity.claims};
  }
}
```

Standard `@authenticator` decorator pattern. Same as Auth0/Cognito — only the
authenticator class differs.

## What the authenticator validates

`JwtAuthenticator.authenticate(request)` checks:
1. `Authorization: Bearer <token>` header is present.
2. Token is a well-formed JWT (header.payload.signature).
3. Signature verifies against the configured key (static or JWKS-fetched).
4. `exp` claim hasn't passed.
5. `nbf` claim (if present) isn't in the future.

If you need to also validate `iss` or `aud`, configure them:

```ts
kernelConfiguration: {
  "pristine.jwt.jwks-uri": "...",
  "pristine.jwt.expected-issuer": "https://my-idp.com",
  "pristine.jwt.expected-audience": "https://api.my-app.com",
}
```

On success, `identity.id` becomes the token's `sub` and `identity.claims` is the entire
JWT payload.

## Minting tokens

The `@pristine-ts/jwt` module also exposes `JwtManager`, an injectable service for
*issuing* tokens — useful for service-to-service auth where your app produces the tokens:

```ts
import {JwtManager} from "@pristine-ts/jwt";

@injectable()
export class TokenService {
  constructor(private readonly jwt: JwtManager) {}

  async issueServiceToken(serviceId: string): Promise<string> {
    return this.jwt.sign(
      {sub: serviceId, scope: "internal"},
      {expiresIn: "5m"},
    );
  }
}
```

You need a signing key configured (`pristine.jwt.private-key` + `pristine.jwt.algorithm`)
separately from the verifying key. Common production setup: app keeps the private key
locally, distributes the public key to consumers.

## Pitfalls

- **Don't put long-lived secrets in JWTs.** JWT payloads are base64-encoded, NOT encrypted.
  Anyone with the token can read the claims. Treat the token like a signed (not sealed)
  envelope.
- **`alg: none` is rejected by the authenticator.** Some JWT libraries historically had
  bugs accepting unsigned tokens. We refuse them outright — only the configured algorithm
  is accepted.
- **JWKS caches per process.** If you rotate keys, in-flight token validation succeeds for
  the cache TTL window. For zero-downtime rotation: publish the new key in JWKS first,
  wait for the cache TTL, then start signing with it.
