# The log handler

`LogHandlerInterface` is the single entrypoint every Pristine class uses to produce log
output. It abstracts away the actual transport (console, file, CloudWatch, Sentry)
behind seven severity-prefixed methods.

> **You'll learn**
> - The six severity levels and when to use each
> - How to inject `LogHandlerInterface`
> - What the `data` parameter is for
> - Why everything goes through this abstraction

## The interface

```ts
export interface LogHandlerInterface {
  critical(message: string, data?: LogData): void;
  error(message: string, data?: LogData): void;
  warning(message: string, data?: LogData): void;
  notice(message: string, data?: LogData): void;
  info(message: string, data?: LogData): void;
  debug(message: string, data?: LogData): void;
  terminate(): void;
}
```

Six log methods, one teardown method. The severities map to standard syslog levels.

## Injecting and using

```ts
import {inject, injectable} from "tsyringe";
import {LogHandlerInterface} from "@pristine-ts/logging";

@injectable()
export class ProductService {
  constructor(
    @inject("LogHandlerInterface") private readonly logHandler: LogHandlerInterface,
  ) {}

  upsert(product: Product) {
    this.logHandler.info(`Upserted product '${product.id}' (${product.name})`);
  }
}
```

Use the string `"LogHandlerInterface"` as the inject token. The framework registers the
LogHandler under this token at boot.

The actual LogHandler implementation is in `packages/logging/src/handlers/log.handler.ts`.
You don't inject the concrete class â€” the interface lets the framework swap transports
without touching consumer code.

## Severity levels

| Level | When to use |
|---|---|
| `critical` | The app can't continue. About to crash, or just did. Examples: "Cannot connect to required upstream", "Out of memory". |
| `error` | A specific operation failed but the app keeps running. Examples: "Failed to send email", "Database query timed out". |
| `warning` | Something unexpected happened but no operation failed. Examples: "Config key X is deprecated", "Retry attempt 3/5". |
| `notice` | A routine event worth recording for audit/correlation. Examples: "Request completed in 142ms", "User logged in". |
| `info` | Routine application-level events. Default for `console.log`-style usage. |
| `debug` | Per-operation diagnostic detail. Off in production by default. |

There's no `fatal` or `trace` â€” pick the closest match. Most app code lives at `info`,
`warning`, and `error`.

## The `data` parameter

The second argument carries structured data for the transport to serialize. Two
conventional fields the built-in loggers know about:

```ts
this.logHandler.info("Synced product catalog", {
  highlights: {                // surfaced prominently by transports
    productCount: 142,
    durationMs: 853,
  },
  extra: {                     // bulk diagnostic data
    sourceUrl: this.upstreamUrl,
    syncedAt: new Date().toISOString(),
  },
});
```

`highlights` are typically rendered inline with the log line (or as searchable fields in
JSON output). `extra` is appended verbatim. Use `highlights` for cardinality-friendly,
queryable fields; use `extra` for context that's nice to have but not search-keyed.

### `eventId` is filled in automatically

Inside any code that runs as part of an event (HTTP request, queue message, scheduled
command, etc.), the LogHandler attaches the active event's id to every log entry â€” you
don't need to thread `eventId: request.id` through every call site. The framework
installs an `EventContext` (an `AsyncLocalStorage`-backed slot) at the per-event boundary
in the event pipeline, and the LogHandler reads from it as a fallback when the caller
didn't pass an explicit `eventId`.

Two consequences worth knowing:

- **Outside an event** (boot-time logs, code paths reached without going through
  `kernel.handle`), the eventId is omitted â€” same as before.
- **Explicit values still win.** If you want to log against a *different* event id (e.g.
  during a fan-out where you want a parent log line tagged with a synthetic correlation
  id), passing `eventId` in the data argument overrides the auto-resolution.

## Why this abstraction?

You could `console.log` directly. The framework's LogHandler exists for three reasons:

1. **Multiple destinations**: same call can fan out to console + file + CloudWatch.
2. **Severity filtering**: the configured threshold (`pristine.logging.logSeverityLevelConfiguration`)
   suppresses calls below the limit without code changes.
3. **Structured-data discipline**: transports can serialize `data` consistently (JSON for
   CloudWatch, formatted text for console) because the contract is the same everywhere.

## Pitfalls

- **Don't use the global `console.log`.** It bypasses everything â€” severity filtering,
  structured fields, multiple destinations. Track-down-able as a CI grep for
  `console\.log` in `src/`.
- **`critical` doesn't crash the process.** It's a severity classification, not a control-flow
  primitive. If you need to crash, `throw` separately.
- **`terminate()` is the framework's call, not yours.** It's invoked during graceful
  shutdown to flush pending log writes. Don't call it from app code.
