# Loggers

A logger is an actual transport ŌĆö it takes a log entry and sends it somewhere (stdout, a
file, CloudWatch, Sentry). The LogHandler is the front door; loggers are the back doors.

Multiple loggers can be registered at once. Every log call fans out to all of them, and
each one decides for itself whether the entry meets its severity threshold and how to
format it.

> **You'll learn**
> - The actual `LoggerInterface` shape (it's stream-based, not method-based)
> - How `BaseLogger` saves you from re-implementing the stack/throttle/threshold logic
> - How crash isolation works ŌĆö a throwing logger never takes down the app
> - When to extend `BaseLogger` vs. implement `LoggerInterface` directly

## The interface

```ts
import {Readable} from "stream";

export interface LoggerInterface {
  readableStream?: Readable;
  isActive(): boolean;
  terminate(): void;
}
```

Three things, all surprisingly low-level:
- **`readableStream`** ŌĆö a Node.js object-mode `Readable`. The LogHandler `.push()`es
  every log entry into this stream; the logger consumes it via a `'data'` listener.
- **`isActive()`** ŌĆö return `false` and the LogHandler skips the push entirely.
- **`terminate()`** ŌĆö called during graceful shutdown. Destroy the stream and flush any
  pending writes here.

That's it. There's no `log()` method on the interface ŌĆö the dispatch is *via the stream*,
not via a function call. This is deliberate: streams give you backpressure for free, and
they let the LogHandler push synchronously without awaiting each transport.

## Why streams (not async methods)

A pure async `log(): Promise<void>` interface would force the LogHandler to either
`await` every dispatch (slow remote loggers block every request) or fire-and-forget
(silent backpressure problems). Streams give us a third option: the LogHandler pushes
into the stream synchronously, and each logger consumes from its own stream at its own
pace. The LogHandler never blocks on a slow transport.

## The `BaseLogger` shortcut

Implementing `LoggerInterface` directly is a lot of boilerplate ŌĆö you have to set up the
Readable, attach a `'data'` listener, decide whether to use the framework's stacking and
throttling logic, etc. **Every built-in logger extends `BaseLogger`**, and so should
yours unless you have a specific reason not to.

`BaseLogger` provides:
- **Stacked logging** ŌĆö buffer N low-severity entries and flush them only when an error
  arrives, so you get rich context without spamming logs in normal operation.
- **Severity threshold filtering** (per-logger).
- **Format helpers** (`getFormattedOutputLog`) that handle JSON / pretty / depth-truncated
  output.
- **Crash isolation** (see below).

The contract: subclass `BaseLogger`, implement `initialize()` to wire up
`this.readableStream`, and implement `log(log: LogModel): void` to do the actual write.

## Writing your own logger (the recommended way)

```ts
import {injectConfig, tag, ServiceDefinitionTagEnum, moduleScoped} from "@pristine-ts/common";
import {injectable, singleton} from "tsyringe";
import {BaseLogger, LoggerInterface, LogModel, LoggingModuleKeyname, OutputModeEnum, SeverityEnum} from "@pristine-ts/logging";
import {Readable} from "stream";

@moduleScoped(LoggingModuleKeyname)
@singleton()
@tag(ServiceDefinitionTagEnum.Logger)
@injectable()
export class CloudWatchLogger extends BaseLogger implements LoggerInterface {
  public readableStream?: Readable;

  constructor(
    @injectConfig("pristine.logging.numberOfStackedLogs") numberOfStackedLogs: number,
    @injectConfig("pristine.logging.logSeverityLevelConfiguration") logSeverityLevelConfiguration: number,
    @injectConfig("pristine.logging.logDebugDepthConfiguration") logDebugDepth: number,
    @injectConfig("pristine.logging.logInfoDepthConfiguration") logInfoDepth: number,
    @injectConfig("pristine.logging.logNoticeDepthConfiguration") logNoticeDepth: number,
    @injectConfig("pristine.logging.logWarningDepthConfiguration") logWarningDepth: number,
    @injectConfig("pristine.logging.logErrorDepthConfiguration") logErrorDepth: number,
    @injectConfig("pristine.logging.logCriticalDepthConfiguration") logCriticalDepth: number,
    @injectConfig("my.cloudwatch.activated") isActivated: boolean,
  ) {
    super(numberOfStackedLogs, logSeverityLevelConfiguration,
      logDebugDepth, logInfoDepth, logNoticeDepth, logWarningDepth, logErrorDepth, logCriticalDepth,
      isActivated, OutputModeEnum.Json);

    this.initialize();
  }

  protected initialize(): void {
    if (this.isActive()) {
      // Inherited from BaseLogger ŌĆö sets up a Readable wired to captureLog with
      // crash isolation already in place. Always prefer this over building one yourself.
      this.readableStream = this.createSafeReadableStream();
    }
  }

  protected log(log: LogModel): void {
    // The actual write. BaseLogger has already filtered by severity threshold and
    // run the stacking logic; by the time you're here, this entry should be shipped.
    const formatted = this.getFormattedOutputLog(log);
    this.cloudWatchClient.putLogEvents({...});
  }

  terminate(): void {
    this.readableStream?.destroy();
    // Flush any remaining batches your client buffers.
  }
}
```

The `@tag(ServiceDefinitionTagEnum.Logger)` self-registers the class. As long as your
AppModule's import graph reaches the file, the LogHandler picks it up and starts pushing
entries into your stream.

## Crash isolation

A failing logger does **not** crash the LogHandler call site, the calling code, or the
process. There are two layers of protection:

1. **`BaseLogger.createSafeReadableStream()`** wraps the stream's `'data'` listener in
   try/catch and attaches an `'error'` listener. If your `log()` throws (or the stream
   errors), the failure is written to `process.stderr` with a clear
   `[pristine][logger:<ClassName>] ...` prefix and execution continues.

2. **`LogHandler.log()`** wraps every per-logger `.push()` in try/catch. Even if a custom
   logger doesn't extend `BaseLogger` (or builds a stream by hand), one bad logger
   cannot poison the others or propagate an error back to the user code that called
   `logHandler.info(...)`.

Either way, **a throw from a single logger means a stderr line, not a process crash, not
a dropped request.** This was not the case in earlier versions ŌĆö if you're upgrading from
a release where you wrapped your own `log()` body defensively, you can remove that
boilerplate.

The tradeoff: errors are absorbed, not surfaced. Look for `[pristine][logger:...]` in
stderr if a transport seems quietly broken.

## Implementing without `BaseLogger`

If you really need full control (e.g. you're integrating a transport that has its own
stream abstraction and you want to bridge directly), implement `LoggerInterface`
yourself. You give up the framework's stacking, throttling, and crash isolation ŌĆö you'll
need to add try/catch in your `'data'` listener manually if you want the same safety:

```ts
@tag(ServiceDefinitionTagEnum.Logger)
@injectable()
export class CustomStreamLogger implements LoggerInterface {
  public readableStream?: Readable;

  constructor() {
    this.readableStream = new Readable({objectMode: true, read() { return true; }});
    this.readableStream.on("data", (chunk) => {
      try {
        // your transport call
      } catch (error) {
        process.stderr.write(`[my-logger] ${(error as Error).message}\n`);
      }
    });
    this.readableStream.on("error", (error) => {
      process.stderr.write(`[my-logger] stream error: ${error.message}\n`);
    });
  }

  isActive(): boolean { return true; }
  terminate(): void { this.readableStream?.destroy(); }
}
```

Use this path only when `BaseLogger` doesn't fit. For most use cases ŌĆö including remote
transports, batched writers, and custom formats ŌĆö the `BaseLogger` template above is
both shorter and safer.

## Built-in loggers

| Logger | Where it writes | Configuration |
|---|---|---|
| `ConsoleLogger` | `process.stdout` / `process.stderr` | `pristine.logging.consoleLoggerActivated`, `consoleLoggerOutputMode` |
| `FileLogger` | A file on disk | `pristine.logging.fileLoggerActivated`, `fileLoggerOutputMode`, `filePath` |

Both use `BaseLogger`. `ConsoleLogger` is activated by default; `FileLogger` is not.
Override via config (see [Console logger](03.console-logger.md) and
[File logger](04.file-logger.md)).

## Diagnostic mode

When `pristine.logging.activateDiagnostics` is set to `true`, every log entry includes a
stack trace + extra runtime introspection. This is **expensive** ŌĆö meaningfully so on
hot paths. Only enable for short investigations, never in production by default.

## Pitfalls

- **Don't recurse into the LogHandler from inside a logger.** Your `log()` method
  shouldn't itself call `this.logHandler.error(...)` if the network failed ŌĆö that's an
  infinite loop. Write to stderr directly if you really need to surface a logger failure.
  (`BaseLogger.reportLoggerFailure` is the framework's own way of doing exactly this.)
- **Forgetting to call `this.initialize()` from your constructor.** Without it,
  `readableStream` is never created and the LogHandler's pushes are silently no-ops
  (the optional chaining on `logger.readableStream?.push(log)` swallows them). Symptom:
  no logs from your logger, no errors either.
- **Activating without bumping severity threshold.** If your logger is active but its
  threshold is `Critical` and you're sending mostly `Info` logs, nothing comes out.
  Loggers and the LogHandler each have their own threshold ŌĆö they're ANDed.
