# Loggers

A logger is an actual transport — it takes a log entry and sends it somewhere (stdout, a
file, CloudWatch, Sentry). The LogHandler is the front door; loggers are the back doors.

Multiple loggers can be registered at once. Every log call fans out to all of them, each
applying its own severity filter and output format.

> **You'll learn**
> - What `LoggerInterface` looks like
> - How to write a logger for a destination the framework doesn't ship
> - How loggers are registered with the kernel

## The interface

```ts
export interface LoggerInterface {
  readonly identifier: string;

  log(log: LogModel): Promise<void>;
  terminate(): void;
}
```

Three things:
- **`identifier`** — a unique name (used in diagnostics and to disambiguate loggers).
- **`log(log: LogModel)`** — receives every log entry the LogHandler dispatches and ships
  it wherever this logger ships things.
- **`terminate()`** — called during graceful shutdown. Flush pending writes here.

A logger only ships a log entry if the entry's severity meets the logger's own threshold.
Each logger can independently filter — useful when you want, say, errors in CloudWatch
but everything in a local file.

## Built-in loggers

| Logger | Where it writes | Configuration |
|---|---|---|
| `ConsoleLogger` | `process.stdout` / `process.stderr` | `pristine.logging.consoleLoggerActivated`, `consoleLoggerOutputMode` |
| `FileLogger` | A file on disk | `pristine.logging.fileLoggerActivated`, `fileLoggerOutputMode`, `filePath` |

Both are activated by default for `ConsoleLogger`, deactivated by default for
`FileLogger`. Override via config (see [Console logger](03.console-logger.md) and
[File logger](04.file-logger.md)).

## Writing your own logger

For destinations the framework doesn't ship — CloudWatch, Sentry, Datadog, your own log
service — implement `LoggerInterface` and register it with `@tag(ServiceDefinitionTagEnum.Logger)`:

```ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {LoggerInterface, LogModel} from "@pristine-ts/logging";

@tag(ServiceDefinitionTagEnum.Logger)
@injectable()
export class CloudWatchLogger implements LoggerInterface {
  readonly identifier = "cloudwatch";

  async log(log: LogModel): Promise<void> {
    // Ship the log entry. log.severity, log.message, log.date, log.extra are all available.
    await this.cloudWatchClient.putLogEvents({...});
  }

  terminate(): void {
    // Flush any buffered batches here.
  }
}
```

The `@tag` decorator self-registers the class as a logger. As long as your AppModule's
import graph reaches the file, the LogHandler picks it up.

## How fan-out works

Every call to `logHandler.info(...)` does:
1. Build a `LogModel` from the message + data + current trace context.
2. For each registered `Logger`, check if the model's severity meets the logger's
   threshold. If yes, dispatch.
3. Wait for all dispatches to resolve (or not — see "Async loggers" below).

The dispatch is per-logger; one slow logger doesn't block the others.

## Async loggers

`log()` returns a Promise. The LogHandler awaits it before returning, which means a slow
remote logger (e.g. one that blocks on a network round-trip) blocks the caller. Best
practice:

- **Local loggers** (console, file) should `await` their writes — losing a log line on
  process crash is worse than the few microseconds of latency.
- **Remote loggers** (CloudWatch, etc.) should fire-and-forget *or* buffer locally and
  flush in batches. If `log()` blocks every app operation on a network call, latency
  cascades.

## Diagnostic mode

When `pristine.logging.activateDiagnostics` is set to `true`, every log entry includes a
stack trace + extra runtime introspection. This is **expensive** — meaningfully so on
hot paths. Only enable for short investigations, never in production by default.

## Pitfalls

- **A logger that throws synchronously kills the log call.** Wrap your `log()` body in
  try/catch and absorb errors — a broken logger shouldn't break the app it's logging.
- **Don't recurse into the LogHandler from inside a logger.** Your `log()` method
  shouldn't itself call `this.logHandler.error(...)` if the network failed — that's an
  infinite loop. Write to stderr directly if you really need to surface a logger failure.
- **The `identifier` must be unique.** Two loggers with the same identifier will both be
  registered, but diagnostics that reference one by identifier become ambiguous.
