# Console logger

`ConsoleLogger` is the default Pristine logger. It writes log entries to `stdout` (or
`stderr` for `error`/`critical`) using either JSON or simple-text output. Activated by
default whenever `LoggingModule` is imported, which is "always" for any non-trivial app.

## Configuration

| Parameter | Default | Effect |
|---|---|---|
| `pristine.logging.consoleLoggerActivated` | `true` | Master switch. Set to `false` to disable. |
| `pristine.logging.consoleLoggerOutputMode` | `OutputModeEnum.Json` | `"Json"` or `"Simple"`. |
| `pristine.logging.logSeverityLevelConfiguration` | `SeverityEnum.Info` | Minimum severity logged — calls below this are dropped. |

Set via `pristine.config.ts` `kernelConfiguration`, env vars, or runtime overrides. The
common-case env vars:

- `PRISTINE_LOGGING_CONSOLE_LOGGER_ACTIVATED=true`
- `PRISTINE_LOGGING_CONSOLE_LOGGER_OUTPUT_MODE=Simple`
- `PRISTINE_LOGGING_LOG_SEVERITY_LEVEL_CONFIGURATION=3`  (3 = Error)

## Output modes

**`OutputModeEnum.Json`** (default for prod): emits one JSON object per line. Suitable
for log aggregators that parse structured data (CloudWatch, Splunk, Loki).

```json
{"severity":"INFO","message":"Synced 142 products","date":"2026-05-12T...","module":"application","traceId":"...","highlights":{"productCount":142},"extra":{...}}
```

**`OutputModeEnum.Simple`** (default for the CLI in interactive mode): emits a single
human-readable line per log entry. Suitable for local development.

```
2026-05-12T... [INFO] Synced 142 products
```

The CLI defaults to `Simple` when an HTTP-less command runs (so `pristine help` doesn't
flood you with JSON) and lets your AppModule's configuration take over otherwise. For
production deploys with `pristine start`, set the output mode to `Json` via
`kernelConfiguration` or env var.

## Severity threshold

Calls below `logSeverityLevelConfiguration` are silently dropped at the LogHandler layer
— they never reach the ConsoleLogger. This is the threshold knob you turn for verbosity
without changing code:

```ts
// pristine.config.ts
kernelConfiguration: {
  "pristine.logging.logSeverityLevelConfiguration": 5,  // SeverityEnum.Debug
}
```

Threshold values follow `SeverityEnum`:
- `0` = Critical (only criticals logged)
- `1` = Error
- `2` = Warning
- `3` = Notice
- `4` = Info (default)
- `5` = Debug (everything)

## Depth configuration

For deeply nested data, the ConsoleLogger truncates at a configurable depth per severity:

| Parameter | Default depth |
|---|---|
| `pristine.logging.logDebugDepthConfiguration` | 5 |
| `pristine.logging.logInfoDepthConfiguration` | 5 |
| `pristine.logging.logNoticeDepthConfiguration` | 5 |
| `pristine.logging.logWarningDepthConfiguration` | 5 |
| `pristine.logging.logErrorDepthConfiguration` | 5 |
| `pristine.logging.logCriticalDepthConfiguration` | 5 |

Useful when your `extra` data contains a tracing span with deeply nested children that
would otherwise dump a wall of JSON. Per-severity depth lets you cap chattier severities
without throttling critical errors.

## Rate limiting

`pristine.logging.maximumLogsPerSecond` (default `50`) caps how many entries per second
the ConsoleLogger ships. Excess entries get dropped. This prevents a tight loop from
flooding the log pipeline (and your APM bill).

If you genuinely need higher throughput, raise it. If you're hitting the limit, that's
usually a symptom — find the loop logging too much.

## Pitfalls

- **`Simple` mode loses structured fields.** The flat text format can't represent
  `highlights` / `extra` as queryable fields. Use `Json` in any environment where you
  search logs.
- **Rate limit drops are silent.** You won't see "100 logs dropped" — they just don't
  appear. If you suspect drops, raise `maximumLogsPerSecond` temporarily and re-run.
- **Stack traces aren't included by default.** Set `pristine.logging.activateDiagnostics`
  to `true` to attach them. Expensive — only enable for short investigations.
