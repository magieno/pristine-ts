# File logger

`FileLogger` writes log entries to a file on disk instead of (or in addition to) stdout.
Deactivated by default — most apps either stream stdout to a log aggregator or run in
containers where filesystems are ephemeral. File logging is useful for: development
debugging, single-machine deployments without an aggregator, and audit trails where
process crashes must not lose recent entries.

## Configuration

| Parameter | Default | Effect |
|---|---|---|
| `pristine.logging.fileLoggerActivated` | `false` | Master switch. |
| `pristine.logging.fileLoggerOutputMode` | `OutputModeEnum.Json` | `"Json"` or `"Simple"`. |
| `pristine.logging.fileLoggerPretty` | `false` | When true, JSON is indented for human reading. |
| `pristine.logging.filePath` | `"./logs.txt"` | Where to write. Relative to process cwd. |

Activate via env var or config:

```sh
PRISTINE_LOGGING_FILE_LOGGER_ACTIVATED=true \
PRISTINE_LOGGING_FILE_PATH=/var/log/my-app.log \
  pristine start
```

Or in `pristine.config.ts`:

```ts
kernelConfiguration: {
  "pristine.logging.fileLoggerActivated": true,
  "pristine.logging.filePath": "/var/log/my-app.log",
}
```

## Output format

Same `Json` / `Simple` modes as ConsoleLogger. With `Simple`, you get one human-readable
line per entry. With `Json` (default for the file logger), each line is a JSON object —
this is what `jq` likes:

```sh
tail -f /var/log/my-app.log | jq 'select(.severity == "ERROR")'
```

`fileLoggerPretty: true` enables multi-line indented JSON. Easier to read by eye, harder
to grep. Don't combine pretty + tail with line-oriented tools like `grep`.

## Rotation

The FileLogger doesn't rotate. It opens the file in append mode and writes forever. For
production, run an external rotator like `logrotate`:

```
/var/log/my-app.log {
  daily
  rotate 14
  compress
  delaycompress
  missingok
  notifempty
  copytruncate
}
```

`copytruncate` is the key directive — it lets logrotate move the contents to a rotated
file and truncate the original *without* the Pristine process needing to reopen the file
descriptor. Most file loggers handle this gracefully; we do.

For Docker / Kubernetes deployments, prefer the ConsoleLogger + your platform's log
shipper. File rotation in containers is a hazard.

## Combining with the ConsoleLogger

Both loggers can be active at once. Each filters independently:

```ts
kernelConfiguration: {
  "pristine.logging.consoleLoggerActivated": true,
  "pristine.logging.consoleLoggerOutputMode": "Simple",
  "pristine.logging.fileLoggerActivated": true,
  "pristine.logging.fileLoggerOutputMode": "Json",
  "pristine.logging.filePath": "./debug.log",
}
```

Stdout gets human-readable output (great when you're watching the terminal); the file
gets structured JSON (great for retrospective analysis).

## Pitfalls

- **The file is opened during the first `log()` call**, not at boot. If the path is
  invalid or unwritable, you won't see the error until the first log entry. Test
  permissions before you ship.
- **Concurrent processes appending to the same file is undefined behavior.** Don't run
  two instances of the same Pristine app pointing at the same log file. Use separate
  paths.
- **Log entries flush per-write.** No buffering. This is good for crash-safety, bad for
  throughput. If you're logging at >10k/sec, switch to a logger that batches.
