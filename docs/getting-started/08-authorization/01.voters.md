# Voters

Authorization in Pristine is voter-based. To decide whether an identity (a user, a
service account, anything with an `IdentityInterface`) can perform an action on a
resource, the framework asks every registered voter for an opinion. Each voter answers
**grant**, **deny**, or **abstain**. The PermissionManager combines those votes into a
final decision.

This pattern keeps authorization logic close to the domain it protects. Each voter knows
about *one kind of resource* and *the rules that apply to it*. You don't have to grow a
single monolithic policy.

> **You'll learn**
> - What `VoterInterface` looks like
> - How to write a voter for one of your domain types
> - The grant / deny / abstain semantics

## The interface

```ts
import {VoteEnum} from "@pristine-ts/security";
import {IdentityInterface} from "@pristine-ts/common";

export interface VoterInterface {
  supports(resource: object): boolean;
  vote(identity: IdentityInterface, action: string, resource: object): Promise<VoteEnum>;
}
```

Two methods:
- **`supports(resource)`** — Does this voter know how to vote on this resource? If `false`,
  the manager skips this voter entirely.
- **`vote(identity, action, resource)`** — Given an identity, an action string, and a
  resource, return one of three votes.

## The three votes

```ts
export enum VoteEnum {
  Granted = "GRANTED",
  Denied = "DENIED",
  Abstain = "ABSTAIN",
}
```

- **`Granted`** — "I think this identity should be allowed." Affirmative answer.
- **`Denied`** — "I think this identity should NOT be allowed." Affirmative refusal.
- **`Abstain`** — "I don't have an opinion." Used when a voter could check but the rules
  don't apply (e.g. the action isn't one this voter cares about).

The PermissionManager merges votes per a `VotingStrategyEnum` (the default,
`DenyOnUnanimousAbstention`, refuses if every voter abstains). See
[The permission manager](02.permission-manager.md) for the strategies.

## Writing a voter for the catalog

Say we want to restrict who can `delete` products. The rule: any user can `list` and
`get`; only users whose `identity.id` matches a designated admin list can `delete`.

```ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {VoterInterface, VoteEnum} from "@pristine-ts/security";
import {IdentityInterface} from "@pristine-ts/common";
import {Product} from "../models/product";

@tag(ServiceDefinitionTagEnum.Voter)
@injectable()
export class ProductVoter implements VoterInterface {
  private readonly admins: ReadonlySet<string> = new Set(["alice", "bob"]);

  supports(resource: object): boolean {
    return resource instanceof Product;
  }

  async vote(identity: IdentityInterface, action: string, resource: object): Promise<VoteEnum> {
    if (action === "list" || action === "get") {
      return VoteEnum.Granted;     // anyone can read
    }
    if (action === "delete") {
      return this.admins.has(identity.id) ? VoteEnum.Granted : VoteEnum.Denied;
    }
    return VoteEnum.Abstain;       // don't have an opinion on other actions
  }
}
```

Two design choices here matter:

1. **Action strings are domain-defined.** The framework doesn't ship a list of valid
   actions. Use whatever your code uses (`"delete"`, `"read"`, `"update"`, `"publish"`).
   Consistency within your codebase is the goal; consistency across projects isn't.
2. **Abstain when the action isn't yours.** If `ProductVoter` returned `Denied` for
   unfamiliar actions, it'd interfere with other voters' grants. Abstain lets the
   PermissionManager pick up grants from voters that DO recognize the action.

## Granularity: one voter per resource type

The conventional split: one voter per resource type. `ProductVoter` for products,
`OrderVoter` for orders, etc. Inside each voter, branch on the `action`.

You CAN do other splits — a voter per (resource, action) pair, or a voter per role —
but a voter-per-resource keeps the code organized by what the code is *about*. If you
need to know all the rules that apply to `Product`, you open one file.

## Hierarchical permissions

For role-based scenarios (admin can do anything, editor can do most things, reader can
read), a single voter usually suffices — branch on the role from `identity`:

```ts
async vote(identity, action, resource) {
  if (identity.metadata.role === "admin") return VoteEnum.Granted;
  if (identity.metadata.role === "editor" && action !== "delete") return VoteEnum.Granted;
  if (identity.metadata.role === "reader" && action === "list") return VoteEnum.Granted;
  return VoteEnum.Denied;
}
```

If your authorization model is complex enough to warrant multiple roles voting on the
same resource, splitting into multiple voters is fine — the merge strategy handles
combining their decisions.

## Pitfalls

- **`supports()` must be cheap.** It's called for every voter on every authorization
  check. `instanceof` is fine; database lookups are not.
- **`vote()` runs async, but try to keep it fast too.** It's on the request path. Cache
  permission lookups where possible.
- **Don't throw from `vote()`.** A thrown vote is treated as a denial by some strategies
  but is genuinely ambiguous. Return `Denied` explicitly, or `Abstain` if you don't
  have enough information.
