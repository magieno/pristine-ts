# The permission manager

`PermissionManager` is the service that takes an authorization question — "can this
identity perform this action on this resource?" — polls every registered voter, and
returns a final yes/no based on a configurable voting strategy.

> **You'll learn**
> - How to call `hasAccessToResource` from your code
> - The voting strategies and when to use each
> - Where to plug authorization checks into your request flow

## The minimal call

```ts
import {inject, injectable} from "tsyringe";
import {PermissionManager} from "@pristine-ts/security";

@injectable()
export class ProductService {
  constructor(private readonly permissionManager: PermissionManager) {}

  async delete(identity: IdentityInterface, productId: string): Promise<void> {
    const product = this.products.get(productId);
    if (product === undefined) throw new NotFoundError();

    const allowed = await this.permissionManager.hasAccessToResource(
      identity,
      "delete",
      product,
    );
    if (!allowed) throw new ForbiddenError();

    this.products.delete(productId);
  }
}
```

The contract:
1. Inject `PermissionManager` like any service.
2. Call `hasAccessToResource(identity, action, resource)` with what you have.
3. Branch on the boolean. The manager returns `true` if the voters granted access under
   the active strategy.

## Voting strategies

The strategy decides what happens when **every voter abstains** — the only ambiguous
case. (When at least one voter has an opinion, the result follows: any deny → denied,
otherwise granted.) The manager takes an optional fourth argument (`VotingStrategyEnum`);
the default is `DenyOnUnanimousAbstention`.

| Strategy | If at least one grant | If at least one deny | If all abstain |
|---|---|---|---|
| `DenyOnUnanimousAbstention` (default) | Granted | Denied | **Denied** |
| `GrantOnUnanimousAbstention` | Granted | Denied | **Granted** |

Pick a strategy that matches your default-deny vs. default-allow philosophy. For most
apps `DenyOnUnanimousAbstention` is right — "if no voter says you can, you can't" is the
safe default. Use `GrantOnUnanimousAbstention` only when you have a deliberate
permissive default — e.g. an internal dashboard where the absence of a voter for a
resource type means "anyone can see it."

Override per-call when needed:

```ts
const allowed = await this.permissionManager.hasAccessToResource(
  identity, "publish", post,
  VotingStrategyEnum.GrantOnUnanimousAbstention,  // permissive when no voter recognizes the resource
);
```

## When no voters are registered

If no class is tagged `ServiceDefinitionTagEnum.Voter`, the manager warns at the log
level (once per call) and returns `false`. "No opinion at all" defaults to "no."

This is intentional — it prevents the failure mode where you ship without authorization
and nothing complains. If you genuinely want a permissive default (e.g. in dev), register
a permissive voter explicitly.

## Calling from controllers vs. services

Two common places to authorize:

**In the controller**, before delegating to the service:
```ts
@route(HttpMethod.Delete, "/{id}")
async delete(@routeParameter("id") id: string, @identity() identity: IdentityInterface) {
  const product = await this.productService.get(id);
  if (!await this.permissionManager.hasAccessToResource(identity, "delete", product)) {
    return {status: 403, body: {error: "forbidden"}};
  }
  await this.productService.delete(id);
  return {status: 204};
}
```

**In the service**, deeper:
```ts
async delete(identity: IdentityInterface, productId: string) {
  const product = await this.repository.findById(productId);
  if (!await this.permissionManager.hasAccessToResource(identity, "delete", product)) {
    throw new ForbiddenError();
  }
  // ...
}
```

Service-level checks are more robust — they fire even if some other entrypoint reaches
the service. Controller-level checks are easier to test and explicit. Many apps do both
(belt and suspenders).

## Pitfalls

- **`hasAccessToResource` doesn't cache.** Every call polls every voter. For repeated
  checks within a single request, cache the result yourself.
- **The default strategy denies on unanimous abstention.** If you register one voter
  that grants access but ALSO doesn't `supports()` your resource, the result is "all
  voters abstained" = denied. Make sure at least one voter actually recognizes the
  resource type — otherwise switch to `GrantOnUnanimousAbstention` deliberately.
- **Identity must implement `IdentityInterface`.** A plain object with the right fields
  works structurally, but for type safety construct identities through the auth pipeline.
