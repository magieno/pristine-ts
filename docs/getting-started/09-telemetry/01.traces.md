# Traces

A **trace** is a record of one logical unit of work — usually one HTTP request or one
command invocation. Inside a trace, **spans** nest to describe sub-operations and their
durations. The result is a tree you can read top-to-bottom to understand where time went.

If you've used OpenTelemetry, Datadog APM, or AWS X-Ray, this is the same conceptual
model with a Pristine-specific wire format and lifecycle.

> **You'll learn**
> - The Trace + Span data model
> - Which spans Pristine creates automatically
> - How to continue an upstream trace across services

## The data model

```
Trace                                              ← one per request, has an id (= eventId)
└── Span (root, e.g. "root.execution")             ← exactly one per trace
    ├── Span ("kernel.initialization")             ← only on the very first request
    ├── Span ("event.pre-mapping.interception")
    ├── Span ("event.mapping")
    ├── Span ("child.container.creation")
    ├── Span ("event.dispatcher.resolver")
    ├── Span ("event.post-mapping.interception")
    └── Span ("event.execution")
        └── Span ("router.request.execution")
            └── Span ("payment.charge")            ← yours, via withSpan
                └── Span ("stripe.api.charge")     ← yours
```

Every span has a `keyname` (a stable label like `"event.execution"`), an `id` (UUID),
a `startDate` and `endDate` (epoch milliseconds), an optional `context` (free-form
string-keyed bag), and a list of children. The trace's `id` is also the **eventId** the
framework uses everywhere else (logging, file tracer filenames).

## Built-in spans

Pristine spans the entire request/event lifecycle automatically. The keynames live in
`SpanKeynameEnum`:

| Span keyname | When it starts |
|---|---|
| `root.execution` | Wraps everything; the trace's root span. |
| `kernel.initialization` | Once per process, on the first event. |
| `module.initialization.<keyname>` | Per module during boot. |
| `configuration.initialization` | During the configuration-load phase. |
| `event.initialization` | Setup at the start of every event. |
| `event.pre-mapping.interception` | Pre-mapping interceptors. |
| `event.mapping` | The mapper run itself. |
| `event.post-mapping.interception` | Post-mapping interceptors. |
| `child.container.creation` | Per event, creating the per-event DI child container. |
| `child.container.registration` | Per-event provider registrations. |
| `event.dispatcher.resolver` | Resolving the dispatcher to handle this event. |
| `event.execution` | The handler invocation itself. |
| `router.setup` | HTTP-only: lazy router build on first request. |
| `router.request.execution` | HTTP-only: the per-request routing pipeline. |
| `router.find.method.router.node` | HTTP-only: the route lookup itself. |
| `router.controller.resolver` | HTTP-only: resolving the controller from DI. |
| `router.request.authentication` | HTTP-only: the `@authenticator()` step. |
| `request.interceptors` | HTTP-only: request-interceptor chain. |
| `response.interceptors` | HTTP-only: response-interceptor chain. |
| `error.response.interceptors` | HTTP-only: error-interceptor chain. |

You'll see these in the rendered trace output. The interesting ones for application
performance work are `event.execution` and its children — that's where your code lives.

## Reading a trace

Pristine doesn't show you traces by default. The framework computes them, but unless a
**tracer** is registered to do something with them, they're discarded at end-of-trace.
Two built-in tracers solve this for development:

- **`ConsoleTracer`** — clean ASCII tree to stdout. Off by default; flip
  `pristine.telemetry.console-tracer.activated=true` to see traces while developing.
- **`FileTracer`** — one file per trace, named by traceId. Off by default; flip
  `pristine.telemetry.file-tracer.activated=true` to start writing them.

A captured trace, rendered by `ConsoleTracer` in tree mode, looks like:

```
Trace 1683f41c-88d9-43d9-9559-9aa6aa2683d8 — 142ms
root.execution                              142ms
├── kernel.initialization                     5ms
├── event.pre-mapping.interception            0ms
├── event.mapping                             1ms
├── event.execution                         130ms
│   └── router.request.execution            128ms
│       ├── router.controller.resolver        2ms
│       ├── router.request.authentication     5ms
│       ├── request.interceptors              3ms
│       └── controller.execution            115ms
│           └── db.products.findAll         108ms  ← bottleneck
└── event.post-mapping.interception           1ms
```

The framework adds the standard spans; the `controller.execution → db.products.findAll`
chain is what your code added. Bottleneck detection flags the slowest **leaf** span
(an internal node would always dwarf its children, so it's never useful as a flag).

See [Tracers](03.tracers.md) for both built-in tracers and how to write your own.

## Trace propagation across services

When a Pristine app makes an outgoing call to another Pristine app, the calling service
can put its current trace ID into a header (`x-pristine-trace-id`); the downstream
service picks it up and continues the same trace. Pristine doesn't auto-instrument
outgoing HTTP — wire it explicitly:

```ts
// Outgoing — borrow the current trace's id (= eventId).
const traceId = this.tracingManager.trace?.id;
await fetch(url, {headers: {"x-pristine-trace-id": traceId ?? ""}});

// Incoming — at the boundary where you create your event, pass the upstream id.
this.tracingManager.startTracing(rootKeyname, request.headers["x-pristine-trace-id"]);
```

End-to-end tracing across microservices is built on this pattern. With both services
configured to use `FileTracer`, you can `cat <eventId>.json` in either service and see
exactly what happened on each side.

## Pitfalls

- **A trace without a registered tracer is silently dropped.** All the work computing
  spans happens for nothing. If you turn on tracing and don't see anything, check that
  at least one tracer is registered (`ConsoleTracer`, `FileTracer`, X-Ray, your own).
- **Span keynames are strings.** A typo doesn't break anything, but it loses the
  ability to filter on the misnamed span downstream. Define an enum of keynames per
  project — it's cheap and prevents drift.
- **Spans aren't free.** Each span allocates an object and adds a node to the tree.
  For tight loops (millions of iterations / second), don't wrap each iteration. Trace
  at the per-operation level, not the per-iteration level.
