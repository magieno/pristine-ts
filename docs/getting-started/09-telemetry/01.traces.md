# Traces

A trace is a record of one logical unit of work — typically one HTTP request or one
command invocation. Inside a trace, you nest **spans**: smaller units of work whose
durations and parent/child relationships tell you where time went.

If you've used OpenTelemetry, Datadog APM, or AWS X-Ray, this is the same conceptual
model. Pristine ships with a built-in tracing manager and pluggable tracers to ship trace
data to your destination of choice.

> **You'll learn**
> - The Trace / Span data model
> - How spans nest and what their lifecycle looks like
> - When spans get created automatically vs. when you add them

## The data model

```
Trace
└── Span (root, e.g. "request.execution")
    ├── Span ("kernel.initialization") — only on first request
    ├── Span ("event.pre-mapping.interception")
    ├── Span ("event.mapping")
    ├── Span ("child.container.creation")
    ├── Span ("event.dispatcher.resolver")
    ├── Span ("event.post-mapping.interception")
    └── Span ("event.execution")
        └── Span ("your-controller-method")
            └── Span ("database.query")   ← you add this
```

A trace has exactly one root span. Spans have a start time, end time, keyname (a stable
label like `"event.execution"`), and a list of child spans. The framework creates spans
automatically for every major phase of request handling; you add spans for the parts of
your code that matter to *you*.

## Built-in spans

Pristine ships an opinionated set of automatic spans for HTTP requests and command
dispatch. Names follow `SpanKeynameEnum`:

| Span keyname | When it starts |
|---|---|
| `kernel.initialization` | Once per process, the first time `kernel.handle()` runs |
| `module.initialization.<keyname>` | Per module during boot |
| `configuration.initialization` | During the ConfigurationLoad phase |
| `event.pre-mapping.interception` | Before the EventMapper runs |
| `event.mapping` | The mapper run itself |
| `child.container.creation` | Per request, creating the per-request DI child container |
| `event.dispatcher.resolver` | Resolving the EventDispatcher to handle this event |
| `event.post-mapping.interception` | After mapping, before dispatch |
| `event.execution` | The handler invocation itself |

You see these in the rendered trace output. The interesting ones for performance work
are usually `event.execution` (your code) and its children.

## Reading a trace

The TracingManager emits the full trace through registered `Tracer`s when the request
ends. By default no tracer is registered — so traces are computed and immediately
discarded. Add a tracer to actually capture the data; see [Tracers](03.tracers.md).

A captured trace, formatted, looks like this:

```
trace 1683f41c-88d9-43d9-9559-9aa6aa2683d8 (total: 142ms)
└─ root.execution                       142ms
   ├─ kernel.initialization              5ms
   ├─ event.pre-mapping.interception     0ms
   ├─ event.mapping                      1ms
   ├─ event.execution                  130ms
   │  └─ products.list (yours)         128ms
   │     └─ db.products.findAll          120ms ← 88% of the request
   └─ event.post-mapping.interception    1ms
```

Most of this is automatic. The `products.list` span and the `db.products.findAll` span
are ones you'd add to your own code via the [TracingManager](02.tracing-manager.md).

## Trace propagation across processes

When a Pristine app makes an outgoing HTTP call to another Pristine app, the calling
service can put its current trace ID into a header (`x-pristine-trace-id`); the
downstream service can pick it up and continue the same trace. This requires explicit
wiring on both sides — Pristine doesn't auto-instrument outgoing HTTP. The contract:

```ts
// outgoing
const traceId = tracingManager.trace?.id;
fetch(url, {headers: {"x-pristine-trace-id": traceId}});

// incoming, in your AppModule's request setup
tracingManager.startTracing(rootKeyname, request.headers["x-pristine-trace-id"]);
```

End-to-end tracing across microservices is built on this pattern.

## Pitfalls

- **A trace without a registered tracer is silently dropped.** All the work computing
  spans happens for nothing. If you turn on tracing and don't see traces appear in your
  APM, check that a `Tracer` is registered.
- **Span keynames are strings.** A typo doesn't break anything but loses the ability to
  filter on the misnamed span. Define an enum of keynames per project.
- **Spans aren't free.** Each span allocates an object and adds nesting. For very
  hot loops (millions/sec), don't wrap each iteration in a span. Trace at the
  per-operation level, not the per-iteration level.
