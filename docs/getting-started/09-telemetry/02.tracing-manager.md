# The tracing manager

`TracingManager` (resolved via `TracingManagerInterface`) is the service you use to add
spans to your code. It manages the active trace, lets you start/end spans, and dispatches
the finished trace through registered tracers.

> **You'll learn**
> - How to inject the tracing manager
> - How to start, end, and nest spans
> - The difference between `addSpan` and `startSpan`

## Injection

```ts
import {inject, injectable} from "tsyringe";
import {TracingManagerInterface} from "@pristine-ts/telemetry";

@injectable()
export class ProductService {
  constructor(
    @inject("TracingManagerInterface") private readonly tracingManager: TracingManagerInterface,
  ) {}
}
```

Use the string `"TracingManagerInterface"` as the inject token — same pattern as
LogHandler.

## Adding a span

```ts
async syncFromUpstream(): Promise<number> {
  const span = this.tracingManager.startSpan("catalog.sync-upstream");
  try {
    const data = await fetch(this.upstreamUrl).then(r => r.json());
    return this.applyUpdates(data);
  } finally {
    span.end();
  }
}
```

Three things:
1. `startSpan(keyname)` returns a `Span` and registers it as a child of the currently-active
   span (root-execution if none).
2. Your code runs. Time accumulates.
3. `span.end()` stamps the end time. Always pair `start` with `end` — a span that's never
   ended sits forever as an open child.

The try/finally is the safe pattern. If your code throws, `end()` still runs. If you
forget the finally, an exception leaves the span dangling and the trace looks wrong.

## Nesting spans

Spans nest automatically — `startSpan` returns a child of whatever span is currently
"active." For nested work:

```ts
const outer = this.tracingManager.startSpan("catalog.sync");
try {
  const fetched = await this.fetchUpstream();           // implicitly inside outer
  const validated = await this.validate(fetched);       // implicitly inside outer
  await this.persist(validated);                         // implicitly inside outer
} finally {
  outer.end();
}
```

Each sub-method can itself call `startSpan` to create deeper children. The whole call
tree becomes a span tree in the resulting trace.

## `addSpan` vs. `startSpan`

`startSpan(keyname)` is the high-level API: creates a span, sets its start time to now,
and tracks it as the active child.

`addSpan(span)` registers a pre-built `Span` instance directly. Useful when you already
have a span object (e.g. constructed elsewhere, or reconstructed from serialized data)
and just want to attach it to the trace. Most app code uses `startSpan`.

## Adding context to a span

The `Span` you get back from `startSpan` accepts a `context` object — a free-form bag
that gets serialized with the span:

```ts
const span = this.tracingManager.startSpan("catalog.sync");
span.context = {upstreamUrl: this.upstreamUrl, attempt: 1};
try {
  // ...
} finally {
  span.end();
}
```

Use `context` for span-level metadata your tracer will serialize. Don't dump enormous
objects in — context travels through every tracer, including ones that may have payload
size limits (X-Ray).

## When traces start

`TracingManager.startTracing(rootKeyname, traceId?, context?)` begins a new trace. Pristine
calls this automatically at the start of every `kernel.handle()` invocation — so HTTP
requests and command dispatches each get their own trace. You rarely call it yourself
unless you're writing custom entry points (a queue consumer, say).

The optional `traceId` argument is how you continue an upstream trace — see Trace
propagation in [Traces](01.traces.md).

## When traces end

`TracingManager.endTrace()` finishes the active trace and dispatches it to every
registered tracer. Pristine calls this at the end of `kernel.handle()`. Each tracer's
`trace()` method receives the complete trace and ships it wherever it ships things.

## Pitfalls

- **Spans aren't async-context-aware.** `startSpan` registers as a child of whatever
  span the manager currently thinks is active. If you call `startSpan` from inside an
  async callback after some other span has ended, you'll get unexpected parenting.
  Capture the parent span explicitly in critical sections:
  ```ts
  const parent = this.tracingManager.startSpan("parent");
  setTimeout(() => {
    const child = parent.startSpan("child");  // not tracingManager.startSpan
    child.end();
  }, 100);
  parent.end();
  ```
- **A span that never ends leaks memory.** Each trace holds references to all spans. A
  forgotten `end()` keeps the parent (and all siblings) alive until the request finishes.
  Rare in well-disciplined code but worth knowing.
- **The tracing manager survives across requests, but `trace` doesn't.** Each request
  gets a fresh trace. Don't store a span reference and reuse it across requests.
