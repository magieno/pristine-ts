# The tracing manager

`TracingManager` (resolved as `TracingManagerInterface`) is the service you inject when
you want to add spans to your code. It owns the active trace, exposes `startSpan`/`endSpan`,
and dispatches the finished trace to every registered tracer at the end of the request.

Two ways to use it: the **`SpanRunner` helper** (recommended) and the **raw `startSpan`
/ `endSpan` API** (when you need finer control).

> **You'll learn**
> - How to inject the tracing manager
> - How to wrap any function in a span via `SpanRunner.runWithSpan`
> - How and when to use the raw `startSpan` / `endSpan` API
> - How errors are surfaced in the span context

## Injection

```ts
import {inject, injectable} from "tsyringe";
import {TracingManagerInterface} from "@pristine-ts/telemetry";

@injectable()
export class PaymentService {
  constructor(
    @inject("TracingManagerInterface") private readonly tracingManager: TracingManagerInterface,
  ) {}
}
```

Use the string `"TracingManagerInterface"` as the inject token — same pattern as
`LogHandlerInterface`.

## The recommended path: `@traced` or `SpanRunner.runWithSpan`

Two ergonomic ways to wrap a piece of work in a span — pick whichever fits the shape
of your code. Both handle the start → run → end dance for you, including correctly
ending the span when your function throws.

### `@traced` — decorator form

For tracing a whole method, decorate it. The decorator resolves the `TracingManager`
from the active `EventContext` automatically — no need to inject it:

```ts
import {traced} from "@pristine-ts/telemetry";

class PaymentService {
  @traced()                             // span name = "PaymentService.charge"
  async charge(amount: number): Promise<ChargeResult> {
    return this.client.charge(amount);
  }

  @traced("payment.refund.action")      // explicit name
  async refund(chargeId: string): Promise<void> { ... }
}
```

Behavior:
- The span's name defaults to `${ClassName}.${methodName}`; pass a string to override.
- When called inside an `EventContext` (i.e. inside any Pristine event), a span is
  created, the method runs, the span ends.
- When called outside any `EventContext` (e.g. a unit test running the method
  directly), the decorator is a **no-op** — the method runs unchanged, no span is
  created. Tracing never alters semantics or throws.
- On throw, the span's `context` gets `{error: "true", errorName, errorMessage}` and
  the error re-throws.

Caveat worth knowing: the decorator awaits the wrapped method internally, so a
decorated sync method becomes async at runtime. Apply `@traced` to methods you'd
already be awaiting (DB calls, HTTP, expensive computations), not to tight sync
helpers.

### `runWithSpan` — function form

When you need to trace a block inside a method (not the whole method), or you have a
specific `TracingManager` reference, use `SpanRunner.runWithSpan` directly. It has two
call shapes:

**ALS form** (preferred when you're inside an event):

```ts
import {spanRunner} from "@pristine-ts/telemetry";

return spanRunner.runWithSpan("payment.charge", () => this.client.charge(amount));
```

The `TracingManager` is auto-resolved from the active `EventContext`'s container — no
need to inject it. Outside an `EventContext`, the function runs unchanged (no-op).

**Explicit form** (when you have a manager in hand or are running outside a context):

```ts
return spanRunner.runWithSpan(this.tracingManager, "payment.charge",
  () => this.client.charge(amount));
```

Both shapes accept an optional `options` argument with `parentKeyname`, `parentId`,
and `context`:

```ts
return spanRunner.runWithSpan("payment.charge", () => this.client.charge(amount), {
  parentKeyname: "controller.execution",   // attach as child of a specific span
  context: {                                // free-form metadata recorded on the span
    paymentMethod: "stripe",
    amountCents: String(amount),
  },
});
```

`context` values are strings (Span context is `{[k: string]: string}` by design —
small, cheap to serialize, safe for transports with payload-size limits).

## The raw API

When `runWithSpan` doesn't fit (e.g. spans whose lifetime crosses multiple methods, or
explicit hand-control of nesting), use `startSpan` / `endSpan` directly:

```ts
async syncCatalog() {
  const span = this.tracingManager.startSpan("catalog.sync");
  try {
    const fetched = await this.fetchUpstream();
    const validated = await this.validate(fetched);
    await this.persist(validated);
  } finally {
    span.end();
  }
}
```

The `try/finally` is non-negotiable — a span that's never ended sits forever as an open
child, distorts duration calculations, and may keep the trace alive past the actual
request end.

`startSpan(keyname, parentKeyname?, parentId?, context?)` returns a `Span`. Without an
explicit parent, the new span attaches to the trace's root. With `parentKeyname`, it
attaches to the most recent span with that keyname (use `parentId` to disambiguate
when multiple spans share the keyname).

## `addSpan` vs. `startSpan`

`startSpan` is what you want 99% of the time. It creates a span, time-stamps it, and
registers it.

`addSpan(span)` registers an already-constructed `Span` object. Useful if you receive
a span from elsewhere (e.g. reconstructed from a serialized form). Most application
code does not need this.

## When traces start and end

`TracingManager.startTracing(rootKeyname, traceId?, context?)` begins a new trace.
Pristine calls this automatically at the start of every event. You only call it
yourself when wiring a custom entry point (a queue consumer, a manual reentry, etc.) —
or when [continuing an upstream trace](01.traces.md#trace-propagation-across-services)
by passing the inbound `traceId`.

`TracingManager.endTrace()` finishes the active trace and dispatches it to every
registered tracer. Pristine calls this at the end of every event. You won't normally
call it yourself.

## Error annotations from `runWithSpan`

When `runWithSpan`'s callback throws, it stamps the span's context before re-throwing:

```
{
  error: "true",
  errorName: "DatabaseConnectionError",
  errorMessage: "ECONNREFUSED 127.0.0.1:5432"
}
```

These show up in the rendered trace output as part of the span's metadata — useful when
scanning a `FileTracer` JSON dump for failures. Pre-existing context fields are
preserved (the error fields are merged on top, not overwritten).

## Pitfalls

- **`TracingManager` is per-event-container.** Each request gets its own kernel child
  container with its own `TracingManager` singleton holding its own active trace. Don't
  cache a `TracingManager` reference across requests — and don't pass spans between
  requests. (A future ALS-based context propagation would let utilities auto-resolve
  the right manager; for now, inject explicitly.)
- **Spans aren't async-context-aware.** `startSpan` attaches to the span the manager
  thinks is currently active. If you start work inside a `setTimeout` after the parent
  span ended, the new span will look orphaned. For long-lived async chains, capture the
  parent's keyname or pass it explicitly via `parentKeyname`.
- **Tracing must never throw.** The framework holds this invariant defensively — a span
  that can't be added is logged and dropped, the request continues. `runWithSpan` is also
  defensive: if `startSpan` returns an unattached fallback span, ending it is still safe.
  If you find a code path where tracing throws, that's a framework bug to file.
- **A forgotten `end()` doesn't break tracing.** When `endTrace` runs, every still-open
  span is auto-ended. But the duration is wrong (it's the moment of trace-end, not the
  moment your code actually finished). Use `runWithSpan` if you can't guarantee a finally.
