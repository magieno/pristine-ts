# Tracers

A tracer takes a finished trace and ships it somewhere — your APM (Datadog, New Relic),
your log aggregator (CloudWatch, Loki), AWS X-Ray, OpenTelemetry, your own backend. The
TracingManager dispatches every completed trace to every registered tracer. No tracers
registered = traces silently dropped.

> **You'll learn**
> - What `TracerInterface` looks like
> - How to write a tracer for a destination the framework doesn't ship
> - When to use a logging-tracer vs. an APM-tracer

## The interface

```ts
import {Trace} from "@pristine-ts/telemetry";

export interface TracerInterface {
  trace(trace: Trace): Promise<void>;
}
```

One method. Receives the complete trace (root span + nested children + every context
field). Do whatever you want with it.

## Writing your own tracer

Register your tracer with `@tag(ServiceDefinitionTagEnum.Tracer)`:

```ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {Trace, TracerInterface} from "@pristine-ts/telemetry";

@tag(ServiceDefinitionTagEnum.Tracer)
@injectable()
export class DatadogTracer implements TracerInterface {
  async trace(trace: Trace): Promise<void> {
    // Walk the span tree and build whatever payload Datadog expects.
    const payload = {
      traceId: trace.id,
      spans: this.flatten(trace.rootSpan),
    };
    await fetch("https://trace.agent.datadoghq.com/api/v0.4/traces", {
      method: "POST",
      headers: {"content-type": "application/json"},
      body: JSON.stringify([payload]),
    });
  }

  private flatten(span: Span): any[] {
    return [
      {name: span.keyname, start: span.startDate, end: span.endDate, /* ... */},
      ...span.children.flatMap(c => this.flatten(c)),
    ];
  }
}
```

The `@tag` decorator self-registers the class. Pristine's TracingManager picks it up via
`@injectAll(ServiceDefinitionTagEnum.Tracer)`.

## A simple logging tracer

The smallest useful tracer just dumps traces to the LogHandler:

```ts
@tag(ServiceDefinitionTagEnum.Tracer)
@injectable()
export class LoggingTracer implements TracerInterface {
  constructor(@inject("LogHandlerInterface") private readonly logHandler: LogHandlerInterface) {}

  async trace(trace: Trace): Promise<void> {
    this.logHandler.debug(`Trace ${trace.id}`, {
      extra: {trace},
    });
  }
}
```

Good for local development — every request emits a debug log with the full trace tree.
Set `pristine.logging.logSeverityLevelConfiguration` to `5` (Debug) to see them.

## Multiple tracers

Multiple tracers can be active at once. Each runs its own `trace()` independently. A
common production setup:

- One tracer that ships to your APM (Datadog, X-Ray).
- One tracer that logs the trace ID at notice severity (so log entries can be correlated
  back to traces).
- Optionally, a debug-only LoggingTracer.

## Tracers in the catalog demo

The catalog demo doesn't register any tracer — the trace data is computed and discarded.
That's by design for a tutorial demo. In production, register at least one.

## Pitfalls

- **`trace()` is awaited per-tracer.** A slow tracer blocks `endTrace()`, which blocks
  `kernel.handle()`'s return. For network-bound tracers, fire-and-forget or batch
  internally — don't await the network round-trip.
- **Don't `throw` from `trace()`.** Trace dispatch happens at the end of every request;
  a thrown error there can mask other issues. Wrap your body in try/catch and log
  failures via `LogHandlerInterface`.
- **Tracers see every trace, including their own creation.** If your tracer creates
  spans internally, they end up in subsequent traces. Generally fine, but worth knowing
  if your APM bills per-span.
