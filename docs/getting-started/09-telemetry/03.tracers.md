# Tracers

A **tracer** is a transport — it takes a finished trace and ships it somewhere (stdout,
a file, AWS X-Ray, your APM, your own backend). The TracingManager dispatches every
completed trace to every registered tracer at the end of the request.

Pristine ships two built-in tracers (`ConsoleTracer`, `FileTracer`) and one optional
package (`@pristine-ts/aws-xray`) for AWS X-Ray. Writing your own is small.

> **Reading a trace = reading what happened.** Pristine logs carry their own content
> (message, severity, eventId, traceId) — they do not carry an inline trail of "what
> happened around this log." That trail lives in the trace. To investigate an event,
> use the eventId from a log line to look up the trace via your tracer of choice:
>
> - **In dev**: enable `ConsoleTracer` and the full span tree (plus any
>   `addEventToCurrentSpan` markers) prints right alongside your logs.
> - **In prod**: enable `FileTracer` (one JSON file per trace, keyed by eventId) or
>   ship traces to your APM via a custom tracer.
>
> One log line + one trace lookup = the same information that used to be inline.

> **You'll learn**
> - The actual `TracerInterface` shape (it's stream-based, like loggers)
> - The two built-in tracers and when to enable each
> - How to disable AWS X-Ray without removing the package
> - How to write a custom tracer — and the crash-isolation guarantees

## The interface

```ts
import {Readable} from "stream";

export interface TracerInterface {
  spanStartedStream?: Readable;
  spanEndedStream?: Readable;
  traceStartedStream?: Readable;
  traceEndedStream?: Readable;
}
```

Four optional `Readable` streams. The TracingManager pushes events into the streams it
finds on each tracer; tracers consume them via `'data'` listeners. There's no `trace()`
method on the interface — dispatch is **via the stream**, not via a function call. Same
deliberate design as `LoggerInterface`: streams give the manager backpressure for free
and let it push synchronously without awaiting each transport.

Most tracers only implement `traceEndedStream` — the moment when the full trace is
available with all spans complete. Implement the others if you need real-time visibility
into span open/close events.

## Built-in: `ConsoleTracer`

Prints completed traces to stdout. Off by default — enable when developing locally to
see what your requests are doing without wiring up a backend.

```ts
// pristine.config.ts
export default defineConfig({
  kernelConfiguration: {
    "pristine.telemetry.console-tracer.activated": true,
  },
});
```

Or via environment: `PRISTINE_TELEMETRY_CONSOLE_TRACER_ACTIVATED=true`.

Three output modes (`pristine.telemetry.console-tracer.output-mode`):

- **`tree`** (default) — ASCII tree, durations right-aligned, slowest leaf flagged.
- **`json`** — pretty-printed JSON dump.
- **`flat`** — one line per span sorted by start time, no indentation.

Plus a noise filter (`pristine.telemetry.console-tracer.minimum-duration-ms`): skip
rendering traces shorter than N ms. Useful when most requests are fast and you only
want to see the slow ones.

Sample tree output:
```
Trace abc-123 — 142ms
root.execution                    142ms
├── event.execution               130ms
│   └── controller.execution      115ms
│       └── db.products.findAll   108ms  ← bottleneck
└── event.post-mapping.interception  1ms
```

## Built-in: `FileTracer`

Writes one file per completed trace to a configured directory. Files are named after
the trace's id (= eventId), so locating a request's trace is `cat traces/<eventId>.json`.

```ts
// pristine.config.ts
export default defineConfig({
  kernelConfiguration: {
    "pristine.telemetry.file-tracer.activated": true,
    "pristine.telemetry.file-tracer.directory": "./traces",            // default: ./traces
    "pristine.telemetry.file-tracer.output-mode": "json",              // default: json
    "pristine.telemetry.file-tracer.filename-pattern": "<traceId>.json", // default
  },
});
```

The filename pattern supports three placeholders:

| Placeholder | Resolves to |
|---|---|
| `<traceId>` | The trace's UUID, also the eventId |
| `<date>` | UTC `YYYY-MM-DD` at the moment the trace ended |
| `<timestamp>` | Milliseconds since epoch at the moment the trace ended |

Subdirectories in the pattern are created on demand — `<date>/<traceId>.json`
partitions traces into per-day folders.

Pitfalls:
- High-traffic apps will fill disk fast. Default to off, opt in deliberately, and add
  a cron job (or use `<date>/` partitioning) so you can rotate.
- Writes are **synchronous** by design — the tracer pipeline is synchronous, and an
  async write would silently leak rejections. The cost is bounded (one trace = typically
  <50KB), but every request now has a small fs write on its critical path.

## Optional: AWS X-Ray (`@pristine-ts/aws-xray`)

Add `@pristine-ts/aws-xray` to your dependencies and import it in your AppModule. The
package registers an `XrayTracer`, but **export is off by default** — you must opt in
explicitly to start shipping segments to X-Ray:

```ts
// pristine.config.ts
export default defineConfig({
  kernelConfiguration: {
    "pristine.aws-xray.activated": true,   // or: PRISTINE_AWS_XRAY_ACTIVATED=true
  },
});
```

The opt-in default exists so importing the package — directly or transitively through
some other dependency — never accidentally tries to ship segments to X-Ray in
environments where AWS isn't reachable (local dev, tests, CI without AWS access).

When deactivated (the default, or whenever you set `activated: false`), the tracer is
still constructed (it has to be — the framework wires its stream listener at construction
time), but the listener early-returns instead of touching the AWS X-Ray SDK. No segments
are sent, no AWS credentials are required.

A typical pattern: turn it on per-environment:

```ts
kernelConfiguration: {
  "pristine.aws-xray.activated": process.env.NODE_ENV === "production",
}
```

## Writing your own tracer

The smallest useful tracer just observes trace-ends:

```ts
import {Readable} from "stream";
import {tag, ServiceDefinitionTagEnum, moduleScoped} from "@pristine-ts/common";
import {injectable, singleton} from "tsyringe";
import {Trace, TracerInterface, TelemetryModuleKeyname} from "@pristine-ts/telemetry";

@moduleScoped(TelemetryModuleKeyname)
@singleton()
@tag(ServiceDefinitionTagEnum.Tracer)
@injectable()
export class DatadogTracer implements TracerInterface {
  public traceEndedStream: Readable;

  constructor() {
    this.traceEndedStream = new Readable({objectMode: true, read() { return true; }});
    this.traceEndedStream.on("data", (trace: Trace) => {
      try {
        this.shipTrace(trace);
      } catch (error) {
        // Critical: never throw out of the listener. A throw here would propagate
        // through the manager's .push() and could break other tracers' delivery.
        process.stderr.write(`[my-app][datadog-tracer] ${(error as Error).message}\n`);
      }
    });
    this.traceEndedStream.on("error", (error) => {
      process.stderr.write(`[my-app][datadog-tracer] stream error: ${error.message}\n`);
    });
  }

  private shipTrace(trace: Trace) {
    // Your transport call. Don't await network round-trips here on the hot path —
    // batch internally or fire-and-forget if your downstream allows it.
  }
}
```

The `@tag(ServiceDefinitionTagEnum.Tracer)` self-registers the class. As long as your
AppModule's import graph reaches the file, the TracingManager picks it up and starts
pushing finished traces into your stream.

## Crash isolation

A failing tracer never takes down the app:

1. **Inside the tracer**: wrap your `'data'` listener in try/catch and attach an
   `'error'` listener. Both built-in tracers (`ConsoleTracer`, `FileTracer`) do this and
   write any failure to `process.stderr` with a `[pristine][tracer:<ClassName>]` prefix.
2. **Inside the TracingManager**: the manager's `.push()` calls aren't currently wrapped
   — so a tracer that throws synchronously from its listener *can* propagate back into
   `endTrace()`. Defending at the listener (option 1) is the practical safety net.

If you write a custom tracer, follow the pattern shown above. The framework will keep
working even if your transport is broken; your transport's failures show up on stderr
instead of crashing requests.

## Multiple tracers

Multiple tracers can be active at once. Each consumes its own stream independently. A
common production setup:

- One tracer that ships to your APM (X-Ray, Datadog, ...).
- `ConsoleTracer` off in production, on in development.
- `FileTracer` off everywhere except for short investigation windows where you want
  per-request artifacts.
- A logging tracer that emits the trace ID at notice severity (so log entries can be
  correlated back to traces).

## Pitfalls

- **A trace without a registered tracer is silently dropped.** Computing trace data
  is meaningful work; doing it without consumers is pure waste. If you don't see
  traces, check at least one tracer is registered and active.
- **Don't await network calls inside the listener.** The listener fires synchronously
  in the manager's `.push()` call. A slow transport blocks `endTrace()`, which blocks
  the request from returning. Buffer / batch / fire-and-forget instead.
- **Tracers run for every event.** That includes events created by other tracers if
  they call into instrumented code. This rarely matters, but watch out if your tracer
  itself logs heavily — you can create a feedback loop.
