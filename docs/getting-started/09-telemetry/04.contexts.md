# Contexts in Pristine

Pristine propagates a handful of cross-cutting values implicitly so your code doesn't
have to thread them through every method signature: the **event id** (correlation key
shared between logs, breadcrumbs, traces, and the file tracer), the **trace id**, the
per-event **DI container**, and — for HTTP-routed code — the active **request** and
**identity**.

The mechanism is `AsyncLocalStorage` (built into Node.js). Pristine installs a context
at the right boundary of each event lifecycle; your code reads from it through small
typed managers.

> **You'll learn**
> - The two context managers (`EventContextManager`, `RequestContextManager`) and what each owns
> - When the framework installs each context
> - How to read from them, and what happens when no context is active
> - The escape hatch (`bind()`) for background work that breaks the natural async chain

## What is `AsyncLocalStorage`?

A Node primitive (`node:async_hooks`) that propagates a value across `await`, `Promise.all`,
`setTimeout`, `setImmediate`, etc. — wherever Node tracks an async resource chain. You
install a value at the start of a logical operation (`als.run(value, fn)`), and any code
reachable from `fn` (no matter how deep the async chain) can read the same value back
(`als.getStore()`).

If you've used NestJS's `cls-hooked`, Fastify's `request-context`, or OpenTelemetry's
context API, this is the same idea.

## The two contexts

| Context | Owned by | Holds | Installed at | Lives until |
|---|---|---|---|---|
| `EventContext` | `@pristine-ts/common` | `eventId`, `traceId?`, `container?` | start of every event in the pipeline | the event finishes |
| `RequestContext` | `@pristine-ts/networking` | `request`, `methodNode?`, `identity?` | start of router request handling | the response is returned |

Two contexts, two ALS instances, two managers. They **nest** naturally: when an HTTP
request is being handled, the router installs a `RequestContext` *inside* the framework's
`EventContext`. Both are queryable from controller code:

```ts
import {EventContextManager} from "@pristine-ts/common";
import {RequestContextManager} from "@pristine-ts/networking";

class ProductController {
  @route(HttpMethod.Get, "/products/:id")
  async show() {
    const eventId = EventContextManager.eventId();          // from EventContext
    const request = RequestContextManager.request();         // from RequestContext
    const identity = RequestContextManager.identity();       // from RequestContext
    // ...
  }
}
```

Apps that don't import `@pristine-ts/networking` (e.g. an event-only worker) never see
`RequestContext`. The `EventContext` is always available because it's installed by the
framework's core event pipeline.

> **Confusing name alert.** `ExecutionContextInterface` (in `@pristine-ts/core`) is
> something different — a *static* descriptor of *where the framework is running* (AWS
> Lambda, CLI, Express). It's set once per `kernel.handle()` call and never changes.
> `EventContext` is per-event runtime state that varies request to request.

## What lives in `EventContext`

```ts
import {EventContext, EventContextManager} from "@pristine-ts/common";

class EventContext {
  eventId!: string;                  // correlation id, unique per event
  traceId?: string;                  // filled in when TracingManager.startTracing runs
  container?: DependencyContainer;   // the per-event DI child container
}
```

`EventContextManager` exposes both instance methods (resolve via DI) and static
convenience methods (no constructor needed):

```ts
const id      = EventContextManager.eventId();
const trace   = EventContextManager.traceId();
const child   = EventContextManager.container();
const all     = EventContextManager.current();   // the whole context, or undefined
```

When no `EventContext` is active (e.g. boot-time code, or a unit test calling a service
directly without booting a kernel), every accessor returns `undefined`. Code that reads
from the context must tolerate that — never assume an id is present.

## What lives in `RequestContext`

```ts
import {RequestContext, RequestContextManager} from "@pristine-ts/networking";

class RequestContext {
  request!: Request;                 // the inbound request — set the moment a route matches
  methodNode?: MethodRouterNode;     // the matched route — set with `request`
  identity?: IdentityInterface;      // populated after the authentication pipeline runs
}
```

`identity` is `undefined` until auth has completed. For code that runs *before* auth
(e.g. a request interceptor with high priority), reading `identity()` returns `undefined`
— check before using it.

## What the framework wires up automatically

You don't have to do anything to get these — they're consequences of the contexts being
installed:

- **Logs are auto-tagged with eventId.** Calling `logHandler.info("...")` from inside
  an event picks up the eventId without you passing `data.eventId`. (See
  [The log handler](../07-logging/01.log-handler.md).)
- **Breadcrumbs land under the right event.** `breadcrumbHandler.add(undefined, "...")`
  uses the active EventContext's id.
- **Tracers see the right trace.** The `TracingManager` and any tracers you've installed
  use the EventContext's container to resolve per-event state.
- **`@traced` works.** The decorator resolves `TracingManager` from
  `EventContextManager.container()`. (See [The tracing manager](02.tracing-manager.md).)
- **`runWithSpan(name, fn)` works.** Same lookup. The explicit-manager form
  (`runWithSpan(tm, name, fn)`) still works for callers who want to opt out of the
  auto-resolve.

## The escape hatch: `bind()`

ALS propagates across **most** async patterns but not all. The known cases where the
chain breaks:

- Raw `EventEmitter`-style callbacks registered *outside* the event lifecycle.
- Detached `setImmediate` whose callback runs after the parent function returned and
  the framework already tore down its `als.run(...)`.
- Some native modules that use libuv work-queues without honoring the async-hooks model.

If you spawn work that escapes the natural chain, use `bind()` to capture the current
context and restore it on the callback:

```ts
import {EventContextManager} from "@pristine-ts/common";

class Worker {
  async process(item: Item) {
    // Schedule a follow-up that runs after we return. Without bind, the callback
    // would see no EventContext and lose the eventId / traceId / container.
    const followUp = EventContextManager.bind(() => this.audit(item));
    setImmediate(followUp);
  }

  private async audit(item: Item) {
    // EventContextManager.eventId() returns the right value here, even though
    // setImmediate ran on a different async tick.
  }
}
```

`bind()` works the same on `RequestContextManager` for HTTP-specific state.

## Pitfalls

- **Outside any event context, everything is `undefined`.** Code that runs from a
  custom entry point (a queue consumer that builds its own kernel-handle call, a
  background daemon) won't have an `EventContext`. Either install one yourself by
  building an `EventContext` and calling `eventContextManager.run(ctx, fn)`, or accept
  that helpers that depend on context become no-ops.
- **Long-lived in-memory work outlives the event.** If a request handler kicks off a
  background promise and returns the response without awaiting, the promise's `await`s
  will continue to see the request's `EventContext` for a while — but the per-event
  child container has been torn down. Resolving services from `EventContextManager.container()`
  in detached work is unsafe. Either resolve at the start and capture references, or
  use `bind()` and accept the lifecycle.
- **Don't store contexts in module-level state.** They're per-event by design. If you
  cache a `RequestContext` reference in a singleton, you'll leak request state across
  unrelated requests.
- **Testing without a kernel.** Unit tests that call service methods directly (without
  going through `kernel.handle`) run outside any `EventContext`. `@traced` is a no-op
  in that situation, `EventContextManager.eventId()` returns `undefined`, and log calls
  have no eventId attached. This is correct behavior — but if a test relies on
  context-derived values, install a fake context yourself:
  ```ts
  const ecm = new EventContextManager();
  const ctx = new EventContext();
  ctx.eventId = "test-evt";
  await ecm.run(ctx, () => myService.work());
  ```
