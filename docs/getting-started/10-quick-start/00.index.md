# Quick start

Five minutes from a fresh terminal to a running Pristine app. This chapter walks through
`pristine init`, looks at what it scaffolds, and boots the result. The next chapter
([The Pristine CLI](../11-cli/00.index.md)) goes deeper on what the CLI is doing; this
one keeps moving.

> **You'll learn**
> - How to install the Pristine CLI
> - How to scaffold a project with `pristine init`
> - What `pristine.config.ts` is and why it exists
> - How to build and start a Pristine app
> - How graceful shutdown works on `Ctrl+C`

## Prerequisites

- Node.js **18 or later**. Run `node --version` to check.
- npm 8+. Ships with Node, so usually you have it.

That's it. No global tools, no scaffold generator separate from the CLI.

## Step 1: Make a project directory

```sh
mkdir my-pristine-app
cd my-pristine-app
npm init -y
```

You now have an empty project with a `package.json`. The exact contents don't matter
yet — `pristine init` is about to edit it.

## Step 2: Install the Pristine CLI

```sh
npm install --save-dev @pristine-ts/cli @pristine-ts/common @pristine-ts/core
```

You need at least `@pristine-ts/cli` (for the `pristine` binary), `@pristine-ts/common`
(the foundation types every Pristine package uses), and `@pristine-ts/core` (the Kernel
and module-loading machinery). Other Pristine packages — `@pristine-ts/http`,
`@pristine-ts/logging`, `@pristine-ts/networking`, etc. — get installed as you start using
those features in later chapters.

## Step 3: Run `pristine init`

```sh
npx pristine init
```

You'll get prompted for four things (just press Enter to accept the defaults — they're
sensible for greenfield projects):

```
? Where does your AppModule source file live? (src/app.module.ts)
? Where should the compiled AppModule output land? (dist/app.module.js)
? Which tsconfig should `pristine build` use? (tsconfig.json)
? Which build format do you want? (Use arrow keys)
❯ esm  (modern, recommended)
  cjs  (CommonJS)
  both (publish ESM + CJS)
? Scaffold a starter AppModule at src/app.module.ts? (Y/n)
? Add `build`, `start`, `verify` scripts to package.json? (Y/n)
```

When you accept the defaults, `pristine init` creates:

```
my-pristine-app/
├── package.json            (← scripts added: build, start, verify)
├── pristine.config.ts      (← new)
└── src/
    └── app.module.ts       (← scaffolded)
```

And appends `.pristine/` to `.gitignore` (if one exists).

> **Pitfall**: in non-interactive contexts (CI, Docker), `pristine init` requires every
> value as a flag — `pristine init --source-path=src/app.module.ts --output-path=dist/app.module.js
> --tsconfig=tsconfig.json --format=esm`. It won't silently pick defaults; it'll fail
> loudly so you don't end up with a scaffold that doesn't match your project layout.

## Step 4: Look at what got created

Open `pristine.config.ts`:

```ts
import {defineConfig} from "@pristine-ts/cli";

export default defineConfig({
  appModule: {
    sourcePath: "src/app.module.ts",
    outputPath: "dist/app.module.js",
  },
  build: {
    tsconfig: "tsconfig.json",
    format: "esm",
  },
});
```

This is the Pristine config file. Every CLI command reads it. `appModule.sourcePath` is
what `pristine build` compiles; `appModule.outputPath` is what `pristine start` /
`pristine verify` load at runtime. The CLI uses these to write a build manifest that
tracks freshness (covered in chapter 11).

Open `src/app.module.ts`:

```ts
import {AppModuleInterface} from "@pristine-ts/common";
import {CoreModule} from "@pristine-ts/core";

export const AppModule: AppModuleInterface = {
  keyname: "my-app",
  importModules: [
    CoreModule,
  ],
  importServices: [],
};
```

This is your application's root module. Right now it imports just `CoreModule` —
everything the framework needs to boot. `importServices` is where you list classes the
kernel should register (we'll add some in chapter 03). `keyname` is a stable identifier
used by `@moduleScoped()` decorators.

Open `package.json` and look at the `scripts` block:

```json
{
  "scripts": {
    "build": "pristine build",
    "start": "pristine start",
    "verify": "pristine verify"
  }
}
```

These three are enough to use the CLI without typing `npx pristine` every time. Inside
`npm run`, `node_modules/.bin/pristine` is already on PATH.

You'll also need a `tsconfig.json` if you don't have one. The minimum:

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "commonjs",
    "rootDir": "./src",
    "outDir": "./dist",
    "esModuleInterop": true,
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true,
    "strict": false,
    "skipLibCheck": true
  },
  "include": ["src"]
}
```

`experimentalDecorators` + `emitDecoratorMetadata` are non-negotiable — Pristine's DI
container depends on the decorator metadata TypeScript emits at compile time.

## Step 5: Build

```sh
npm run build
```

Output:

```
ℹ Info: Compiling with tsconfig.json
npx tsc -p tsconfig.json
Exit (code: 0).
ℹ Info: Manifest written: .pristine/build-manifest.json (built at 2026-05-12T...)
✔ Success: Build complete (1 tsconfig).
```

Two new things on disk:
- `dist/app.module.js` — what `pristine start` will load.
- `.pristine/build-manifest.json` — a small JSON file recording which source produced the
  output and a content hash. The CLI checks this before every `start`/`verify` so it can
  warn you when your source has changed and the build is stale.

## Step 6: Start the app

```sh
npm run start
```

Output:

```
✔ Success: Pristine app running. Send SIGTERM (or Ctrl+C) to stop.
```

The process stays running. It's not serving HTTP yet — `CoreModule` alone doesn't
register an HTTP server. We'll add `HttpModule` in chapter 03. For now, "running" means
"the kernel booted, every module's `onInit` succeeded, and the process is waiting on a
signal."

## Step 7: Stop the app

In another terminal, send SIGTERM. Or just `Ctrl+C` in the same terminal:

```
ℹ Info: Received SIGINT, shutting down gracefully...
✔ Success: Shutdown complete.
[status:'Success', code:'0'] - Command 'start' exited.
```

What happened:
1. Pristine caught the signal.
2. `Kernel.stop()` walked every module's `onShutdown` hook (outer-to-inner) — none of
   our modules have one yet, so this finishes instantly.
3. Process exited with code 0.

A second `Ctrl+C` during shutdown would force-exit immediately (`exit 130`). After 30
seconds, a hard-exit timeout kicks in regardless — so a wedged shutdown can never block
your orchestrator.

## You're set

Recap of what you just built:
- An empty Pristine app boots, idles, and shuts down gracefully.
- A `pristine.config.ts` drives every CLI command.
- A build manifest tracks staleness between source and output.

This isn't useful yet — there's nothing to interact with. The next chapter walks through
every CLI command in detail, including how to write your own. After that, chapter 01
(Overview) and chapter 05 (Modules) explain the framework concepts; chapter 03
(Controllers) adds an HTTP server and your first route.
