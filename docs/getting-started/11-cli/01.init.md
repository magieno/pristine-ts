# `pristine init` and `pristine.config.ts`

`pristine init` is the canonical setup command. It interactively (or via flags) writes
the `pristine.config.ts` file every other CLI command reads, scaffolds a starter
AppModule, and adds npm scripts to your `package.json`.

## What it produces

Running `pristine init` with the defaults creates four things:

```
my-app/
├── package.json            (← `build`, `start`, `verify` scripts added)
├── pristine.config.ts      (← new)
├── .gitignore              (← `.pristine/` appended if .gitignore exists)
└── src/
    └── app.module.ts       (← scaffolded)
```

The full config:

```ts
// pristine.config.ts
import {defineConfig} from "@pristine-ts/cli";

export default defineConfig({
  appModule: {
    sourcePath: "src/app.module.ts",
    outputPath: "dist/app.module.js",
  },
  build: {
    tsconfig: "tsconfig.json",
    format: "esm",
  },
});
```

Three things matter here:

- **`appModule.sourcePath`** — the TypeScript file `pristine build` compiles.
- **`appModule.outputPath`** — the JavaScript file `pristine start` / `pristine verify` load
  at runtime. Must match what your `tsconfig.json`'s `outDir` actually produces.
- **`build.format`** — `"esm"` (most common), `"cjs"`, or `"both"`. Affects which
  tsconfig(s) `pristine build` runs.

## Non-interactive use (CI, Docker)

`pristine init` in a non-TTY context (no terminal attached) refuses to silently pick
defaults — it fails fast with a list of missing flags. Pass everything explicitly:

```sh
pristine init \
  --source-path=src/app.module.ts \
  --output-path=dist/app.module.js \
  --tsconfig=tsconfig.json \
  --format=esm \
  --scaffold \
  --scripts
```

`--scaffold` enables the starter AppModule write; pass `--no-scaffold` to skip. Same for
`--scripts` (the npm script additions).

## Migrating from older Pristine projects

If your project already has the legacy `pristine.appModule.cjsPath` field in
`package.json`:

```json
{
  "pristine": {
    "appModule": { "cjsPath": "dist/lib/cjs/app.module.js" }
  }
}
```

`pristine init` detects this, generates `pristine.config.ts` with the path migrated, and
tells you to remove the `package.json` entry. The legacy field still works for one minor
cycle (with a deprecation warning at startup), so you can migrate at your own pace.

## The full config schema

```ts
interface PristineConfig {
  appModule?: {
    sourcePath?: string;     // for pristine build
    outputPath?: string;     // for pristine start, verify, etc.
    export?: string;          // default: "AppModule"
  };
  build?: {
    outDir?: string;
    tsconfig?: string;        // default: "tsconfig.json"
    format?: "esm" | "cjs" | "both";  // default: "esm"
    clean?: boolean;
  };
  plugins?: Array<string | { name: string; options?: unknown }>;
  kernelConfiguration?: Record<string, unknown>;
}
```

Every field is optional. `kernelConfiguration` lets you override `kernel.start()`
configuration values from the config file — useful for CLI-only overrides that shouldn't
pollute your AppModule's `configurationDefinitions`. Example:

```ts
kernelConfiguration: {
  "pristine.logging.logSeverityLevelConfiguration": 1,
  "pristine.http.kernel-server.port": 4000,
}
```

## Config file discovery

The CLI walks **up** from `process.cwd()` looking for one of these names, in order:

1. `pristine.config.ts` (recommended — full IDE autocomplete via `defineConfig`)
2. `pristine.config.mts`
3. `pristine.config.cts`
4. `pristine.config.js`
5. `pristine.config.mjs`
6. `pristine.config.cjs`

`.ts` files load via `jiti` at runtime — no separate compile step. Walking up matters in
monorepos: invoking `pristine` from `packages/foo/` still finds a `pristine.config.ts` at
the repo root.

## Inspecting the resolved config

```sh
pristine p:config:print
```

Prints the loaded config plus the file path it came from. Use this when the CLI is doing
something unexpected and you suspect the wrong config file is in play.

## Pitfalls

- **`outputPath` must match what `tsconfig.json` actually emits.** If your tsc `outDir` is
  `./build/` and `outputPath` in config says `dist/app.module.js`, the CLI won't find the
  compiled output. `pristine build` flags this with a warning ("output not produced").
- **`appModule.export` defaults to `AppModule`.** If you export under a different name
  (e.g. `export const MyApp = ...`), set `appModule.export = "MyApp"` in config.
- **In monorepos, the upward walk can pick a parent's config.** That's usually what you
  want; if not, set `--config <path>` (planned) or run `pristine` with a tighter cwd.
