# `pristine build` and the build manifest

`pristine build` is a `tsc` wrapper with one extra step: after a successful compile, it
writes a **build manifest** at `<project>/.pristine/build-manifest.json`. Every other
runtime command (`pristine start`, `pristine verify`, etc.) reads that manifest first to
detect when the build is stale.

## The minimal example

With the config that `pristine init` produces:

```sh
npm run build
```

Output:

```
â„ą Info: Compiling with tsconfig.json
npx tsc -p tsconfig.json
Exit (code: 0).
â„ą Info: Manifest written: .pristine/build-manifest.json (built at 2026-05-12T...)
âś” Success: Build complete (1 tsconfig).
```

Two artifacts:
- **`dist/app.module.js`** (plus the rest of your compiled tree) â€” what tsc emitted.
- **`.pristine/build-manifest.json`** â€” Pristine's record of what just happened.

The manifest looks like:

```json
{
  "appModuleSourcePath": "/abs/path/src/app.module.ts",
  "appModuleOutputPath": "/abs/path/dist/app.module.js",
  "sourceHash": "sha256:...",
  "builtAt": "2026-05-12T14:46:59.089Z"
}
```

Four fields:
- **`appModuleSourcePath`** and **`appModuleOutputPath`** â€” absolute paths, snapshotted at
  build time.
- **`sourceHash`** â€” SHA-256 of the AppModule source file's bytes. This is what
  staleness detection compares against.
- **`builtAt`** â€” ISO timestamp; informational, used in log lines.

## Staleness detection

Before `pristine start` (and any other runtime command) loads your AppModule, it reads the
manifest and checks four things:

| Condition | Meaning |
|---|---|
| Manifest missing | You haven't run `pristine build` yet. |
| Source path in config no longer matches the manifest | You changed `appModule.sourcePath` since the last build. |
| Output path in config no longer matches the manifest | You changed `appModule.outputPath` since the last build. |
| Output file not on disk | Someone deleted `dist/`. |
| Source file's current hash â‰  manifest hash | You edited `src/app.module.ts` (or its imports) since the last build. |

Any of these triggers staleness. In a TTY, the CLI prompts:

```
âś– Your AppModule source has changed since the last build. Run `pristine build` to recompile.
  Run `pristine build` now to refresh? (Y/n)
```

In CI / non-TTY, the CLI prints the same message and exits non-zero. CI never auto-rebuilds
â€” that hides bugs. Run `pristine build` explicitly in your CI pipeline before
`pristine start` or `pristine verify`.

## Build formats

`build.format` in `pristine.config.ts` controls which tsconfig(s) `pristine build` runs:

- **`"esm"`** (default) â€” runs `tsc -p tsconfig.json`. Modern ESM output. Most projects.
- **`"cjs"`** â€” runs `tsc -p tsconfig.cjs.json` (or falls back to the primary tsconfig).
  For projects that need CommonJS output specifically.
- **`"both"`** â€” runs the primary tsconfig and then the `.cjs.json` sibling. For library
  packages that publish dual ESM + CJS.

`"both"` requires you to maintain two tsconfig files. The Pristine packages themselves use
this pattern â€” see `packages/cli/tsconfig.json` and `packages/cli/tsconfig.cjs.json` for a
reference.

## Cleaning the output directory

```ts
build: {
  outDir: "dist",
  clean: true,
}
```

When `clean: true`, `pristine build` deletes `outDir/` before invoking tsc. Catches the
"old file left behind after rename" footgun.

## Custom tsconfig path

```ts
build: {
  tsconfig: "tsconfig.build.json",
}
```

Useful when your editor's tsconfig (with `noEmit` and strict checks) differs from the build
tsconfig. The CLI passes whatever path you give it to `tsc -p ...`.

## The catalog example

[`demo/projects/catalog/pristine.config.ts`](../../../demo/projects/catalog/pristine.config.ts)
configures `format: "esm"` and points at a standard `tsconfig.json`. Running
`npm run build` in that directory writes the manifest to
`demo/projects/catalog/.pristine/build-manifest.json` â€” you can `cat` it to see all four
fields populated.

## Pitfalls

- **The manifest only writes when both `sourcePath` and `outputPath` are configured.**
  Without both, `pristine build` still compiles, but no manifest is written and downstream
  commands skip the staleness check. You'll get a warning at build time: "Build succeeded
  but appModule.outputPath '...' was not produced; skipping manifest."
- **Manifests are per-project, not per-developer.** `.pristine/` is gitignored â€” every
  contributor builds locally and gets their own manifest.
- **Rebuilding with a different Node version is not detected.** The hash is of the
  source, not the output. If you switch from Node 18 to Node 22 and that changes your
  output bytes, the manifest won't flag it as stale. In practice this rarely matters â€”
  tsc output is deterministic across Node versions â€” but be aware.

## Where to next

`pristine build` produces the output. [`pristine start`](03.start.md) consumes it,
boots your kernel, and keeps your app running.
