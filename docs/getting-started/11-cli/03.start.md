# `pristine start` and graceful shutdown

`pristine start` is the production-grade entry point. It loads your compiled AppModule,
boots the kernel, starts every registered runtime server (HTTP, gRPC, anything tagged
`RuntimeServer`), and keeps the process alive until SIGTERM or SIGINT — at which point it
runs every module's `onShutdown` hook before exiting.

You can use `pristine start` in dev, in CI smoke tests, and in production. It's the same
binary either way.

## The minimal example

From the [catalog demo](../../../demo/projects/catalog/readme.md):

```sh
npm run start
# ✔ Success: Pristine app running with 1 server(s): http. Send SIGTERM (or Ctrl+C) to stop.

# In another terminal:
curl http://localhost:3000/products
# {"products":[{"id":"peach",...},{"id":"banjo",...}]}
```

When the AppModule imports `@pristine-ts/http`, `pristine start` automatically detects the
`KernelHttpServer` (a tagged `RuntimeServer`) and launches it. No glue code — importing
the module is enough.

## Configuring port and address

Three layers, highest priority wins:

```sh
# 1. CLI flag — one-off override
pristine start --port=4000 --address=127.0.0.1
```

```ts
// 2. Config file — project default
export default defineConfig({
  // ...
  kernelConfiguration: {
    "pristine.http.kernel-server.port": 4000,
    "pristine.http.kernel-server.address": "127.0.0.1",
  },
});
```

```sh
# 3. Environment variable — deploy-time override
PRISTINE_HTTP_KERNEL_SERVER_PORT=4000 \
PRISTINE_HTTP_KERNEL_SERVER_ADDRESS=0.0.0.0 \
  pristine start
```

Defaults: `0.0.0.0:3000`.

## HTTPS

Set TLS key and cert paths and `KernelHttpServer` switches from `http.Server` to
`https.Server`:

```sh
PRISTINE_HTTP_KERNEL_SERVER_TLS_KEY_PATH=/etc/ssl/key.pem \
PRISTINE_HTTP_KERNEL_SERVER_TLS_CERT_PATH=/etc/ssl/cert.pem \
  pristine start
# KernelHttpServer: listening on https://0.0.0.0:3000
```

Both paths must be set to non-empty values. The server reads the PEM files at boot.

## Graceful shutdown

SIGTERM or SIGINT triggers:

```
ℹ Info: Received SIGTERM, shutting down gracefully...
ℹ Info: KernelHttpServer: server.close() — refusing new connections
ℹ Info: Waiting for 2 in-flight requests to finish (timeout 10s)
ℹ Info: KernelHttpServer: stopped.
✔ Success: Shutdown complete.
[status:'Success', code:'0'] - Command 'start' exited.
```

What happens in order:
1. Pristine catches the signal.
2. `Kernel.stop()` walks every module's `onShutdown` hook in **outer-to-inner** order
   (your AppModule first, leaf dependencies last) so a higher-level module can still
   call into its dependencies during teardown.
3. For `HttpModule` specifically, `onShutdown` calls `KernelHttpServer.stop()` which
   `server.close()` (refuses new connections, lets in-flight requests finish), then
   waits up to 10s for connection drain, then force-closes remaining sockets.
4. After every hook completes, the process exits 0.

Two safety nets:
- **A second signal forces immediate exit** (exit code 130). Mirrors most production
  runtimes — one Ctrl+C = graceful, two = right now.
- **A 30-second hard-exit timeout.** If the orderly shutdown doesn't finish in that
  window, the process is force-exited. Prevents a misbehaving `onShutdown` hook from
  hanging Kubernetes / ECS / systemd indefinitely.

## Implementing `onShutdown` in your modules

Add the optional `onShutdown` hook to any `ModuleInterface`:

```ts
export const DatabaseModule: ModuleInterface = {
  keyname: "my-app.database",
  // ...
  onShutdown: async (container) => {
    const pool = container.resolve(DatabaseConnectionPool);
    await pool.drain();   // wait for in-flight queries
    await pool.close();   // release sockets
  },
};
```

Per-hook timeout defaults to 10 seconds. If a hook throws, `Kernel.stop()` logs the
error via `LogHandlerInterface` and continues with the next module — the goal is to
release as much as possible, not to abort on the first failure.

## Production vs. `node dist/main.js`

Two equally supported production entry points:

- **`node dist/main.js`** — traditional. You wire your own `Kernel` and `kernel.handle()`.
  No `@pristine-ts/cli` in your deploy unit. Lifecycle and signals are your problem.
- **`pristine start`** — production-grade. Lifecycle, signals, graceful drain, hard-exit
  timeout — all for free. Install `@pristine-ts/cli` globally on the host (or include it
  as a runtime dep) and use `CMD ["pristine", "start"]` in your Dockerfile.

The catalog demo uses `pristine start`. The
[Production chapter](../12-production/00.index.md) covers deploy patterns in more depth.

## Pitfalls

- **`pristine start` in a non-TTY with a stale manifest will exit non-zero**, not
  rebuild. This is intentional — CI / Docker shouldn't auto-rebuild. Run `pristine build`
  explicitly before `pristine start`.
- **The CLI's `kernel.start()` runs once for every command**, not just `pristine start`.
  This means `pristine help` boots your full kernel. If your AppModule has expensive
  `onInit` work, every CLI invocation pays that cost. (`@pristine-ts/cli` has a fallback
  path that uses a stripped-down `CliModule`-only kernel if your real AppModule can't be
  loaded — that's why broken configs don't break `pristine p:config:print`.)
- **`--port` and `--address` flags propagate to every `RuntimeServer`.** If you have both
  an HTTP server and a gRPC server, both will see the same port flag. Use per-module
  configuration values (e.g. `pristine.grpc.port`) instead when servers should differ.
