# `pristine verify` and instantiation tests

`pristine verify` answers "would my app boot cleanly?" without actually running it. It
builds a throw-away kernel, walks every boot phase (module registration, configuration
load, after-init), runs every registered `InstantiationTest`, and exits non-zero if
anything fails. It's the natural CI smoke test for a Pristine project.

## The minimal example

```sh
npm run verify
# (kernel boot output)
# [status:'Success', code:'0'] - Command 'verify' exited.
```

On a healthy app, it exits 0 and prints the per-phase outcomes via the LogHandler.

On a broken app â€” say, you removed a required config value â€” it exits non-zero with the
exact phase + reason:

```
âś– ConfigurationLoad failed: Required configuration value 'my-app.api-key' is missing.
[status:'Error', code:'1']
```

## What it actually does

Internally `pristine verify` calls `Kernel.verifyInstantiation()`, which runs **seven
phases** on a fresh kernel:

| # | Phase | What it does |
|---|---|---|
| 1 | `ModuleRegistration` | Walks every module's `providerRegistrations` and registers them with the container. |
| 2 | `ConfigurationCheck` | Statically inspects which required config values are missing (without invoking default resolvers â€” surfaces what *might* be missing). |
| 3 | `ConfigurationLoad` | Actually resolves every config value. If any required value can't be resolved, this fails. |
| 4 | `ServiceTagRegistration` | Registers every non-module-scoped tagged service (`@tag(...)` decorators without `@moduleScoped`). |
| 5 | `AfterInit` | Runs every module's `afterInit` hook. |
| 6 | `BootProbe` | Resolves `LogHandlerInterface` from the container â€” a smoke test that the logging pipeline survived boot. |
| 7 | `InstantiationTests` | Runs every `@tag(InstantiationTest)` service's `run()` method. |

Each phase is independently reported. If phase 3 fails, phases 4-7 are marked `Skipped`
(no point running them on a broken container). If phase 6 fails, phase 7 is skipped.

`pristine verify` exits **non-zero** if any phase failed *or* any instantiation test
failed.

## Skipping the tests phase

```sh
pristine verify --skip-tests
```

Runs phases 1-6 only. Useful when you want to verify boot purely, without paying for
whatever your `InstantiationTest`s do (often: hitting a DB or upstream API).

## Adding your own instantiation tests

This is where `pristine verify` becomes more than a boot smoke test. Want CI to fail if
your DB credentials are wrong? Implement `InstantiationTestInterface`:

```ts
// src/health/database-connectivity.test.ts
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable, DependencyContainer} from "tsyringe";
import {InstantiationTestInterface, InstantiationTestResultInterface} from "@pristine-ts/core";
import {DatabaseClient} from "../database/database.client";

@tag(ServiceDefinitionTagEnum.InstantiationTest)
@injectable()
export class DatabaseConnectivityTest implements InstantiationTestInterface {
  name = "database connectivity";
  description = "Pings the database to confirm credentials and network reachability.";

  async run(container: DependencyContainer): Promise<InstantiationTestResultInterface> {
    try {
      await container.resolve(DatabaseClient).ping();
      return {passed: true};
    } catch (e) {
      return {passed: false, message: (e as Error).message};
    }
  }
}
```

Register it like any service (via `importServices` or by relying on the `@tag` decorator's
self-registration when your AppModule imports the module that contains it). `pristine
verify` finds and runs it automatically.

Tests should be cheap and idempotent. They run on every boot probe, not just CI.

## CI integration

Put it in your GitHub Actions workflow before deploy:

```yaml
- run: npm ci
- run: npm run build
- run: npm run verify   # fails fast if your AppModule won't boot
```

Treats your boot graph as something CI is responsible for protecting. Lots of bugs show up
as "I forgot to import a module" or "I changed a config key but missed one usage" â€” `verify`
catches those at PR time instead of at deploy time.

## Pitfalls

- **`pristine verify` requires a fresh build.** Like every runtime command, it reads the
  build manifest first. Run `pristine build` in your CI step before `pristine verify`.
- **InstantiationTests run during normal `pristine start` too** if you're using the
  bootstrap path that includes them. Keep them cheap â€” they're on the boot critical path.
- **A test that throws is treated as failed.** You don't need try/catch in `run()` if
  you're happy with throws-as-failure. The `{passed, message}` shape exists so you can
  attach diagnostic info.
