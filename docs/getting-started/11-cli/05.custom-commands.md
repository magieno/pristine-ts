# Writing your own command

The Pristine CLI is extensible: any class that implements `CommandInterface` and tags
itself with `ServiceDefinitionTagEnum.Command` becomes a `pristine <name>` subcommand.
This is how the framework ships `pristine list`, `pristine info`, etc. ŌĆö and how your
project adds its own.

## The minimal example

The catalog demo has a `sync-catalog` command. Here's the full file
([`demo/projects/catalog/src/catalog/commands/sync-catalog.command.ts`](../../../demo/projects/catalog/src/catalog/commands/sync-catalog.command.ts)):

```ts
import {injectable} from "tsyringe";
import {moduleScoped, ServiceDefinitionTagEnum, tag} from "@pristine-ts/common";
import {CommandInterface, ConsoleManager, ExitCodeEnum} from "@pristine-ts/cli";
import {CatalogModuleKeyname} from "../catalog.module.keyname";
import {ProductService} from "../services/product.service";

@tag(ServiceDefinitionTagEnum.Command)
@moduleScoped(CatalogModuleKeyname)
@injectable()
export class SyncCatalogCommand implements CommandInterface<null> {
  optionsType = null;
  name = "sync-catalog";
  description = "Re-sync the local product cache from upstream.";

  constructor(
    private readonly consoleManager: ConsoleManager,
    private readonly productService: ProductService,
  ) {
  }

  async run(): Promise<ExitCodeEnum | number> {
    const count = await this.productService.syncFromUpstream();
    this.consoleManager.writeSuccess(`Synced ${count} products.`);
    return ExitCodeEnum.Success;
  }
}
```

Three decorators, three properties, one method. Run it:

```sh
pristine sync-catalog
# Ō£ö Success: Synced 2 products.
```

Pristine handed the command:
- A `ConsoleManager` via constructor injection (for output)
- The `ProductService` (your own injectable, also via DI)

And dispatched `run()` when the user typed the command name on the CLI.

## Anatomy of a command

```ts
@tag(ServiceDefinitionTagEnum.Command)        // ŌåÉ registers the class as a discoverable command
@moduleScoped(CatalogModuleKeyname)            // ŌåÉ only loaded if CatalogModule is imported
@injectable()                                   // ŌåÉ tsyringe constructor injection
export class SyncCatalogCommand implements CommandInterface<null> {
  optionsType = null;                           // ŌåÉ no CLI flags expected (see next section)
  name = "sync-catalog";                        // ŌåÉ the subcommand name (becomes `pristine sync-catalog`)
  description = "Re-sync...";                   // ŌåÉ shown by `pristine help`

  constructor(
    private readonly consoleManager: ConsoleManager,  // ŌåÉ any DI'd service
    private readonly productService: ProductService,
  ) {}

  async run(args: null): Promise<ExitCodeEnum | number> {
    // ... do work
    return ExitCodeEnum.Success;
  }
}
```

Three pieces matter:

- **`name`** ŌĆö what the user types on the CLI. Avoid the `p:` prefix; that's reserved for
  framework-built-in commands.
- **`optionsType`** ŌĆö `null` for commands with no flags. For commands with flags, set this
  to the constructor of a class with `class-validator` decorators (see below).
- **`run(args)`** ŌĆö your code. Return an `ExitCodeEnum` (`Success` = 0, `Error` = 1) or
  any number.

## Typed CLI options

For commands that accept flags, declare an options class:

```ts
// src/commands/sync-catalog.command-options.ts
import "reflect-metadata";
import {IsBoolean, IsNumber, IsOptional} from "@pristine-ts/class-validator";

export class SyncCatalogOptions {
  @IsOptional() @IsNumber() limit?: number;
  @IsOptional() @IsBoolean() "dry-run"?: boolean;
}
```

Then point the command's `optionsType` at that class (the class itself, not `new
SyncCatalogOptions()`):

```ts
@tag(ServiceDefinitionTagEnum.Command)
@injectable()
export class SyncCatalogCommand implements CommandInterface<SyncCatalogOptions> {
  optionsType = SyncCatalogOptions;
  name = "sync-catalog";
  description = "...";

  async run(args: SyncCatalogOptions): Promise<ExitCodeEnum | number> {
    const limit = args.limit ?? Infinity;
    const dryRun = args["dry-run"] === true;
    // ...
  }
}
```

Run with flags:

```sh
pristine sync-catalog --limit=50 --dry-run
```

Pristine:
1. Parses the raw `--key=value` pairs from `process.argv`.
2. Calls `plainToInstance(SyncCatalogOptions, parsed)` ŌĆö produces a real instance.
3. Runs class-validator on the instance.
4. If validation fails, prints constraint errors and exits non-zero. Your `run()` never
   gets called.
5. If validation passes, hands the typed instance to `run(args)`.

This is the contract: `args` inside `run()` is guaranteed to be a validated instance of
your `optionsType`, never a plain object.

> **Pitfall**: `optionsType` must be the class itself, **not** an instance. `optionsType
> = new SyncCatalogOptions()` is the older, broken pattern ŌĆö the data mapper expects a
> constructor, and an instance silently makes validation skip every check. TypeScript
> won't catch this; the new `CommandInterface<T>.optionsType: ClassConstructor<T> | null`
> contract makes it impossible since the recent CLI cleanup.

## Getting your command discovered

There are two ways:

**Option 1: Reference it in `importServices`** of the module that owns it (or your
AppModule):

```ts
export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  // ...
};

export const AppModule: AppModuleInterface = {
  // ...
  importServices: [
    SyncCatalogCommand,  // ŌåÉ ensures the file is imported, so @tag runs
  ],
};
```

**Option 2: Re-export it from a barrel** that the AppModule imports. The `@tag` decorator
runs as a side effect when the file is loaded; you just need *some* import path that
reaches it.

## What `pristine help` does with your command

`pristine help` lists every registered `Command`-tagged class, sorted alphabetically, with
its `description`. Your command shows up automatically. The framework-built-in commands
all use the `p:` prefix and a one-line description; your commands should follow the same
style.

## Pitfalls

- **`@injectable()` is required.** Without it, tsyringe can't construct your command and
  you'll see `TypeInfo not known for SyncCatalogCommand`.
- **Don't reuse a framework-reserved name.** Names starting with `p:` are framework-owned.
  Your command's name should be a plain word or a `feature:action` pattern.
- **Commands run in the same kernel as your app.** They share container state with
  everything `pristine start` runs. If your command depends on something only initialized
  by `afterInit`, that's fine ŌĆö the kernel runs through `afterInit` before dispatching
  any command.
