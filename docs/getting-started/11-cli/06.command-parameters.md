# Command parameters & options

A command's flags are declared as an **options class** â€” a plain class whose properties
are the flags, annotated with [`@pristine-ts/class-validator`](../../../packages/class-validator)
rules and the `@commandParameter` decorator. Point your command's `optionsType` at the
class and Pristine maps `--flag=value` argv onto a validated instance before `run()` is
called:

```ts
// src/commands/deploy.command-options.ts
import "reflect-metadata";
import {IsIn, IsOptional, IsString} from "@pristine-ts/class-validator";
import {commandParameter} from "@pristine-ts/cli";

export class DeployOptions {
  @commandParameter({question: "Which environment?"})
  @IsIn(["dev", "staging", "prod"])
  environment!: string;

  @commandParameter({flag: "dry-run"})
  @IsOptional()
  @IsString()
  dryRun?: string;
}
```

```ts
// src/commands/deploy.command.ts
import {injectable} from "tsyringe";
import {moduleScoped, ServiceDefinitionTagEnum, tag, ExitCode} from "@pristine-ts/common";
import {CommandInterface, CliOutput} from "@pristine-ts/cli";
import {DeployOptions} from "./deploy.command-options";

@tag(ServiceDefinitionTagEnum.Command)
@injectable()
export class DeployCommand implements CommandInterface<DeployOptions> {
  optionsType = DeployOptions;            // â†گ the class itself, never `new DeployOptions()`
  name = "deploy";
  description = "Deploy the app to an environment.";

  constructor(private readonly cliOutput: CliOutput) {}

  async run(args: DeployOptions): Promise<ExitCode | number> {
    // `args` is a validated DeployOptions instance â€” never a plain object.
    this.cliOutput.writeLine(`Deploying to ${args.environment}â€¦`);
    return ExitCode.Success;
  }
}
```

> The value `run()` receives is **guaranteed** to be a validated instance of `optionsType`.
> If mapping or validation fails, Pristine prints a clean error and exits non-zero â€” `run()`
> never runs. See [Helpful error messages](#helpful-error-messages) below.

## `@commandParameter` at a glance

`@commandParameter` is a **property decorator** that describes one CLI parameter. Every
field is optional â€” a bare `@commandParameter()` just marks the property as a known flag.

| Field          | Type                                   | What it does                                                                 |
| :------------- | :------------------------------------- | :--------------------------------------------------------------------------- |
| `flag`         | `string`                               | Bind the property to a differently-named flag (default = the property name). |
| `question`     | `string`                               | Ask for the value interactively when it's missing (TTY only).                |
| `sensitive`    | `boolean`                              | Mask the interactive input and never echo the value back.                    |
| `choices`      | array \| function \| provider class    | Constrain to a set; drives an arrow-key menu when asked interactively.       |
| `valueHint`    | `string`                               | The `<â€¦>` placeholder for this flag in the generated `Usage:` line.          |
| `errorMessage` | `string`                               | Override the whole "invalid value" message (with `%value%` / `%flag%`).      |

You don't need `@commandParameter` on every property â€” a property with only
`class-validator` rules is still a valid flag. Add `@commandParameter` when you want a flag
alias, an interactive prompt, choices, or custom error/usage wording.

## Binding flags (`flag`)

Arguments bind to properties **by name**, so a `camelCase` property would need a
`--camelCase` flag. Use `flag` to expose a dash-separated flag instead:

```ts
export class BuildOptions {
  @commandParameter({flag: "out-dir"})
  @IsOptional()
  @IsString()
  outputDirectory?: string;            // â†گ bound to --out-dir
}
```

```sh
pristine build --out-dir=./dist
```

Two parameters resolving to the same flag is a programming error and throws at startup.

## Required vs optional

A property is **required** unless it carries `@IsOptional()`. Required parameters that are
missing become a `Usage:` error (or an interactive prompt, if they declare a `question`);
optional ones simply stay `undefined`.

```ts
export class DeployOptions {
  @IsIn(["dev", "staging", "prod"])
  environment!: string;                // â†گ required

  @commandParameter({flag: "dry-run"})
  @IsOptional()
  @IsString()
  dryRun?: string;                     // â†گ optional
}
```

> **Pattern:** model a *required-but-askable* value as a normal **required** field carrying
> `@commandParameter({question})` â€” not as an optional field you hand-check inside `run()`.

## Asking interactively (`question`)

When a parameter declares a `question` and the value is absent, Pristine asks for it on the
terminal and uses the answer:

```ts
export class MigrateOptions {
  @commandParameter({question: "What is the database URL?"})
  @IsString()
  databaseUrl!: string;
}
```

```sh
pristine migrate
# What is the database URL? postgres://localhost/app
```

Prompting is **gated**: it only happens when the `pristine.cli.interactiveParameters`
configuration is `true` (the default) **and** stdin is an interactive terminal. Under CI or
a pipe, the prompt is skipped and the missing value falls through to validation â€” so CI
fails loudly instead of hanging. The answer is coerced and validated exactly like a typed
flag, and re-asked on an invalid answer.

### Secrets (`sensitive`)

Mark passwords, tokens and connection strings `sensitive` â€” the interactive input is masked
and the value is never echoed back (even in error messages):

```ts
export class LoginOptions {
  @commandParameter({question: "API token?", sensitive: true})
  @IsString()
  token!: string;
}
```

```sh
pristine login
# API token? ******
```

## Choices (single-select menu)

`choices` constrains a parameter to a set of allowed values. When the value is asked for
interactively, Pristine shows an **arrow-key menu** (`â†‘`/`â†“`, `Enter`) instead of a
free-text prompt. There are three shapes.

### Static list

A fixed array â€” bare strings, or `{name, value}` pairs when the label should differ from the
bound value:

```ts
export class DeployOptions {
  @commandParameter({
    question: "Which environment?",
    choices: ["dev", "staging", "prod"],
  })
  @IsIn(["dev", "staging", "prod"])      // â†گ also enforces the set for `--environment=â€¦`
  environment!: string;

  @commandParameter({
    question: "Which region?",
    choices: [
      {name: "US East (N. Virginia)", value: "us-east-1"},
      {name: "EU West (Ireland)", value: "eu-west-1"},
    ],
  })
  @IsString()
  region!: string;
}
```

```sh
pristine deploy
# ? Which environment?
#  â‌¯ dev
#    staging
#    prod
```

> A static `choices` list is **not** itself a validator â€” it powers the menu. Add
> `@IsIn(...)` / `@IsEnum(...)` if you also want `--environment=banana` rejected when passed
> as a flag.

### `@IsIn` / `@IsEnum` (automatic)

You don't even need `choices` for enums: a property constrained by `@IsIn([...])` or
`@IsEnum(MyEnum)` is surfaced as a menu automatically.

```ts
enum Format { Json = "json", Yaml = "yaml" }

export class ExportOptions {
  @commandParameter({question: "Output format?"})
  @IsEnum(Format)
  format!: Format;                       // â†گ menu of json / yaml, no `choices` needed
}
```

### Dynamic â€” resolver function

Pass a function to compute the list at prompt time (sync or async). It receives a context
whose `args` holds the values resolved so far, so a list can depend on an earlier answer:

```ts
export class ReleaseOptions {
  @commandParameter({question: "Repository?"})
  @IsString()
  repository!: string;

  @commandParameter({
    question: "Which tag?",
    // `ctx.args.repository` is whatever was passed or answered above.
    choices: async (ctx) => listTags(ctx.args.repository),
  })
  @IsString()
  tag!: string;
}
```

### Dynamic â€” provider class (dependency injection)

When the list needs framework services (an HTTP client, a file manager, a DB), reference an
**injectable provider class** instead of a closure. Pristine resolves it from the container
at prompt time, so its dependencies are injected normally:

```ts
// src/commands/region-choices.provider.ts
import {injectable} from "tsyringe";
import {moduleScoped} from "@pristine-ts/common";
import {
  CommandParameterChoicesProviderInterface,
  CommandParameterChoicesContext,
} from "@pristine-ts/cli";
import {RegionService} from "../services/region.service";

@injectable()
@moduleScoped(MyModuleKeyname)
export class RegionChoicesProvider implements CommandParameterChoicesProviderInterface {
  constructor(private readonly regionService: RegionService) {}

  async getChoices(context: CommandParameterChoicesContext) {
    const regions = await this.regionService.listEnabled();
    return regions.map((r) => ({name: r.label, value: r.id}));
  }
}
```

```ts
export class DeployOptions {
  @commandParameter({question: "Which region?", choices: RegionChoicesProvider})
  @IsString()
  region!: string;
}
```

> **Dynamic choices drive the interactive menu only.** A value passed as a flag
> (`--region=us-east-1`, e.g. in CI) is taken as-is â€” the resolver/provider is **not** run
> off the prompt path, so CI never pays its cost (network, credentials) and never fails on a
> transient lookup. If a dynamic value must also be validated when supplied non-interactively,
> add a `@IsIn(...)` (or a custom validator) alongside it. Static choices and `@IsIn`/`@IsEnum`
> are enforced everywhere.

## Helpful error messages

Argument problems are reported as a real CLI would, with exit code `64` (`ExitCode.Usage`) â€”
not as a raw constraint dump.

### Missing required â†’ a `Usage:` synopsis

When a required value is missing (and can't be prompted for), Pristine prints the command's
generated usage line. `valueHint` sets each flag's `<â€¦>` placeholder:

```ts
export class AddKeyOptions {
  @commandParameter({valueHint: "name"})
  @IsString()
  name!: string;

  @commandParameter({flag: "pubkey", valueHint: "key-or-file"})
  @IsOptional()
  @IsString()
  pubkey?: string;

  @commandParameter()
  @IsOptional()
  @IsBoolean()
  rotate?: boolean;
}
```

```sh
myapp key:add
# Usage: myapp key:add --name=<name> [--pubkey=<key-or-file>] [--rotate]
# exit 64
```

Required values render as `--flag=<hint>`, optional as `[--flag=<hint>]`, and booleans as
`[--flag]`. The program name (`myapp` above) comes from the `pristine.cli.binName`
configuration, falling back to the invoked binary's name, then `pristine` â€” see
[Configuration](#configuration).

### Invalid value â†’ `Invalid <flag> '<value>'. <reason>`

When a value *is* supplied but fails validation, Pristine prints a one-line reason. By
default that's the validator's own message; `errorMessage` overrides the **whole** line and
supports `%value%` and `%flag%` placeholders:

```ts
export class AddKeyOptions {
  @commandParameter({
    errorMessage: "Invalid name '%value%'. Use lowercase letters, digits and dashes.",
  })
  @IsString()
  @Matches(/^[a-z][a-z0-9-]*$/)
  name!: string;
}
```

```sh
myapp key:add --name=Web
# Invalid name 'Web'. Use lowercase letters, digits and dashes.
# exit 64
```

For a `sensitive` parameter the value is rendered as `<hidden>`, so secrets never leak into
an error line.

## Configuration

Two CLI configuration keys affect parameters. Set them in `pristine.config.ts` under
`config` (keyed by the configuration key), or via the matching environment variable.

| Configuration key                     | Env var                              | Default | Effect                                                                 |
| :------------------------------------- | :----------------------------------- | :------ | :--------------------------------------------------------------------- |
| `pristine.cli.interactiveParameters`   | `PRISTINE_CLI_INTERACTIVE_PARAMETERS`| `true`  | Master switch for interactive prompting. Always skipped off a TTY too. |
| `pristine.cli.binName`                 | `PRISTINE_CLI_BIN_NAME`              | *(auto)*| Program name shown in `Usage:` lines (e.g. `myapp`).                   |

```ts
// pristine.config.ts
import {defineConfig} from "@pristine-ts/cli";

export default defineConfig({
  config: {
    "pristine.cli.binName": "myapp",
    // Turn prompting off everywhere (e.g. a non-interactive image):
    // "pristine.cli.interactiveParameters": false,
  },
});
```

```sh
# Or per-invocation via env:
PRISTINE_CLI_BIN_NAME=myapp PRISTINE_CLI_INTERACTIVE_PARAMETERS=false pristine deploy
```

When `pristine.cli.binName` is unset, the program name is derived from `basename(argv[1])`
(so a binary launched as `myapp` shows `myapp`), falling back to `pristine`.

## A complete example

```ts
// src/commands/deploy.command-options.ts
import "reflect-metadata";
import {IsBoolean, IsEnum, IsIn, IsOptional, IsString} from "@pristine-ts/class-validator";
import {commandParameter} from "@pristine-ts/cli";
import {RegionChoicesProvider} from "./region-choices.provider";

export class DeployOptions {
  @commandParameter({question: "Which environment?", valueHint: "env"})
  @IsIn(["dev", "staging", "prod"])
  environment!: string;

  @commandParameter({question: "Which region?", choices: RegionChoicesProvider})
  @IsString()
  region!: string;

  @commandParameter({
    errorMessage: "Invalid tag '%value%'. Expected something like v1.2.3.",
  })
  @IsOptional()
  @IsString()
  tag?: string;

  @commandParameter({question: "Deploy token?", sensitive: true})
  @IsString()
  token!: string;

  @commandParameter()
  @IsOptional()
  @IsBoolean()
  force?: boolean;
}
```

```sh
# Fully interactive (a TTY, nothing passed):
pristine deploy
# ? Which environment?  â‌¯ dev / staging / prod
# ? Which region?       â‌¯ US East (N. Virginia) / EU West (Ireland)
# Deploy token? ******

# Fully scripted (CI): everything on flags, no prompts, validated against @IsIn etc.
pristine deploy --environment=prod --region=us-east-1 --token=$TOKEN --force

# A required flag omitted, non-interactively:
pristine deploy --environment=prod --region=us-east-1
# Usage: pristine deploy --environment=<env> --region=<region> [--tag=<tag>] --token=<token> [--force]
# exit 64
```

## Reference

**`@commandParameter(options)` fields:** `flag`, `question`, `sensitive`, `choices`,
`valueHint`, `errorMessage` â€” see [the table above](#commandparameter-at-a-glance).

**Exit codes for argument problems:** `64` (`ExitCode.Usage`) for a missing required value
or an invalid value; a successful command returns whatever `run()` returns (`0` =
`ExitCode.Success`).

## Pitfalls

- **`optionsType` must be the class, not an instance.** `optionsType = DeployOptions`, never
  `new DeployOptions()` â€” the `CommandInterface<T>.optionsType: ClassConstructor<T> | null`
  contract enforces this.
- **`reflect-metadata` must be imported** wherever decorators run (top of the options file),
  or `class-validator` and `@commandParameter` metadata won't be read.
- **A `choices` array is not validation.** Pair it with `@IsIn`/`@IsEnum` if a flag value
  must be rejected when it's outside the set.
- **Dynamic choices won't catch a bad `--flag` in CI.** They only power the interactive menu;
  add `@IsIn(...)` to enforce a dynamic value off the prompt path.
