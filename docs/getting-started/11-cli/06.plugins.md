# Plugins (commands from other packages)

Custom commands you write live in your project. Plugins let *external* npm packages ship
commands into your CLI without your AppModule importing them at runtime. Useful for:

- Tooling-only commands you don't want loaded in production (generators, codemods, linters)
- Shared internal commands across multiple projects in an organization
- Distributing CLI extensions as ordinary npm packages

## The minimal example

A plugin is just an npm package that exports one or more `*Module` symbols. Each module
contributes its services (commands, typically) via the same `@tag(Command)` pattern your
project would use:

```ts
// my-plugin/src/index.ts
import {ModuleInterface, moduleScoped, ServiceDefinitionTagEnum, tag} from "@pristine-ts/common";
import {injectable} from "tsyringe";
import {CommandInterface, ConsoleManager, ExitCodeEnum} from "@pristine-ts/cli";

@tag(ServiceDefinitionTagEnum.Command)
@moduleScoped("my-plugin")
@injectable()
export class HelloCommand implements CommandInterface<null> {
  optionsType = null;
  name = "my-plugin:hello";
  description = "Says hello.";
  constructor(private readonly consoleManager: ConsoleManager) {}
  async run() {
    this.consoleManager.writeLine("Hello!");
    return ExitCodeEnum.Success;
  }
}

export const MyPluginModule: ModuleInterface = {
  keyname: "my-plugin",
};
```

The consumer's `pristine.config.ts`:

```ts
import {defineConfig} from "@pristine-ts/cli";

export default defineConfig({
  appModule: {path: "dist/app.module.js"},
  plugins: [
    "my-plugin",
    {name: "@my-org/codegen", options: {/* reserved for future use */}},
  ],
});
```

That's it. Run:

```sh
npm install --save-dev my-plugin
pristine my-plugin:hello
# Hello!
```

The plugin's commands now show up in `pristine help` alongside the built-ins and your
own.

## How resolution works

The CLI resolves each entry in `plugins[]` from the **consumer's** `node_modules` (using
`createRequire` anchored at the `pristine.config.ts` location), not from
`@pristine-ts/cli`'s install location. This is the same module-resolution semantics as
any other `require("my-plugin")` from your project — monorepos with hoisted deps work
naturally.

For each resolved plugin, the CLI:
1. Dynamic-imports the package's entry file.
2. Walks the exports collecting every name that ends in `Module` and looks like a
   `ModuleInterface` (has a `keyname` string).
3. Synthesizes a wrapper AppModule whose `importModules` includes the user's AppModule
   plus every plugin-contributed module (deduplicated by reference).

After this wrapping, the kernel boots normally and the plugin's modules go through the
same lifecycle as everything else — `onInit`, configuration definitions, `afterInit`,
etc.

## What plugin authors must export

A plugin must export **at least one** symbol matching `*Module` whose value has a
`keyname` string. If you export nothing matching that shape, the CLI fails loud:

```
[pristine] Plugin 'my-plugin' exported no '*Module' symbols. A plugin must export at
least one ModuleInterface (e.g. `export const FooModule: ModuleInterface = { keyname:
"...", ... }`).
```

Silent loading is a footgun — "why isn't my command showing up?" would otherwise be
debuggable only by reading source.

## Diagnostics

`pristine info` lists every loaded plugin under a dedicated section:

```
Plugins (2):
  - my-plugin
  - @my-org/codegen
Imported modules (14):
  - my-plugin
  - @my-org/codegen
  - ...
```

The wrapped AppModule's keyname is `<your-keyname>.with-plugins` (so `catalog.app`
becomes `catalog.app.with-plugins`) — that change is also visible in `pristine info`.

## Command-name collisions

If two plugins (or a plugin and a built-in command) register commands with the same
name, the CLI prints a stderr warning at boot:

```
[pristine] WARNING: command 'sync' is registered 2 times. Only the first match will be
dispatched.
```

The first registered match wins. To resolve, rename one. Or, more robustly, prefix every
plugin command with the plugin's package name (the catalog command would have been called
`catalog:sync` if it shipped from a plugin).

## When a plugin fails to load

Missing package, broken import, exported nothing — any of these prints a clear stderr
error and **the CLI continues**:

```
[pristine] Failed to resolve plugin 'nonexistent-plugin' from
'/abs/path/pristine.config.ts': Cannot find module 'nonexistent-plugin'. Make sure the
plugin package is installed in your project's node_modules.
[pristine] Continuing without plugins. Fix the plugin config and re-run.
```

Built-in commands (`pristine p:help`, `pristine p:config:print`, etc.) still work. You
can use them to debug. This was deliberate — a broken plugin shouldn't lock you out of
the tools you'd use to fix it.

## Distinguishing plugins from `importModules`

Both approaches add modules to your runtime. The difference is **which kind of module**:

| | `importModules` (in AppModule) | `plugins` (in pristine.config.ts) |
|---|---|---|
| Loaded in production by `node dist/main.js`? | Yes | No |
| Loaded by `pristine start` / `verify` / etc.? | Yes | Yes |
| Source of truth | Your AppModule code | Your config file |
| Right tool for | Framework modules, app features | Tooling-only commands |

Plugins are explicitly *not* part of your production AppModule. A generator command
shouldn't run inside your production process; it should only be available when you're
running the CLI.

## Pitfalls

- **Plugins resolve from `pristine.config.ts`'s directory.** In a monorepo, that's
  usually fine, but if your config lives at the repo root and the plugin is hoisted into a
  workspace's `node_modules`, you might need to declare it as a root devDep.
- **Plugin options (`{name, options}`) aren't passed anywhere yet.** Reserved for a
  future config-passing mechanism. Use the string form for now.
- **Plugin modules' configurationDefinitions still apply.** If a plugin declares a
  required config value, your boot will fail until you provide it. Plugins aren't
  "isolated" — they participate in your kernel's lifecycle fully.
