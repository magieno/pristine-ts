# Production

Everything that's about *shipping* the app, not building it. Up to this point the guide
has been about authoring code; this chapter is about running that code on a real machine
where signals arrive, hosts die, orchestrators reschedule containers, and you do not get
to attach a debugger.

Pristine gives you four production-relevant primitives:

| Primitive | Where defined | Purpose |
|---|---|---|
| `pristine start` | `@pristine-ts/cli` | Boot AppModule, run servers, handle SIGTERM/SIGINT, exit |
| `onShutdown` | `ModuleInterface` | Per-module graceful-teardown hook |
| `pristine verify` | `@pristine-ts/cli` | Boot the app in dry-run mode, run instantiation tests, report |
| `InstantiationTestInterface` | `@pristine-ts/core` | Application-defined readiness/health checks |

If you've read the [Quick start](../10-quick-start/00.index.md) and the [CLI chapter](../11-cli/00.index.md),
you've used `pristine start` and `pristine verify` informally. This chapter goes deeper:
what the lifecycle actually looks like, what signal handling guarantees you get, where
your hooks fit, and the deployment-shape decisions you'll make.

> **You'll learn**
> - The full `pristine start` lifecycle, signal-by-signal
> - How `onShutdown` and `Kernel.stop()` cooperate
> - How to wire `InstantiationTestInterface` for readiness probes
> - The deployment topologies Pristine works well in (and where it doesn't fit)
> - The pitfalls that come up only at production scale (zombie processes, lost shutdown signals, partial boots)
