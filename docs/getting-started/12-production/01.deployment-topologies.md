# Deployment topologies

Pristine is a Node.js framework with no opinions about your hosting platform. It runs
anywhere Node 18+ runs. That said, the assumptions `pristine start` makes about its
environment are worth being explicit about — they're what decide whether a deployment
shape is a natural fit or a fight.

> **You'll learn**
> - What `pristine start` assumes about the host
> - The four canonical topologies (Docker, systemd, ECS, Lambda) and how each maps to Pristine
> - When *not* to use `pristine start` (Lambda, edge runtimes)

## What `pristine start` assumes

The `pristine start` command is the production entry point. It assumes:

1. **The process is allowed to receive signals.** SIGTERM triggers graceful shutdown; SIGINT
   does the same (for interactive `Ctrl+C`). Anything that strips signals — `npm start`
   wrappers that don't `exec`, certain container PID 1 setups — breaks shutdown.
2. **The process is the long-running unit of work.** Pristine sets up a heartbeat timer to
   keep the event loop alive after servers start. Restart-on-exit policies (Kubernetes
   `restartPolicy: Always`, systemd `Restart=always`, ECS service restarts) are how the
   app gets new processes after a crash. Pristine does not self-restart in place.
3. **There's a writable filesystem somewhere for build artifacts.** `pristine build`
   produces `dist/` and `.pristine/build-manifest.json` at build time, not at start time.
   You're expected to bake these into the image. `pristine start` reads them; it does not
   write.
4. **The kernel boots once per process.** No fork/cluster pattern is provided. Scale by
   running more processes (multiple containers, multiple ECS tasks, multiple systemd
   instances), not by forking inside one.

If your platform meets these four assumptions, `pristine start` is the right entry point.
If it doesn't (Lambda, Cloudflare Workers, edge functions), you skip `pristine start`
entirely and call `kernel.handleRequest()` directly from the platform's handler.

## Docker

The most common topology. Build the image with `pristine build` baked in:

```dockerfile
FROM node:20-slim AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npx pristine build

FROM node:20-slim
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/.pristine ./.pristine
COPY --from=build /app/node_modules ./node_modules
COPY --from=build /app/package.json ./package.json
COPY --from=build /app/pristine.config.ts ./pristine.config.ts
# Signal-forwarding init so SIGTERM reaches Node, not just PID 1.
RUN apt-get update && apt-get install -y --no-install-recommends tini && rm -rf /var/lib/apt/lists/*
ENTRYPOINT ["tini", "--"]
CMD ["npx", "pristine", "start"]
```

Three points worth calling out:

- **Use an init like `tini` (or `--init` on `docker run`).** Without it, Node runs as PID 1
  and doesn't get default signal handlers. SIGTERM from `docker stop` or Kubernetes is
  delivered, but the handler-not-installed-by-default semantics differ from typical
  Node setups. `tini` reaps zombies and forwards signals cleanly. Bake it in or use
  Docker's built-in init flag.
- **Don't `npm start` to launch.** `npm` does not exec child processes by default — it
  spawns Node as a grandchild and may eat signals. Either configure your `start` script
  to use `exec` explicitly, or invoke `npx pristine start` directly.
- **Build artifacts go into the image, not into a volume.** `dist/` and `.pristine/` should
  be immutable per-image. Mounting them from a volume means a single image can have its
  behavior changed by what's on the host, which defeats the point of containerization.

## systemd

For long-running VMs (the platform-as-a-service experience without the platform). A
service unit at `/etc/systemd/system/myapp.service`:

```ini
[Unit]
Description=My Pristine app
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=myapp
WorkingDirectory=/srv/myapp
ExecStart=/usr/bin/npx pristine start
Restart=on-failure
RestartSec=5
KillSignal=SIGTERM
TimeoutStopSec=35
Environment=NODE_ENV=production
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

The important details:

- `KillSignal=SIGTERM` matches what `pristine start` listens for. systemd's default is
  SIGTERM already, but state it explicitly so future-you doesn't have to dig.
- `TimeoutStopSec=35` sits just above Pristine's 30-second hard-exit timeout — gives the
  app a chance to finish its own teardown before systemd escalates to SIGKILL.
- `Restart=on-failure` (not `always`) means clean exits (e.g., from `pristine verify`
  reporting failure) don't get retried. If you want crash-and-restart, that's still
  on-failure; "always" loops forever even on intentional shutdowns.

## ECS / Fargate

ECS is conceptually Docker + a scheduler. Same Docker assumptions apply (use `tini`,
exec correctly, bake build artifacts). ECS-specific:

- **Set `stopTimeout` on the container definition to ≥35 seconds.** ECS's default is 30
  seconds, which exactly matches Pristine's hard-exit timeout — meaning ECS may send
  SIGKILL the same instant Pristine was about to force-exit. Give yourself 5+ seconds
  of headroom.
- **Health checks**: ECS supports container-level `HEALTHCHECK` (from the Dockerfile)
  and ELB target group health checks (HTTP). Wire ELB health checks to a route in your
  app — see [Health checks](03.health-checks.md). Container-level `HEALTHCHECK` is fine
  for "process is responsive" but doesn't tell ECS whether the app's dependencies are
  reachable.
- **Task placement**: scaling is by task count, not by in-process concurrency. Aim for
  a few small tasks rather than one large one — Pristine's lifecycle is per-process.

## Lambda (and other request-per-invocation runtimes)

`pristine start` does **not** apply here. Lambda's contract is one invocation = one
request; there's no long-running process to manage signals on. Instead:

```ts
// handler.ts
import "reflect-metadata";
import {Kernel} from "@pristine-ts/core";
import {LambdaEventMapper} from "@pristine-ts/aws";
import {AppModule} from "./app.module";

const kernelPromise = (async () => {
  const k = new Kernel();
  await k.start(AppModule);
  return k;
})();

export const handler = async (event: any, context: any) => {
  const kernel = await kernelPromise;
  const mapper = kernel.container.resolve(LambdaEventMapper);
  const request = mapper.map(event);
  const response = await kernel.handleRequest(request);
  return mapper.reverseMap(response);
};
```

Top-level kernel initialization gives you a warm start: subsequent invocations on the
same container reuse the booted kernel. The first invocation pays the boot cost.

Two important caveats:

- **`onShutdown` doesn't run.** Lambda kills containers without a graceful signal. Don't
  rely on `onShutdown` for anything other than `pristine start` setups.
- **`@pristine-ts/aws` exists for this case.** It provides event mappers for API
  Gateway, SQS, SNS, S3, EventBridge, and DynamoDB streams. Don't roll your own.

## Cloudflare Workers / edge runtimes

Pristine is built on Node APIs (`process`, signals, `Buffer`, etc.) so it doesn't run on
V8-isolate-only edge runtimes. If your workload requires edge deployment, Pristine is the
wrong choice — pick a runtime-native framework.

## Pitfalls

- **Wrapping `pristine start` in a shell loop.** Don't `while true; do npx pristine start;
  done`. The framework expects to be the long-running process; if you re-exec on every
  crash you defeat orchestrator retry logic and lose telemetry.
- **Forgetting `tini` in containers.** Most "my SIGTERM isn't doing anything" bug
  reports trace back to this.
- **`npm start` in the Dockerfile.** Use `npx pristine start` or `node` directly. `npm`
  in the middle silently breaks signal delivery on many setups.
- **Mounting `dist/` from outside the container.** Treat it as immutable per-image.
