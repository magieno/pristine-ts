# Graceful shutdown and `onShutdown`

When `pristine start` receives SIGTERM (or SIGINT), it doesn't immediately exit. It runs
a coordinated teardown sequence so that in-flight requests finish, connections close
cleanly, and external state (DB transactions, queue messages) lands in a defined state
before the process disappears.

This section walks through that sequence end-to-end and shows you where your code plugs in.

> **You'll learn**
> - The full sequence of events between SIGTERM and `process.exit`
> - How to implement `onShutdown` in your modules
> - The hard-exit timeout and what it protects against

## The lifecycle, signal-by-signal

```
[pristine start running]
            |
            v
   SIGTERM arrives
            |
            v
   "Received SIGTERM, shutting down gracefully..."
            |
            v
   Start hard-exit timer (30s) — unref'd
            |
            v
   Kernel.stop() begins
            |
            +-> AppModule.onShutdown()          (outer first)
            +-> child module 1.onShutdown()
            +-> child module 2.onShutdown()
            +-> ...
            +-> deepest module.onShutdown()     (inner last)
            |
            v
   "Shutdown complete."
            |
            v
   process exits 0
```

The pristine-managed parts:

1. **Signal registration.** `pristine start` calls `process.on("SIGTERM", ...)` and
   `process.on("SIGINT", ...)` once after the runtime servers are up. It also pins a
   heartbeat `setInterval` so Node's event loop stays alive after servers stop and
   before shutdown finishes.
2. **Idempotency.** A second signal during shutdown does not re-enter `Kernel.stop()`. It
   short-circuits to `process.exit(130)` — interpreted as "user forced exit." This mirrors
   Ctrl+C behavior of most production runtimes.
3. **Hard-exit timeout.** A 30-second timer is armed when the first signal arrives. If
   `Kernel.stop()` doesn't return before then, the process is `process.exit(1)`'d. This
   prevents wedged `onShutdown` hooks from blocking orchestrator restarts indefinitely.

## Implementing `onShutdown` in your module

`ModuleInterface.onShutdown` is the hook every module gets:

```ts
import {ModuleInterface} from "@pristine-ts/common";
import {DependencyContainer} from "tsyringe";
import {CatalogModuleKeyname} from "./catalog.module.keyname";
import {ProductRepository} from "./repositories/product.repository";

export const CatalogModule: ModuleInterface = {
  keyname: CatalogModuleKeyname,
  importModules: [],
  providerRegistrations: [
    {token: ProductRepository, useClass: ProductRepository},
  ],

  async onShutdown(container: DependencyContainer): Promise<void> {
    // Close the DB pool, flush write buffers, etc.
    const repo = container.resolve(ProductRepository);
    await repo.close();
  },
};
```

What you put in `onShutdown`:

- **Close pooled resources** (DB connection pools, HTTP keep-alive agents, Redis clients).
- **Drain background work** (queue consumers stop accepting new messages, finish what's
  in flight).
- **Flush buffered writes** (telemetry exporters, log batches, file-buffered output).
- **Cancel timers** that `setInterval`'d themselves into the event loop.

What NOT to put in `onShutdown`:

- **Long-running operations** (network calls that might hang). Use a per-hook timeout
  internally — the framework gives you 10 seconds before it logs a warning and moves on.
- **Anything that can fail loudly**: a thrown error in one module's hook does not stop
  the next module's hook from running, but it's still noisy. Catch and log instead.
- **Code that touches *other* modules' resources.** Hooks run outer-to-inner; by the
  time your inner module's `onShutdown` runs, the outer one is done. Dependency
  inversion still applies — own your own resources.

## Hook ordering: outer-to-inner

If `AppModule` imports `HttpModule` which imports `NetworkingModule`, the hook order on
shutdown is:

```
AppModule.onShutdown()         <-- runs first
HttpModule.onShutdown()         <-- stops KernelHttpServer here
NetworkingModule.onShutdown()   <-- runs last
```

The rationale: outer modules typically *use* inner modules. If `NetworkingModule` tore
itself down before `HttpModule.onShutdown` had a chance to drain pending requests, the
http module's drain would call into a half-dead router. Outer-to-inner means the high
levels can still rely on their dependencies during teardown.

You generally only implement `onShutdown` on **your own** modules. The framework's
internal modules (NetworkingModule, HttpModule, etc.) handle their own teardown.

## Idempotency requirement

`Kernel.stop()` will only invoke each module's `onShutdown` once. But your hook should
still be safe against being called multiple times — defensive programming for the case
where a future kernel API change loosens that guarantee, or where you have a custom
runtime that calls `kernel.stop()` itself:

```ts
async onShutdown(container) {
  if (this.shuttingDown) return;
  this.shuttingDown = true;
  await this.realShutdown(container);
}
```

For most apps this is overkill — but it's the kind of safety belt that costs nothing.

## When you can't use `pristine start`

For Lambda, Workers, or any setup where Pristine isn't the long-running process,
`onShutdown` is essentially dead code: the runtime kills your process without warning,
SIGTERM never arrives, and the framework's signal handlers are never installed. In those
cases:

- Don't rely on `onShutdown` for correctness. Persist state continuously, not at shutdown.
- Make every operation idempotent — assume your process may vanish mid-request.
- Use the platform's lifecycle hooks (Lambda extensions, etc.) if you really need
  shutdown notification.

## Pitfalls

- **Forgetting to make hooks idempotent.** The framework's guarantee is "at most once,"
  but you should code as if it could be called twice.
- **Doing too much in `onShutdown`.** Long DB cleanup, calling external APIs, "let me
  just flush this last batch" — these regularly blow past 30 seconds and get
  hard-killed. Move long teardown to a separate `pristine` command that runs *before*
  rolling the process, not after the signal.
- **Assuming hooks run in import order.** They run outer-to-inner, which is *roughly*
  inverse to construction order. Don't rely on a specific peer-module sibling order — if
  module A's shutdown depends on module B being alive, A should import B (so B is
  inner, and B's shutdown runs after A's).
- **Stopping `RuntimeServer`s manually.** `Kernel.stop()` runs your modules'
  `onShutdown` hooks, and `HttpModule.onShutdown` calls `KernelHttpServer.stop()`
  internally. Calling `server.stop()` yourself causes double-stop. Just let `Kernel.stop`
  do it.
