# Health checks with `InstantiationTestInterface`

Production orchestrators want two distinct yes/no signals from every process:

- **Liveness**: is the process responsive? (If no, kill and restart.)
- **Readiness**: should this process receive traffic right now? (If no, route around it.)

Pristine doesn't provide HTTP endpoints for these — that's a controller-author decision.
But it does provide two pieces that make health checks easy to wire up consistently:

1. **`pristine verify`** — boots the app in dry-run mode and reports whether every module
   could be instantiated. Run this as part of CI or as a post-deploy smoke test.
2. **`InstantiationTestInterface`** — your custom assertions about the app's readiness.
   `pristine verify` runs them; you can also resolve and run them from a controller for
   a runtime readiness endpoint.

> **You'll learn**
> - What `pristine verify` actually does
> - How to write an `InstantiationTestInterface`
> - How to expose readiness as an HTTP endpoint
> - The liveness/readiness distinction in Pristine-shaped apps

## `pristine verify`: build-time boot sanity check

The command boots the AppModule against its configuration in a throwaway kernel, captures
the per-phase outcomes, runs every embedder-registered `InstantiationTestInterface`, and
reports the result through the project's `LogHandlerInterface`. Exit code is 0 on full
success, non-zero on any error.

Typical CI integration:

```yaml
# .github/workflows/ci.yml
- name: Verify app instantiates
  run: npx pristine verify

- name: Build
  run: npx pristine build

- name: Verify built artifacts still instantiate
  run: npx pristine verify
```

Two `verify` runs is intentional: the first catches configuration errors before you
spend time building; the second catches "the dist/ output is broken even though source
was fine" (mostly a sanity check, but cheap).

`pristine verify --skip-tests` skips the `InstantiationTestInterface` phase. Useful when
your tests require external dependencies that CI can't reach — but in that case you
probably want a second `pristine verify` run from a staging environment.

## Writing an `InstantiationTestInterface`

The interface is two methods plus a name:

```ts
import "reflect-metadata";
import {tag, ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {injectable, inject} from "tsyringe";
import {DependencyContainer} from "tsyringe";
import {
  InstantiationTestInterface,
  InstantiationTestResultInterface,
} from "@pristine-ts/core";
import {ProductRepository} from "./catalog/repositories/product.repository";

@tag(ServiceDefinitionTagEnum.InstantiationTest)
@injectable()
export class CatalogDatabaseReachableTest implements InstantiationTestInterface {
  name = "catalog-database-reachable";
  description = "Catalog repository can connect to the product database.";

  async run(container: DependencyContainer): Promise<InstantiationTestResultInterface> {
    const repo = container.resolve(ProductRepository);
    try {
      await repo.ping();  // however your repo signals reachability
      return {passed: true};
    } catch (error) {
      return {
        passed: false,
        message: "ProductRepository.ping() failed",
        details: {error: (error as Error).message},
      };
    }
  }
}
```

Three contract details worth knowing:

1. **Tests should be cheap.** They run every time `pristine verify` runs (and possibly
   every time your readiness endpoint is hit). A 5-second probe is too slow; a 50ms ping
   is fine.
2. **Tests should be side-effect-free.** Don't `INSERT` into the DB to check reachability;
   `SELECT 1` instead. A test that mutates state will surprise you when verify runs in CI.
3. **Return, don't throw.** The framework catches throws and records them as failed, but
   structured returns let you attach `message` and `details` that surface in the report.
   Use that.

## Exposing readiness over HTTP

`InstantiationTestInterface` is registered as a normal tagged service. Inject them all
into a controller and run them on demand:

```ts
import "reflect-metadata";
import {controller, route} from "@pristine-ts/networking";
import {HttpMethod} from "@pristine-ts/common";
import {injectable, injectAll} from "tsyringe";
import {ServiceDefinitionTagEnum} from "@pristine-ts/common";
import {InstantiationTestInterface} from "@pristine-ts/core";

@controller("/health")
@injectable()
export class HealthController {
  constructor(
    @injectAll(ServiceDefinitionTagEnum.InstantiationTest)
    private readonly tests: InstantiationTestInterface[],
  ) {}

  @route(HttpMethod.Get, "/live")
  live() {
    return {status: 200, body: {ok: true}};
  }

  @route(HttpMethod.Get, "/ready")
  async ready() {
    const results = await Promise.all(
      this.tests.map(async t => ({
        name: t.name,
        ...(await t.run(/* container — see below */ undefined as any)),
      })),
    );
    const allPassed = results.every(r => r.passed);
    return {
      status: allPassed ? 200 : 503,
      body: {ok: allPassed, checks: results},
    };
  }
}
```

A note on the `container` argument: `InstantiationTestInterface.run(container)` is typed
to receive the verifier's container. When you call it from a controller, you have two
options:

- **Resolve dependencies in the test's constructor**, not in `run()` — then ignore the
  container argument. Cleaner for the readiness-endpoint use case.
- **Inject the kernel's container** into the controller and pass it through. Slightly
  more flexible but couples the controller to the kernel.

The first option is what most apps do. Treat `run()`'s container argument as a `verify`-
only feature you can opt out of by depending on constructor injection.

## Liveness vs. readiness in Pristine

A clean split:

| Probe | Endpoint | Says | Failure response |
|---|---|---|---|
| Liveness | `GET /health/live` | "the process is responding" | Restart the container |
| Readiness | `GET /health/ready` | "the app can serve traffic right now" | Route around this container |

`GET /health/live` should be **trivial** — just return 200. If the controller method can
execute, the process is alive. Resist the urge to add DB checks here; you'll cause
restart loops when the DB hiccups for 30 seconds.

`GET /health/ready` is where you put dependency probes. If the DB is down, returning 503
tells the load balancer to stop sending traffic *to this container* without restarting
it — the operator (or autoscaler) decides what to do.

The split matters because Kubernetes (and ECS, and most orchestrators) treat them
differently: failing liveness restarts, failing readiness routes around. Conflating them
turns "the DB is briefly slow" into "restart every container in the fleet."

## Pitfalls

- **DB checks in the liveness endpoint.** Causes restart loops during dependency
  flapping. Liveness should ask "is Node alive," not "is my Postgres alive."
- **Tests that call external APIs you don't control.** Your readiness depends on
  third-party SLAs. Either accept that and document it, or fall back to a cached "last
  known good" check.
- **`pristine verify` in production paths.** It boots a full new kernel. Use it for CI
  and post-deploy smoke testing, not as the implementation of your readiness probe — that
  would re-boot the app every time the load balancer probes.
- **Skipping tests with `--skip-tests` permanently.** Once you've added the flag to
  "make CI green," it tends to stay. If a test is flaky, fix the test or remove it; don't
  silently skip.
