# Performance and warm-up

This section is about the operational tail: cold starts, the build manifest, what
Pristine caches, and when you have to think about all of that vs. when you can ignore it.

> **You'll learn**
> - What `.pristine/build-manifest.json` is and what it does for startup time
> - Pristine's cold-start characteristics
> - How to make container readiness gates correctly waited-for
> - When `pristine build` staleness detection saves you and when it bites

## The build manifest

`pristine build` produces two things:

1. The compiled `dist/` output (TypeScript → JavaScript).
2. A manifest at `.pristine/build-manifest.json` describing what was built and how.

The manifest is what `pristine start`, `pristine verify`, and other commands use to
**avoid recompiling when nothing changed**. On a hot machine where you've already built,
running `pristine start` is fast: the framework reads the manifest, decides the dist is
up to date, and boots from it directly.

Staleness is detected by comparing source-tree hashes against what the manifest recorded.
If anything in the configured `sourcePath` changed since the last build, the next command
that needs a built app will rebuild before proceeding.

For production:

- **Bake the manifest into your image.** It's in `.pristine/build-manifest.json` — copy
  it alongside `dist/`. Without it, the next `pristine start` invocation has nothing to
  compare against and may try to rebuild at startup.
- **Don't ignore `.pristine/` in `.dockerignore`.** Common mistake when copying docker
  configs from other projects.
- **The manifest is the contract between build-time and runtime.** Treat it as part of
  the deployable artifact, not as a developer-machine cache.

## Cold-start characteristics

A Pristine app's cold start does roughly:

1. Node process spawn (~30-50ms on a warm host, more on small VMs)
2. `reflect-metadata` import + module graph evaluation (~50-200ms depending on imports)
3. `kernel.start(AppModule)`: tsyringe registrations, configuration resolution, module
   onInit hooks (~50-500ms depending on app size)
4. `kernel.start(AppModule)` afterInit phase: routing tree built, `@route` decorators
   processed (~10-100ms)
5. RuntimeServer.start() for every registered server (HTTP server `listen` call: ~5ms)

Total: typically 200ms to 1 second on real hardware. Lambdas with conservative memory
allocations run higher. The dominant variable is the number of modules imported and how
many `providerRegistrations` they each have.

If cold-start latency matters (Lambda, autoscaling that scales from 0):

- Strip unused module imports. Every import means more bytecode parsed at start.
- Avoid `import "..."` of large libraries at the top of frequently-loaded files unless
  you actually use them — Node parses imported modules even before you call into them.
- Defer expensive constructor work behind lazy initialization. tsyringe doesn't construct
  classes until they're resolved, but constructor work happens on first resolve — if
  that first resolve happens during request handling, you've moved the cost to the hot
  path.

## Container readiness gates

A common deployment mistake is letting the load balancer route traffic to a container
before it's actually ready. The fix has two parts:

**1. The orchestrator must wait for the readiness probe before routing.**

```yaml
# Kubernetes Pod spec
readinessProbe:
  httpGet:
    path: /health/ready
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 3
```

`initialDelaySeconds: 5` is what gives the kernel time to finish booting before the
first probe arrives. Tune based on your measured boot time + safety margin.

**2. The readiness endpoint must actually answer "no" until boot is done.**

`pristine start` doesn't bind the HTTP server until after the kernel boots and runtime
servers come up. That means before the server is listening, probes will fail with
connection-refused — which Kubernetes treats as "not ready," which is the right answer.
After the server is up, your `/health/ready` endpoint is responsible.

The fragile case is **partial readiness**: the HTTP server is up, but a dependency
(database, downstream service) isn't reachable yet. If your readiness endpoint returns
200 unconditionally, traffic will arrive while the app can't actually serve it. Use
`InstantiationTestInterface` for dependency checks (see [Health checks](03.health-checks.md)).

## Warming caches

If your app has cold-start sensitive code paths (compiling a regex set, building a
routing index, prefetching reference data), put it in `afterInit`:

```ts
export const ProductPricingModule: ModuleInterface = {
  keyname: "pristine.product-pricing",

  async afterInit(container: DependencyContainer) {
    // afterInit runs once during boot, after every module's onInit but before the
    // kernel hands off to the runtime servers. By the time you're here, all
    // tagged services are registered and resolvable.
    const cache = container.resolve(PricingCache);
    await cache.warmUp();
  },
};
```

`afterInit` blocks the kernel's `start()` from returning. That means it blocks the HTTP
server from binding, which means readiness probes won't pass until warm-up completes.
Use it deliberately — long warm-up here trades cold-start latency against first-request
latency.

**Anti-pattern**: doing warm-up in a `setTimeout(..., 0)` or detached promise from
`onInit`. That moves warm-up off the critical path of `start()`, so the HTTP server
binds and the readiness probe passes — but you're not actually warm yet. First requests
hit the cold path. Either own the warm-up cost in boot, or expose the warm/cold state
in your readiness probe so traffic waits.

## Memory and the event loop

Two operational considerations specific to Node:

- **`--max-old-space-size`**: Node's default heap (~1.5GB on x64) is fine for most apps.
  If you're seeing OOM under load, increase before changing application code. ECS task
  memory should be ~30% above expected heap to leave room for V8's own overhead.
- **Event-loop blocking**: long synchronous work (CPU-bound parsing, sync hashing, sync
  compression) blocks every other request handler in the process. Pristine doesn't
  protect against this; that's a Node-level concern. Profile with `--prof` or use Clinic
  if you suspect blocking.

## Pitfalls

- **Forgetting to include `.pristine/` in the image.** The manifest's missing, every
  `pristine start` rebuilds, you wonder why containers take 30 seconds to boot.
- **Building inside the container at runtime.** `pristine build` is a build-time step;
  bake the result into the image. Building at runtime makes `pristine start` slow and
  needs a writable filesystem and devDependencies and a TypeScript compiler — all of
  which a production image shouldn't have.
- **Setting `initialDelaySeconds: 0`.** Probes hit before `kernel.start` has finished,
  fail, and Kubernetes counts failures toward restart threshold. Even fast apps need
  3-5 seconds of grace.
- **Warm-up in detached promises.** Looks fast, isn't fast where it matters. Put
  warm-up in `afterInit` or block readiness on it.
