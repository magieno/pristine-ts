# Common pitfalls

A grab bag of production-only failure modes. Each shows up only after deployment —
unit tests pass, the dev loop works, but the first time you put it in front of real
traffic something goes sideways.

> **You'll learn**
> - The dozen-or-so issues that come up in production but not in development
> - Quick diagnosis and the canonical fix for each

## SIGTERM doesn't trigger shutdown

**Symptom**: `docker stop`, `kubectl delete pod`, `systemctl stop` all take exactly the
escalation timeout (10s for Docker, 30s for Kubernetes default) and then your container
is killed hard. `onShutdown` hooks never run.

**Likely cause**: Node is running as PID 1 with no signal handler, OR a wrapper
(`npm start`, a shell `&&` chain) is between the orchestrator and Node.

**Fix**:
- Add `tini` (or `docker run --init`) so something forwards signals.
- Invoke `npx pristine start` directly, not via `npm start`.
- In systemd, `Type=simple` + `ExecStart=/usr/bin/npx pristine start` works as-is — but
  make sure your shell isn't between systemd and Node.

## Build manifest is missing in production

**Symptom**: container takes 20-60 seconds to be ready. Pristine logs say something
about rebuilding.

**Likely cause**: `.pristine/` is in your `.dockerignore` or wasn't copied from the
build stage.

**Fix**: Add `COPY --from=build /app/.pristine ./.pristine` to your Dockerfile, and
make sure `.pristine/` isn't excluded.

## reflect-metadata is loaded multiple times

**Symptom**: `TypeError: Reflect.getMetadata is not a function`, OR `TypeInfo not known
for ...`, OR decorators silently no-op.

**Likely cause**: Two copies of `reflect-metadata` in `node_modules`. Common with
monorepos using `file:..` package references where each package brings its own copy.

**Fix**: Hoist `reflect-metadata` to the root. In a monorepo, delete the copies from
sub-package `node_modules` directories. In a single repo, only `import "reflect-metadata"`
once, at the top of the entry file (`main.ts` / `handler.ts`).

## Configuration validation error at start

**Symptom**: `ConfigurationValidationError: There are no ConfigurationDefinition... for
'pristine.http.kernel-server.address'` (or similar key).

**Likely cause**: Configuration overrides reference a module that isn't actually in the
import graph at boot time. This bites first-time `pristine build` runs when no `dist/`
exists yet, because the CLI falls back to a minimal module set.

**Fix**: Move override values into module-level `configurationDefinitions` defaults, or
ensure the module owning the key is in the import graph for every code path that boots
the kernel.

## Process exits 0 immediately under `pristine start`

**Symptom**: `pristine start` prints "Pristine app running" and then the process exits
right away.

**Likely cause**: No runtime server is registered. The heartbeat interval should keep
the loop alive, but if you've overridden anything Node-process-level (custom
`process.on("beforeExit")` handler, `process.exit()` in `afterInit`), you can defeat it.

**Fix**: Check that `HttpModule` (or whatever runtime server you expect) is in the
`importModules` chain reachable from your AppModule. Run `pristine verify` — it will
catch missing-server cases.

## `onShutdown` hangs forever

**Symptom**: SIGTERM arrives, "Shutting down gracefully..." prints, then nothing for 30
seconds, then "Shutdown exceeded 30000ms — forcing exit."

**Likely cause**: A hook is awaiting a Promise that never resolves. Common culprits: a
DB pool waiting on in-flight queries that take longer than the timeout; a queue
consumer waiting for a poll cycle that's longer than the timeout; a network call to a
service that's down.

**Fix**: Add per-hook timeouts in your own code, fall back gracefully when external
deps are slow, and don't try to drain everything within the shutdown window — make
your work checkpointable elsewhere.

## Configuration values appear undefined despite being set

**Symptom**: `process.env.MY_KEY` is set in the container, but injected configuration
arrives as `undefined`.

**Likely cause**: `environmentResolver` isn't matching the key name. Pristine
configuration keys use dotted names like `pristine.http.kernel-server.port`, which by
default map to env var `PRISTINE_HTTP_KERNEL_SERVER_PORT` (dots → underscores,
uppercased). Custom keys must follow the same convention or specify their own
`environmentVariable` in the definition.

**Fix**: Either rename the env var to match, or set `environmentVariable: "MY_KEY"`
explicitly in the `ConfigurationDefinition`.

## "First request after deploy is slow"

**Symptom**: P99 latency spikes for the first ~30 seconds after a rolling deploy.

**Likely cause**: tsyringe lazily constructs services on first resolution. The first
request hits cold caches, opens DB connections, JIT-compiles hot paths. This is normal
Node-level behavior, not specifically Pristine.

**Fix**: Warm up critical paths in `afterInit` (see [Performance](04.performance.md)).
For Lambda, [provisioned concurrency](https://docs.aws.amazon.com/lambda/latest/dg/configuration-concurrency.html)
solves this differently.

## Logs are missing or doubled

**Symptom**: structured log calls don't show up, or show up twice.

**Likely cause**: Multiple `LogHandlerInterface` implementations registered (each
prints), OR none registered (everything goes to a no-op handler).

**Fix**: Check `providerRegistrations` for `LogHandlerInterface`. The CLI auto-registers
a console handler if none is set — make sure you're not registering a second one
manually.

## "Why is my Lambda cold start so slow?"

**Symptom**: 3-10 second cold starts on a small Lambda.

**Likely cause**: The combination of (a) low memory allocation (≤512MB caps CPU
proportionally), (b) `reflect-metadata` + tsyringe doing constructor walks at first
resolve, and (c) bundling pulling in more code than needed.

**Fix**: 
- Bump Lambda memory to 1024MB+ (CPU scales with memory).
- Initialize the kernel **outside** the handler (at module top level) so the boot
  cost happens during the init phase, not on every cold start's first request.
- Use a real bundler (esbuild) and tree-shake.

## `pristine verify` passes locally but fails in CI

**Symptom**: CI's `pristine verify` step reports "instantiation test failed" or
"configuration missing"; local runs pass.

**Likely cause**: CI is missing environment variables that your local `.env` (loaded by
`@pristine-ts/configuration`) provided silently. Or, more subtly, CI is running against
a different `NODE_ENV` and your config has env-conditional defaults.

**Fix**: Make every required config key explicit. Set it in CI's secrets, OR provide
a non-secret default that's safe in CI. Don't depend on `.env` files in CI.
