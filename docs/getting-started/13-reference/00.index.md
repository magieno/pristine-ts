# API reference

Auto-generated symbol reference for every published `@pristine-ts/*` package. Use this
chapter to look up a specific decorator, interface, or class by name. Use the tutorial
chapters (01-12) to learn how concepts fit together.

This reference is produced by [TypeDoc](https://typedoc.org/) from the TypeScript
sources under `packages/*/src/`. JSDoc comments in the source become prose; everything
else is derived from the types.

> **You'll learn**
> - How to view the auto-generated reference
> - How to regenerate it locally and in CI
> - How the reference is organized and what's included
> - When to consult the reference vs. the tutorial

## Viewing the reference

The reference is regenerated locally — it's not checked in. To generate it for the
first time:

```bash
npm install            # one-time, pulls TypeDoc as a devDependency
npm run docs:reference # generates docs/getting-started/13-reference/api/
```

After generation, open `docs/getting-started/13-reference/api/index.html` in a browser.
The output is a static site with full-text search, package-grouped navigation, and
clickable links between symbols.

### Live preview while editing

```bash
npm run docs:reference:watch
```

This watches the source and regenerates on change. Useful when you're authoring JSDoc
comments and want to verify the rendered output.

## What you'll find on each symbol page

For each exported class, interface, decorator, enum, or type, the reference shows:

1. **Signature** — the canonical TypeScript declaration.
2. **Description** — the JSDoc comment (if present in source).
3. **Members** — methods, properties, parameters, with their own signatures and types.
4. **References** — other symbols that link to or from this one.
5. **Source link** — where in `packages/*/src/` this symbol is defined.

Reference pages are deliberately *thin on prose* — they describe the shape, not the
story. If you find yourself wishing the reference had an example, that example belongs
in the tutorial chapter for that concept. Add it to the corresponding chapter file
under `docs/getting-started/`, not into the JSDoc comment (which would have to be
duplicated across every symbol that participates in the example).

## When to consult the reference

- **You know the name, you want the signature.** Reference.
- **You know the shape, you want to know what's available.** Reference (browse by
  package).
- **You don't know what to use for X.** Tutorial chapter for X.
- **You want to copy a working example.** Tutorial chapter; the reference does not
  ship examples by design.

## CI integration

Regenerate as part of CI to catch JSDoc syntax errors and broken cross-references:

```yaml
# .github/workflows/docs.yml
- name: Generate API reference
  run: npm run docs:reference
- name: Deploy to GitHub Pages    # optional
  uses: peaceiris/actions-gh-pages@v3
  with:
    publish_dir: docs/getting-started/13-reference/api
```

The output is self-contained static HTML and can be served from any static host
(GitHub Pages, S3 + CloudFront, Cloudflare Pages, etc.).

## Adding good JSDoc

The reference is only as good as the comments. A few conventions that pay off:

- **Lead with intent**, not mechanism: "Validates an inbound JWT and returns the
  decoded claims" beats "calls jwt.verify and returns the result."
- **Document parameters with `@param` only when the type alone isn't enough.** If a
  parameter is named `userId: string`, `@param userId` is noise. If it's `mode:
  number` and `0` means "strict" while `1` means "lenient," document it.
- **Use `@example` blocks** for non-obvious decorator combinations. TypeDoc renders
  them as code blocks in the symbol page.
- **`@internal`** on anything not part of the public API — exporters that exist for
  testing, helpers exposed for cross-package use but not for embedders. The reference
  filters these out.
