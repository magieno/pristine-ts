# Pristine 2.0.0 Migration Guide

> **Audience:** an AI coding agent fixing a downstream consumer after a `@pristine-ts/*` bump from 1.x to 2.0.0. Every section below is a self-contained recipe: what broke, how to find it (`rg` / `grep` pattern), how to fix it. Apply in order.
>
> **Scope:** API-facing breaking changes only. Internal refactors that don't affect consumer code are omitted.

---

## 0. Pre-flight

Before mechanical edits, audit the surface area:

```sh
# All Pristine imports in the codebase
rg -l "from ['\"]@pristine-ts/" -t ts

# Things this migration will touch
rg "LoggableError|HttpError|BadRequestHttpError|NotFoundHttpError|ForbiddenHttpError|InvalidBodyHttpError|BreadcrumbHandler|BreadcrumbModel|SpanTrailProvider|Object\.setPrototypeOf|originalError|EventDispatchingError|EventMappingError|EventInterceptionExecutionError|ErrorResponseSanitizerRequestInterceptor" -t ts
```

If the second command returns nothing, only sections **3** (`@traced` re-export) and **6** (`HttpErrorResponder` constructor) might apply.

## 1. Breadcrumbs are gone

### 1.1 `BreadcrumbHandler` / `BreadcrumbHandlerInterface` deleted

`@pristine-ts/logging` no longer exports `BreadcrumbHandler`, `BreadcrumbHandlerInterface`, or `BreadcrumbModel`. The whole concept is replaced by **span events** on the active trace.

**Find:**
```sh
rg "BreadcrumbHandlerInterface|BreadcrumbHandler|BreadcrumbModel" -t ts
```

**Fix — per import site:**

```ts
// BEFORE
import {BreadcrumbHandlerInterface, LogHandlerInterface} from "@pristine-ts/logging";
constructor(
  @inject("LogHandlerInterface") private readonly logHandler: LogHandlerInterface,
  @inject("BreadcrumbHandlerInterface") private readonly breadcrumbHandler: BreadcrumbHandlerInterface,
) {}
// usage:
this.breadcrumbHandler.add(eventId, "validation passed", {field: "email"});
```

```ts
// AFTER
import {LogHandlerInterface} from "@pristine-ts/logging";
import {TracingManagerInterface} from "@pristine-ts/common";
constructor(
  @inject("LogHandlerInterface") private readonly logHandler: LogHandlerInterface,
  @inject("TracingManagerInterface") private readonly tracingManager: TracingManagerInterface,
) {}
// usage:
this.tracingManager.addEventToCurrentSpan("validation passed", {field: "email"});
```

Note: `TracingManagerInterface` now imports from `@pristine-ts/common`, not `@pristine-ts/telemetry`. Both paths still work (re-exported) but prefer common.

### 1.2 `LogData.breadcrumb` field deleted

```sh
rg "breadcrumb:\s*['\"]" -t ts          # the data-arg form
rg "outputBreadcrumbs" -t ts             # the OutputHints field
```

**Fix:**

```ts
// BEFORE
this.logHandler.info("Done", {breadcrumb: "service.done"});
this.logHandler.notice("...", {outputHints: {outputBreadcrumbs: true}});
```

```ts
// AFTER
this.tracingManager.addEventToCurrentSpan("service.done");
this.logHandler.info("Done");
// outputBreadcrumbs has no replacement — `outputHints: {outputBreadcrumbs: true}` is
// removed entirely. Delete the option. To see the trace alongside logs in dev, enable
// ConsoleTracer (PRISTINE_TELEMETRY_CONSOLE_TRACER_ACTIVATED=true).
this.logHandler.notice("...");
```

### 1.3 `LogModel.breadcrumbs` and `OutputHints` shape change

If you have a **custom `LoggerInterface` implementation** that reads `log.breadcrumbs` or `log.outputHints.outputBreadcrumbs`:

```sh
rg "log\.breadcrumbs|outputHints\.outputBreadcrumbs|log\.outputHints" -t ts
```

**Fix:** delete those reads. The trail of "what happened" lives in the trace now — render via tracers, not loggers. If your logger genuinely needs the trail, resolve `TracingManagerInterface` from the active container and call `getCurrentTrail()`:

```ts
import {EventContextManager} from "@pristine-ts/common";

const trail = EventContextManager.container()
  ?.resolve<TracingManagerInterface>("TracingManagerInterface")
  ?.getCurrentTrail() ?? [];
```

### 1.4 `SpanTrailProviderInterface` deleted

Custom implementations of `SpanTrailProviderInterface` are no longer registered or consumed. Delete the implementing class and any registration.

```sh
rg "SpanTrailProviderInterface" -t ts
```

## 2. `LoggableError` → `PristineError`

`LoggableError` is deleted. Every custom error class that extended it must move to `PristineError`.

### 2.1 Mechanical class migration

**Find:**
```sh
rg "extends LoggableError" -t ts
rg "from ['\"]@pristine-ts/common['\"]" -t ts | rg "LoggableError"
```

**Fix — each class:**

```ts
// BEFORE
import {LoggableError} from "@pristine-ts/common";

export class TokenExpiredError extends LoggableError {
  public constructor(public readonly tokenId: string) {
    super("The token has expired", {tokenId});
    Object.setPrototypeOf(this, TokenExpiredError.prototype);
  }
}
```

```ts
// AFTER
import {PristineError} from "@pristine-ts/common";

export class TokenExpiredError extends PristineError {
  public constructor(public readonly tokenId: string) {
    super("The token has expired", {
      code: "TOKEN_EXPIRED",      // stable slug for the error category
      httpStatus: 401,             // omit if no natural HTTP status
      exitCode: 77,                // omit to fall back to 1 (expected) or 70 (unexpected)
      expected: true,              // default; set false for system bugs
      details: {tokenId},          // structured fields safe to surface
    });
    // Object.setPrototypeOf is now handled automatically by PristineError's constructor.
    // Remove the line.
  }
}
```

**Key transformations:**
- `extends LoggableError` → `extends PristineError`
- Remove the `Object.setPrototypeOf(this, X.prototype)` line and its 3-line comment block.
- The old free-form `extra` object becomes `details` inside the typed `options` bag.
- Add a `code` (string slug) — used as the join key for logs / alerting / i18n.
- Add `httpStatus` if the error has natural HTTP meaning, `exitCode` for CLI.
- Set `expected: false` for system bugs / wrapped library errors (sanitized in production).
- Wrap any "the underlying error I'm wrapping" in `cause:` instead of stashing on `extra.originalError` — uses the JS standard `Error.cause` chain.

### 2.2 Reading error options at the call site

If consumer code inspected `error.extra` or `error.httpStatus` directly:

```sh
rg "\.extra\b" -t ts | rg -v "// node_modules"
rg "\.httpStatus\b|\.originalError\b" -t ts
```

**Fix:**

```ts
// BEFORE
if (error instanceof HttpError) {
  response.status = error.httpStatus;
  response.body = error.extra;
}
const cause = error.extra?.originalError;
```

```ts
// AFTER
import {PristineError} from "@pristine-ts/common";

const e = PristineError.from(error);   // normalizes anything
const status = e.options.httpStatus ?? 500;
const code = e.options.code ?? "INTERNAL_ERROR";
const details = e.options.details;
const cause = e.cause;                  // standard Error.cause, walked chain
```

Most consumers should **not** be inspecting errors manually — let the channel reporters do it (sections 6 and 7).

## 3. `HttpError` hierarchy deleted

Every typed HTTP error is replaced by the standard library in `@pristine-ts/common`.

### 3.1 Find and replace

| Old class (`@pristine-ts/networking`) | New class (`@pristine-ts/common`) |
|---|---|
| `HttpError` | `PristineError` (with `httpStatus`) |
| `BadRequestHttpError(msg, errors?)` | `BadRequestError(msg, {details: {errors}})` |
| `NotFoundHttpError(msg)` | `NotFoundError(msg)` |
| `ForbiddenHttpError(msg)` | `ForbiddenError(msg)` |
| `InvalidBodyHttpError(msg)` | `BadRequestError(msg, {code: "INVALID_BODY"})` |

**Find:**
```sh
rg "BadRequestHttpError|NotFoundHttpError|ForbiddenHttpError|InvalidBodyHttpError|\bHttpError\b" -t ts
```

**Fix — per call site:**

```ts
// BEFORE
import {BadRequestHttpError, NotFoundHttpError, ForbiddenHttpError, InvalidBodyHttpError, HttpError} from "@pristine-ts/networking";
throw new BadRequestHttpError("Bad payload", validationErrors);
throw new NotFoundHttpError("User not found");
throw new ForbiddenHttpError("Not allowed");
throw new InvalidBodyHttpError("Body is not valid JSON");
if (err instanceof HttpError) { /* ... */ }
```

```ts
// AFTER
import {BadRequestError, NotFoundError, ForbiddenError, PristineError} from "@pristine-ts/common";
throw new BadRequestError("Bad payload", {details: {errors: validationErrors}});
throw new NotFoundError("User not found");
throw new ForbiddenError("Not allowed");
throw new BadRequestError("Body is not valid JSON", {code: "INVALID_BODY"});
if (err instanceof PristineError && err.options.httpStatus !== undefined) { /* ... */ }
```

### 3.2 Custom subclasses of `HttpError`

If consumer code defined its own typed HTTP error like `StripeAuthError extends HttpError`, change the base class to `PristineError` and pass `httpStatus` through options:

```ts
// BEFORE
export class StripeAuthError extends HttpError {
  public constructor(httpStatus: number, message: string, errors?: any[]) {
    super(httpStatus, message, errors);
    Object.setPrototypeOf(this, StripeAuthError.prototype);
  }
}
```

```ts
// AFTER
import {PristineError} from "@pristine-ts/common";

export class StripeAuthError extends PristineError {
  public constructor(httpStatus: number, message: string, errors?: any[]) {
    super(message, {
      code: "STRIPE_AUTH_FAILED",
      httpStatus,
      exitCode: 1,
      expected: true,
      details: errors ? {errors} : undefined,
    });
  }
}
```

## 4. Pipeline wrapper errors deleted

These eight classes are deleted; the pipeline no longer wraps thrown errors. Original error types now propagate intact with their `instanceof` identity preserved.

| Deleted (`@pristine-ts/core`) | Replacement |
|---|---|
| `EventDispatchingError` | original error re-thrown |
| `EventMappingError` | `PristineError` with `code: "EVENT_MAPPING_FAILED"` |
| `EventInterceptionExecutionError` | original error re-thrown |
| `EventPreMappingInterceptionError` | `PristineError` with `code: "EVENT_PRE_MAPPING_INTERCEPTOR_FAILED"` + `cause` |
| `EventPostMappingInterceptionError` | `PristineError` with `code: "EVENT_POST_MAPPING_INTERCEPTOR_FAILED"` + `cause` |
| `EventPreResponseMappingInterceptionError` | `PristineError` with `code: "EVENT_PRE_RESPONSE_INTERCEPTOR_FAILED"` + `cause` |
| `EventPostResponseMappingInterceptionError` | `PristineError` with `code: "EVENT_POST_RESPONSE_INTERCEPTOR_FAILED"` + `cause` |
| `ErrorResponseInterceptionExecutionError` | original error re-thrown |

Also deleted (unused in 1.x already):
- `RequestInterceptionExecutionError` (`@pristine-ts/networking`)
- `ResponseInterceptionExecutionError` (`@pristine-ts/networking`)

### 4.1 Catch blocks that checked these wrapper types

**Find:**
```sh
rg "instanceof Event(Dispatching|Mapping|Interception|PreMapping|PostMapping|PreResponseMapping|PostResponseMapping)\w*Error" -t ts
rg "instanceof (Request|Response)InterceptionExecutionError" -t ts
```

**Fix:** check the actual underlying error type, or the new code:

```ts
// BEFORE
try { await dispatcher.dispatch(event); }
catch (err) {
  if (err instanceof EventDispatchingError) {
    const original = err.extra?.originalError;
    // ...
  }
}
```

```ts
// AFTER — the original error propagates directly; no wrapper
try { await dispatcher.dispatch(event); }
catch (err) {
  if (err instanceof BadRequestError) { /* ... */ }
  // Or walk the cause chain for wrapped pipeline errors:
  const e = PristineError.from(err);
  if (e.options.code === "EVENT_PRE_MAPPING_INTERCEPTOR_FAILED") {
    const cause = e.cause;
    // ...
  }
}
```

### 4.2 Tests that asserted on wrapper types

**Find:**
```sh
rg "toThrow\(new Event(Dispatching|Mapping|Interception|PreMapping|PostMapping|PreResponseMapping|PostResponseMapping)\w*Error" -t ts
```

**Fix:** assert on the error message string or the new `code`:

```ts
// BEFORE
.rejects.toThrow(new EventMappingError("There are no events to execute.", event, event, executionContext));
```

```ts
// AFTER
.rejects.toThrow("There are no events to execute.");
// or:
.rejects.toMatchObject({options: {code: "EVENT_NO_EVENTS"}});
```

## 5. `ErrorResponseSanitizerRequestInterceptor` deleted

Sanitization is now the **default** in `HttpErrorResponder`. The opt-in interceptor + its config key are gone.

**Find:**
```sh
rg "ErrorResponseSanitizerRequestInterceptor|error_response_sanitizer\.is_active|PRISTINE_NETWORKING_ERROR_RESPONSE_SANITIZER_IS_ACTIVE" -t ts -t json
```

**Fix:**
- Delete any custom registration of `ErrorResponseSanitizerRequestInterceptor`.
- Delete any reference to the `pristine.networking.error_response_sanitizer.is_active` config / `PRISTINE_NETWORKING_ERROR_RESPONSE_SANITIZER_IS_ACTIVE` env var. The framework no longer reads it.
- To **opt back into verbose, stack-trace-leaking error responses** for local debugging, set `PRISTINE_MODE=development` (see section 8).

## 6. `HttpErrorResponder` is now a Router dependency

`Router`'s constructor signature changed: a 7th argument `HttpErrorResponder` was added.

### 6.1 If you wire the Router via DI (normal case)

**No action needed.** The framework registers `HttpErrorResponder` automatically and resolves it via DI.

### 6.2 If you instantiate Router manually (custom servers, tests)

**Find:**
```sh
rg "new Router\(" -t ts
```

**Fix:**
```ts
// BEFORE
const router = new Router(
  logHandler,
  resolver,
  authorizerManager,
  authenticationManager,
  cache,
  requestContextManager,
);
```

```ts
// AFTER
import {HttpErrorResponder} from "@pristine-ts/networking";

const router = new Router(
  logHandler,
  resolver,
  authorizerManager,
  authenticationManager,
  cache,
  requestContextManager,
  new HttpErrorResponder(),   // ← new 7th arg
);
```

### 6.3 HTTP error response body shape changed

If consumer code asserts on the JSON body of error responses (integration tests, gateway clients):

**Old body shape:**
```json
{"name": "BadRequestHttpError", "message": "...", "stack": "...", "errors": [...], "extra": {...}}
```

**New body shape (production mode):**
```json
{"code": "BAD_REQUEST", "message": "...", "details": {...}}
```

**New body shape (development mode — `PRISTINE_MODE=development`):**
```json
{"code": "BAD_REQUEST", "message": "...", "details": {...}, "debugMessage": "...", "stack": "...", "cause": [...]}
```

**Fix tests:** update assertions to match the new keys. Specifically:
- `body.name` → `body.code` (and the value is now a `SCREAMING_SNAKE_CASE` slug, not the class name).
- `body.errors` → `body.details.errors` (when the throwing code chose that key).
- `body.stack` → gone unless `PRISTINE_MODE=development`.
- `body.extra` → split into `body.details` (safe) and the dev-only diagnostic fields.

## 7. CLI bin needs an error guard

If consumer code provides its own bin script (not `node_modules/.bin/pristine`), wrap `bootstrap()` so escaped errors render via `CliErrorReporter` instead of Node's default unhandled-rejection dump.

**Find:**
```sh
rg "bootstrap\(\)" -t ts -t js bin/ src/bin*
```

**Fix:**

```ts
// BEFORE (custom bin)
require('@pristine-ts/cli').bootstrap();
```

```ts
// AFTER
const cli = require('@pristine-ts/cli');
cli.bootstrap().catch((err) => {
  const exitCode = cli.cliErrorReporter.report(err);
  process.exit(exitCode);
});
```

The shipped `node_modules/.bin/pristine` already does this — no action needed for the default bin.

## 8. New environment variable: `PRISTINE_MODE`

Controls how errors are surfaced through HTTP and CLI reporters.

- `PRISTINE_MODE=production` (default): sanitized output. No stack traces, no internal-error messages leaked.
- `PRISTINE_MODE=development`: verbose. Stack traces, cause chains, structured details surfaced.

**Action:** set `PRISTINE_MODE=development` in local dev shells / `.env.local` if you want the old verbose output. Production deployments should leave it unset (defaults to safe).

If consumer code reads the mode programmatically:
```ts
import {getPristineMode, PristineMode} from "@pristine-ts/common";
if (getPristineMode() === PristineMode.Development) { /* ... */ }
```

`@traced` is no longer re-exported from `@pristine-ts/telemetry`. Import from `@pristine-ts/common`.

**Find:**
```sh
rg "import.*traced.*@pristine-ts/telemetry|from ['\"]@pristine-ts/telemetry['\"].*traced" -t ts
```

**Fix:**

```ts
// BEFORE
import {traced} from "@pristine-ts/telemetry";
```

```ts
// AFTER
import {traced} from "@pristine-ts/common";
```

`spanRunner`, `SpanRunner`, `SpanRunnerOptions` are still re-exported from telemetry for back-compat — but prefer importing them from `@pristine-ts/common` as well. Same applies to `Trace`, `Span`, `SpanEvent`, `SpanKeynameEnum`, `TracingManagerInterface`.

## 10. `TracingManagerInterface` gained a method

If consumer code mocks `TracingManagerInterface` (manual stubs in tests), add the new `getCurrentTrail` method to the mock:

**Find:**
```sh
rg "TracingManagerInterface\s*=\s*\{" -t ts
```

**Fix:**

```ts
// BEFORE
const tm: TracingManagerInterface = {
  trace: undefined,
  startTracing: jest.fn(),
  endTrace: jest.fn(),
  addSpan: jest.fn(),
  startSpan: jest.fn(),
  endSpan: jest.fn(),
  endSpanKeyname: jest.fn(),
  addEventToCurrentSpan: jest.fn(),
};
```

```ts
// AFTER
const tm: TracingManagerInterface = {
  trace: undefined,
  startTracing: jest.fn(),
  endTrace: jest.fn(),
  addSpan: jest.fn(),
  startSpan: jest.fn(),
  endSpan: jest.fn(),
  endSpanKeyname: jest.fn(),
  addEventToCurrentSpan: jest.fn(),
  getCurrentTrail: jest.fn((): any[] => []),   // ← new
};
```

### 10.1 `TracingManager.spans` removed

If consumer code reads `tracingManager.spans[keyname]`:

**Find:**
```sh
rg "tracingManager\.spans|\.spans\[" -t ts
```

**Fix:** the by-keyname index now lives on the active `Trace`:

```ts
// BEFORE
const rootSpans = this.tracingManager.spans["root.execution"];
```

```ts
// AFTER
const trace = EventContextManager.current()?.trace ?? this.tracingManager.trace;
const rootSpans = trace?.spansByKeyname["root.execution"] ?? [];
```

### 10.2 `addEventToCurrentSpan` warns instead of silently no-op'ing

Behavior change, not API change. If you call `addEventToCurrentSpan` outside an active trace, the framework now logs a warning at `WARNING` severity and drops the marker. Either:
- Ensure `tracingManager.startTracing()` has been called earlier in the flow, **or**
- Suppress the warning in your logger filter if the call is genuinely fire-and-forget.

## 11. `EventContext.trace` is the new source of truth

If consumer code reads the active trace from a specific `TracingManager` instance, switch to `EventContext.trace`. This is the fix for the per-container-instance split-trace problem.

**Find:**
```sh
rg "tracingManager\.trace\b" -t ts
```

**Fix:**
```ts
// BEFORE — reads only this manager's own trace
const trace = this.tracingManager.trace;
```

```ts
// AFTER — reads the trace that's actually active in the current event
import {EventContextManager} from "@pristine-ts/common";
const trace = EventContextManager.current()?.trace ?? this.tracingManager.trace;
```

In most cases you don't need this directly — the public `tracingManager.addEventToCurrentSpan(...)` / `getCurrentTrail()` methods read EventContext correctly on their own.

## 12. CLI internals: `CliEventHandler.resolveArgs` signature changed

If consumer code calls `CliEventHandler.resolveArgs` directly (rare — it's normally called by the framework):

**Old:** returns `{args?: any; exitCode?: ExitCodeEnum}` — a result-tuple shape.

**New:** returns `any` on success; **throws** `UsageError` or `ValidationError` on failure.

**Find:**
```sh
rg "resolveArgs\(" -t ts
```

**Fix:**

```ts
// BEFORE
const result = await handler.resolveArgs(command, rawArgs);
if (result.exitCode !== undefined) {
  process.exit(result.exitCode);
}
await command.run(result.args);
```

```ts
// AFTER
try {
  const args = await handler.resolveArgs(command, rawArgs);
  await command.run(args);
} catch (err) {
  process.exit(cliErrorReporter.report(err));
}
```

## 13. `Object.setPrototypeOf` removal in custom error subclasses

The `PristineError` base class handles prototype chain restoration automatically via `new.target.prototype`. Subclasses no longer need the boilerplate.

**Find:**
```sh
rg "Object\.setPrototypeOf\(this," -t ts
```

**Fix:** delete the line and its 3-line comment block. Example:

```ts
// BEFORE
public constructor(...) {
  super(message, {details: {...}});

  // Set the prototype explicitly.
  // As specified in the documentation in TypeScript
  // https://github.com/Microsoft/TypeScript/wiki/Breaking-Changes#extending-built-ins-like-error-array-and-map-may-no-longer-work
  Object.setPrototypeOf(this, FooError.prototype);
}
```

```ts
// AFTER
public constructor(...) {
  super(message, {details: {...}});
}
```

This is safe only for classes that now extend `PristineError`. Pure `extends Error` classes still need the `setPrototypeOf` workaround if targeting ES5 — but if you're touching them anyway, consider migrating to `PristineError`.

## 14. `TypeScript lib` bump (only if your tsconfig is strict)

The framework's `PristineError` uses Node's standard `Error.cause` slot (Node 16.9+). Type-checking that slot requires `lib: ["es2022.error"]` or higher.

If consumer code extends `PristineError` and your tsconfig has a narrower `lib` (`es2020` or earlier) and gets:

```
error TS2550: Property 'cause' does not exist on type 'Error'.
```

**Fix `tsconfig.json`:**

```json
{
  "compilerOptions": {
    "lib": ["es2020", "es2022.error"]   // append "es2022.error" to existing lib
  }
}
```

Runtime is unaffected — `Error.cause` is natively supported on every Node ≥ 16.9.

## 15. `LogModel` field cleanup

Two fields are removed from `LogModel`. If consumer code wrote a custom `LoggerInterface` that serialized them:

| Removed field | Replacement |
|---|---|
| `log.breadcrumbs` | walk `EventContext.trace` directly (see 1.3) |
| `log.outputHints.outputBreadcrumbs` | no replacement; remove read |

**Find:**
```sh
rg "log\.breadcrumbs|outputBreadcrumbs|outputHints\.outputBreadcrumbs" -t ts
```

The `log.outputHints` field still exists but its type is now `Record<string, unknown>` (empty by default); custom loggers that wrote `outputHints: {outputBreadcrumbs: true}` should remove that.

## 18. New APIs worth knowing

| Name | Location | Use |
|---|---|---|
| `PristineError` | `@pristine-ts/common` | Single base class for every framework error |
| `PristineError.from(unknown)` | `@pristine-ts/common` | Normalize any thrown value into a `PristineError` |
| `BadRequestError`, `UnauthorizedError`, `ForbiddenError`, `NotFoundError`, `ConflictError`, `ValidationError`, `ConfigError`, `UsageError`, `InternalError` | `@pristine-ts/common` | Standard typed errors with sensible HTTP / exit-code defaults |
| `PristineMode`, `getPristineMode()` | `@pristine-ts/common` | Production vs Development mode helper |
